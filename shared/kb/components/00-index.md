# Каталог штатных компонентов — индекс (роутер)

> Это тонкий роутер. Карточку конкретного компонента читай по запросу из нужного
> группового файла — **не загружай весь каталог сразу**.
> Ссылки `[src]` — провенанс на документацию вендора (`dev.1c-bitrix.ru/user_help/components/…`).
> Это указатель на первоисточник, а не материал «для скачивания целиком».

Цель каталога — «золотой путь» вывода: выбрать компонент → узнать ключевые
`arParams` → при необходимости скопировать его шаблон в `/local` (механика —
`../recipes/template/07-component-templates.md`). Карта «задача → компонент» и
кэш-границы — в `../api-map.md` (§5, §6); эталон параметров списка — в
`../recipes/06-output-on-page.md`.

## Формат карточки (7 фиксированных подразделов)

Каждая полная карточка идёт ровно по этой схеме:

1. **Назначение** — что делает компонент (1–2 строки).
2. **Когда брать** — для какой задачи; чем отличается от соседнего компонента.
3. **Ключевые `arParams`** — 5–12 самых важных параметров (имя — смысл — типичное
   значение), не весь список.
4. **Типовой вызов** — `$APPLICATION->IncludeComponent("bitrix:…", "шаблон", [ … ])`
   с минимальным иллюстративным набором значений.
5. **Что кастомизируют в `/local`** — какой шаблон копировать и куда; что правят в
   `template.php` vs `result_modifier.php` vs `component_epilog.php` (кэш-граница;
   `SetTitle`/мета — в эпилог).
6. **Типовые ошибки** — 2–4 частых промаха.
7. **Источник** — `[src]` deep-ссылка на страницу компонента в документации вендора.

Заглушки (map / subscribe / personal / registration / profile) дают только
подразделы 1, 2 и 7 + указание «настраивается мастером / в админке».

# Компоненты: формы и авторизация

Форма обратной связи и вход на сайт — полные карточки; регистрация и профиль —
заглушки (настраиваются мастером системы авторизации). Подробнее про формы —
рецепт `../recipes/10-feedback-form.md`; про веб-формы модуля `form` — `../api-map.md` (§9).

---

## `bitrix:main.feedback`

1. **Назначение.** Простая форма обратной связи: поля «имя / e-mail / сообщение» +
   капча; на отправку шлёт письмо по почтовому событию (`CEvent::Send`).
2. **Когда брать.** Нужна лёгкая форма «напишите нам» без хранения заявок в БД.
   Для форм со статусами/полями/хранением результатов — модуль `form` и компонент
   `bitrix:form` (рецепт `../recipes/10-feedback-form.md`). Это не CRM-формы Б24.
3. **Ключевые `arParams`.**
   - `USE_CAPTCHA` — `"Y"` включить капчу (антиспам).
   - `EVENT_MESSAGE_ID` — ID шаблонов почтовых сообщений (тип события `FEEDBACK_FORM`).
   - `OK_TEXT` — текст после успешной отправки.
   - `EMAIL_TO` — адрес получателя письма.
   - `REQUIRED_FIELDS` — какие поля обязательны.
   - `CACHE_TYPE` — обычно форма некэшируема по сути (`"N"`).
4. **Типовой вызов.**
   ```php
   $APPLICATION->IncludeComponent("bitrix:main.feedback", ".default", [
       "USE_CAPTCHA"      => "Y",
       "EMAIL_TO"         => "info@example.com",
       "EVENT_MESSAGE_ID" => [],          // ID шаблонов письма; пусто — тип по умолчанию
       "OK_TEXT"          => "Спасибо! Сообщение отправлено.",
       "REQUIRED_FIELDS"  => ["NAME", "EMAIL", "MESSAGE"],
       "CACHE_TYPE"       => "N",
   ]);
   ```
5. **Что кастомизируют в `/local`.** Шаблон формы копируют в
   `/local/templates/<tpl>/components/bitrix/main.feedback/<tpl>/template.php` —
   правят разметку полей и классы. Текст письма — в почтовом шаблоне
   (`b_event_message`, плейсхолдеры `#FIELD#`), не в компоненте. Своя обработка
   результата — через события почты/формы, см. `../recipes/13-email-events.md`.
6. **Типовые ошибки.**
   - Письмо не уходит — не настроен SMTP в `.settings.php` или нет почтового шаблона.
   - Капча не проходит — кэширование страницы целиком ломает токен капчи.
   - Вывод полей без экранирования при перерисовке формы → XSS.
7. **Источник.** [src](https://dev.1c-bitrix.ru/user_help/components/sluzhebnie/main_feedback.php)

## `bitrix:system.auth.form`

1. **Назначение.** Выводит форму входа (логин/пароль, «запомнить меня», ссылки на
   регистрацию и восстановление пароля).
2. **Когда брать.** Блок авторизации на публичной части (в шапке или на
   `/auth/`). Системный компонент авторизации (`system.auth.*`) обычно
   подключается шаблоном страницы авторизации автоматически.
3. **Ключевые `arParams`.**
   - `FORGOT_PASSWORD_URL` — ссылка «забыли пароль».
   - `REGISTER_URL`, `PROFILE_URL` — ссылки на регистрацию/профиль.
   - `SHOW_ERRORS` — показывать ошибки входа.
   - `AUTH_RESULT` — внутренняя передача результата авторизации.
4. **Типовой вызов.**
   ```php
   $APPLICATION->IncludeComponent("bitrix:system.auth.form", ".default", [
       "FORGOT_PASSWORD_URL" => "/auth/?forgot_password=yes",
       "REGISTER_URL"        => "/auth/?register=yes",
       "PROFILE_URL"         => "/personal/",
       "SHOW_ERRORS"         => "Y",
   ]);
   ```
5. **Что кастомизируют в `/local`.** Шаблон формы — в
   `/local/templates/<tpl>/components/bitrix/system.auth.form/<tpl>/template.php`.
   Правят разметку полей и оформление; логику входа (это системный компонент) не
   трогают. Вход через соцсети — отдельная подсистема (`../recipes/14-social-login.md`).
6. **Типовые ошибки.**
   - Форма «не логинит» — отсутствует проверка/передача `sessid`; компонент даёт её сам.
   - Кэш страницы целиком прячет состояние «вошёл/не вошёл» — авторизацию держать
     в динамической области (composite).
   - Ссылки `REGISTER_URL`/`FORGOT_PASSWORD_URL` ведут не на ту страницу авторизации.
7. **Источник.** [src](https://dev.1c-bitrix.ru/user_help/components/sluzhebnie/user/system_auth_form.php)

## `bitrix:main.register` (заглушка)

1. **Назначение.** Форма регистрации нового пользователя на сайте.
2. **Когда брать.** Нужна публичная регистрация. Поля и обязательность настраивают
   в админке (модуль `main` → регистрация); компонент чаще ставит мастер страницы
   авторизации. Полную карточку с параметрами здесь не разворачиваем —
   конфигурация идёт через настройки модуля и шаблон страницы `/auth/`.
7. **Источник.** [src](https://dev.1c-bitrix.ru/user_help/components/sluzhebnie/user/main_register.php)

---

## `bitrix:main.profile` (заглушка)

1. **Назначение.** Выводит/редактирует профиль авторизованного пользователя в
   публичной части.
2. **Когда брать.** Страница «мой профиль» в личном кабинете. Набор полей зависит
   от пользовательских полей (UF) и настроек модуля `main`; настраивается в
   админке и шаблоне страницы профиля, а не разворачиванием `arParams` здесь.
7. **Источник.** [src](https://dev.1c-bitrix.ru/user_help/components/sluzhebnie/user/main_profile.php)

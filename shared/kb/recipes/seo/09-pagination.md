# Рецепт SEO-09: Пагинация (canonical, ЧПУ страниц, rel next/prev)

Под-скилл: `1c-bitrix-cms-seo` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x, постраничная навигация `bitrix:system.pagenavigation`, параметр `PAGEN_N`.

## Цель
Сделать постраничную навигацию каталога/новостей SEO-корректной: выбрать стратегию индексации страниц пагинации, убрать дубль первой страницы (`?PAGEN_1=1` = базовый URL), при необходимости перевести `?PAGEN_1=N` в человекопонятный `/page-N/`, проследить, чтобы SEO-текст категории не дублировался на всех страницах, а несуществующие страницы отдавали 404. Итог — глубокие товары/материалы доступны роботу, дублей нет, краулинговый бюджет тратится разумно.

## Когда применять
- Каталог/блог с многостраничными списками: в индекс лезут `?PAGEN_1=2,3,…`, первая страница доступна и по `/catalog/`, и по `/catalog/?PAGEN_1=1`.
- Нужно решить, индексировать ли страницы пагинации, и не потерять при этом индексацию глубоких карточек.
- Хотите человекопонятные URL страниц (`/catalog/page-2/`) вместо GET-параметра.
- Стоит AJAX-подгрузка / «Показать ещё» — и часть товаров недоступна роботу без JS.

## Шаги
1. **Выберите стратегию индексации.** Две рабочие школы, выберите одну под проект:
   - **A. Открытая индексация (для больших каталогов).** Страницы пагинации индексируются, но делаются **не-дублями**: уникальный `title` (`«<Категория> — страница N»`), SEO-текст категории — **только на 1-й странице**, на остальных физически отсутствует. Плюс: робот находит глубокие карточки. Это предпочтительно, когда товаров много и они должны попасть в индекс.
   - **B. Схлопывание в 1-ю страницу (`canonical`).** На `PAGEN > 1` ставится `rel=canonical` на базовый URL раздела. Проще, но Google может игнорировать canonical между разными по содержимому страницами, а глубокие карточки рискуют не проиндексироваться. Подходит для небольших списков.
2. **Уберите дубль первой страницы.** `…/?PAGEN_1=1` отдаёт тот же контент, что и базовый URL → дубль. Поставьте 301 с `?PAGEN_1=1` на чистый URL раздела (или, как минимум, canonical на него).
3. **Канонический URL стройте аккуратно.** `$APPLICATION->GetCurPage()` возвращает путь **без** GET — для пагинации этого мало. Чтобы получить базовый URL раздела без `PAGEN_`, используйте `GetCurPageParam('', ['PAGEN_1'])` (срезает указанный параметр) либо собирайте URL вручную из `SECTION_PAGE_URL`.
4. **(Опционально) Человекопонятные URL страниц.** `?PAGEN_1=N` → `/page-N/` / `/pagenN/`: буферизуется вывод `system.pagenavigation` и переписывается, а правило в `.htaccess` (с флагом `QSA`, чтобы сохранить GET-параметры фильтра) маршрутизирует `/page-N/` обратно в `PAGEN_1=N`. Готовая альтернатива — модуль «ЧПУ-пагинация» (например, в составе «16-в-1 SEO-мастер»).
5. **Контент категории — только на 1-й странице.** SEO-описание раздела выводите при `NavPageNomer == 1`; на остальных страницах его физически не должно быть в HTML (не прятать CSS — это всё равно дубль текста).
6. **(Опционально) `rel next/prev`.** `<link rel="prev/next">` Google **с 2019 не использует** для индексации, но Яндексу/Bing безвредны. Если внедряете — выводите в `<head>` через `AddHeadString()` на событии `OnEpilog` (пагинация рендерится в `<body>` уже после `<head>`, поэтому нужен отложенный вывод). Приоритет низкий — не тратьте на это силы раньше шагов 1–3.
7. **Несуществующие страницы → 404.** `?PAGEN_1=99999` за пределами реального числа страниц должен отдавать 404 (или редирект на последнюю/категорию), а не пустой `200` — см. рецепт `06`.

## Рабочий сниппет

Каноникал и заголовок пагинации в шаблоне списка (`section.php`/`list.php` комплексного компонента или `result_modifier.php`):

```php
<?php
// $arResult["NAV_RESULT"] — объект постраничной навигации (CDBResult/PageNavigation)
$page = (int)($arResult["NAV_RESULT"]->NavPageNomer ?? 1);
$base = "https://" . $_SERVER["SERVER_NAME"]
      . $APPLICATION->GetCurPageParam("", ["PAGEN_1"]);   // URL без PAGEN_1

if ($page > 1) {
    // Стратегия A: уникальный title, индексация открыта (canonical НЕ на 1-ю страницу).
    $APPLICATION->SetPageProperty("title",
        $APPLICATION->GetTitle() . " — страница {$page}");
    // Стратегия B (если выбрана): схлопнуть в 1-ю страницу
    // $APPLICATION->SetPageProperty("canonical", $base);
} else {
    // Первая страница: canonical на себя без параметров (страховка от ?PAGEN_1=1)
    $APPLICATION->SetPageProperty("canonical", $base);
    // SEO-текст категории выводим ТОЛЬКО здесь (на page>1 его в HTML быть не должно)
}
```

301 с дубля первой страницы (`init.php` или `.htaccess`):

```php
// /local/php_interface/init.php
use Bitrix\Main\Context;
$req = Context::getCurrent()->getRequest();
if ($req->get("PAGEN_1") === "1") {
    LocalRedirect(
        $req->getRequestedPage(),               // путь без query
        false, "301 Moved Permanently"
    );
}
```

`rel next/prev` через `OnEpilog` (если внедряете; абсолютные URL):

```php
// init.php
use Bitrix\Main\EventManager;
EventManager::getInstance()->addEventHandler("main", "OnEpilog", function () {
    global $APPLICATION, $NavNext, $NavPrev; // подготовьте URL в шаблоне списка
    if (!empty($NavNext)) $APPLICATION->AddHeadString('<link rel="next" href="' . $NavNext . '">', true);
    if (!empty($NavPrev)) $APPLICATION->AddHeadString('<link rel="prev" href="' . $NavPrev . '">', true);
});
```

## Выбор API
- **Базовый URL без `PAGEN_` — `$APPLICATION->GetCurPageParam('', ['PAGEN_1'])`**, а не `GetCurPage()` (последний теряет/не учитывает параметры пагинации). Это штатный способ собрать canonical/301-цель для пагинации.
- **301 первой страницы — `LocalRedirect($url, false, '301 Moved Permanently')`** (см. рецепт `04`; без третьего аргумента — 302, вес не передаётся). Альтернатива — правило в `.htaccess`.
- **ЧПУ `/page-N/`** — буферизация вывода `system.pagenavigation` + `.htaccess` (`QSA` для сохранения фильтра) **или** готовый модуль. Самописный вариант дешевле модуля, но требует аккуратной обработки совместно с фильтром/сортировкой.
- **`rel next/prev` — `AddHeadString()` на `OnEpilog`** (отложенный вывод в `<head>`). Это единственный корректный момент: компонент пагинации в `<body>` рендерится после закрытия `<head>`. Помните: для Google это сейчас нейтрально.
- **Стратегия A vs B** — это SEO-решение, а не API. Для крупных каталогов с глубоким ассортиментом обычно A (открыть + уникализировать), для коротких списков — B (canonical).

## Проверка

**Режим «только файлы» (без живого Битрикса):**
- В шаблоне списка строится `canonical`/`title` через `GetCurPageParam('', ['PAGEN_1'])`; ветка `page == 1` выводит SEO-текст, `page > 1` — нет.
- Выбранная стратегия согласована: при A — на `page>1` **нет** canonical на 1-ю страницу; при B — есть.
- 301 для `?PAGEN_1=1` использует `LocalRedirect(..., '301 Moved Permanently')`. `php -l` без ошибок.
- `rel next/prev` (если есть) выводится на `OnEpilog`, URL абсолютные.

**Режим «живой Битрикс»:**
- `curl -I "https://site.ru/catalog/?PAGEN_1=1"` → `301` на `https://site.ru/catalog/`.
- На `…/?PAGEN_1=2` `<title>` содержит «страница 2», SEO-текст категории в HTML отсутствует; при стратегии B — есть `rel=canonical` на базовый URL.
- `?PAGEN_1=99999` (за пределом) → `404`, а не пустой `200`.
- Если внедрён `/page-2/`: адрес отдаёт `200`, фильтр/сортировка в query сохраняются (`QSA`), контент совпадает со страницей 2.
- В Яндекс.Вебмастере/GSC дубли пагинации не растут; глубокие карточки индексируются (при стратегии A).

## ⚠️ Риски
- ⚠️ **`canonical` пагинации на 1-ю страницу может «съесть» индексацию глубоких карточек.** Если товары есть только на страницах 2+, а вы схлопнули их в 1-ю, робот рискует не дойти до этих карточек. Для больших каталогов выбирайте стратегию A (открыть + уникализировать), а не «закрыть всё canonical».
- ⚠️ **Дубль первой страницы (`?PAGEN_1=1`).** Самый частый недосмотр: страница доступна по двум адресам. Нужен 301 (или хотя бы canonical) на чистый URL.
- ⚠️ **SEO-текст категории на всех страницах = дубль контента.** Прятать его CSS бесполезно — для поисковика он всё равно в HTML. Выводите физически только на `page == 1`.
- ⚠️ **`GetCurPage()` для canonical пагинации.** Вернёт базовый путь без учёта параметров — легко получить неверный canonical. Используйте `GetCurPageParam('', ['PAGEN_1'])`.
- **`rel next/prev` — не приоритет.** Google их не учитывает с 2019; не тратьте на них время раньше canonical/301/уникализации. Внедрять — только осознанно под Яндекс/Bing.
- **AJAX «Показать ещё» прячет товары от робота.** Если довеска идёт только JS-ом без реальных страниц/ссылок, часть ассортимента недоступна для индексации. Сохраняйте доступные роботу пагинационные URL (`/page-N/` или `?PAGEN_`) как фолбэк.
- **`QSA` в ЧПУ-пагинации.** Без него `/page-2/` потеряет активный фильтр/сортировку из query — выдача на странице не совпадёт.

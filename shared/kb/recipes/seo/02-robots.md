# Рецепт SEO-02. robots.txt через `\Bitrix\Seo\RobotsFile`

## Цель

Программно сформировать и поддерживать `robots.txt` в корне сайта: закрыть служебные разделы
(`/bitrix/`, системные части `/upload/`, GET-параметры `/*?`), указать директиву `Sitemap:`,
вести правила отдельно для каждого сайта/зеркала и держать разные политики индексации между
production и dev. Базовый API — `\Bitrix\Seo\RobotsFile` (D7), он читает и дополняет существующий
файл, не затирая ручные правки.

## Когда применять

- Нужен воспроизводимый, версионируемый `robots.txt` (часть деплоя/миграции), а не разовая правка
  через «Маркетинг → Поисковая оптимизация → Настройка robots.txt».
- Мультисайт/несколько зеркал: у каждого сайта свой корень и свой файл, правила должны различаться.
- Связка с sitemap: после создания карты сайта нужно автоматически добавить `Sitemap:` в robots.
- ⚠️ Разделение сред: на dev/stage индексацию закрывают полностью (`Disallow: /`), на production
  открывают. Перенос production-`robots.txt` на dev (или наоборот) — частый источник SEO-провала.

Если задача — разово сгенерировать стартовый набор кнопкой, это делается в админке (та же
страница `seo_robots.php`); рецепт нужен, когда правила формируются кодом.

## Шаги

1. **Определить `$siteId`** целевого сайта (двухбуквенный код, например `s1`). `RobotsFile`
   наследует `\Bitrix\Main\IO\File` и сам резолвит путь до `robots.txt` в корне этого сайта.

2. **Создать объект** `new RobotsFile($siteId)`. Если файл уже есть — существующие правила
   читаются, новые добавляются поверх (дубли отсекаются по умолчанию).

3. **Добавить Disallow-правила** служебных разделов через `addRule(['Disallow', '/path/'])`.
   По умолчанию они пишутся в секцию `User-Agent: *`.

4. **Добавить директиву `Sitemap:`** через `addRule([RobotsFile::SITEMAP_RULE, $absoluteUrl])` —
   это отдельный тип правила, а не Disallow. URL — абсолютный, с протоколом и доменом.

5. **Для dev-среды** вместо набора Disallow поставить единственное `Disallow: /` (закрыть всё) и
   `Sitemap:` не добавлять. Ветвление по окружению — на стороне вызывающего кода (env-флаг).

6. **Сохранить** — после `addRule(...)` объект записывает файл сам (наследует механику записи
   `IO\File`). Проверить результат по `https://<домен>/robots.txt`.

## Рабочий сниппет

Файл: `/local/php_interface/seo/robots_setup.php` (запускать из CLI/агента, не из публичной зоны).

```php
<?php
require($_SERVER["DOCUMENT_ROOT"] . "/bitrix/modules/main/include/prolog_before.php");

use Bitrix\Main\Loader;
use Bitrix\Seo\RobotsFile;

Loader::requireModule('seo');

$siteId = 's1';
$domain = 'https://example.ru';            // главное зеркало этого сайта
$isProd = (getenv('APP_ENV') === 'production');

$robots = new RobotsFile($siteId);

if (!$isProd) {
    // ⚠️ dev/stage: закрываем сайт от индексации целиком, sitemap не публикуем
    $robots->addRule(['Disallow', '/']);
    return;
}

// production: служебные разделы и динамика
$disallow = [
    '/bitrix/',          // ядро и админка
    '/upload/',          // системные файлы загрузок (кроме публичных каталогов)
    '/personal/',        // личный кабинет
    '/auth/',            // авторизация/регистрация
    '/search/',          // внутренний поиск
    '/*?',               // любые URL с GET-параметрами (фильтры, сортировки, utm)
    '/*&',               // составные GET-цепочки
    '/index.php$',       // дубль главной через index.php
];
foreach ($disallow as $path) {
    // addRule сам отсекает дубли (третий аргумент $bCheckUnique = true по умолчанию)
    $robots->addRule(['Disallow', $path]);
}

// ⚠️ открыть render-ресурсы внутри /bitrix/ — иначе Google не отрисует страницу
foreach (['/bitrix/*.css', '/bitrix/*.js', '/bitrix/images/', '/bitrix/templates/'] as $allow) {
    $robots->addRule(['Allow', $allow]);
}

// директива Sitemap (тип правила — RobotsFile::SITEMAP_RULE, не Disallow)
$robots->addRule([RobotsFile::SITEMAP_RULE, $domain . '/sitemap.xml']);

// чтение для контроля: какие Sitemap уже прописаны
$sitemaps = $robots->getRules(RobotsFile::SITEMAP_RULE);
```

Замечания по сигнатурам:
- `addRule(array $rule, string $section = '*', bool $bCheckUnique = true)` — `$rule` это пара
  `['Disallow', '/path/']` или `[RobotsFile::SITEMAP_RULE, 'https://…/sitemap.xml']`, сериализуется
  в строку `Disallow: /path/`. По умолчанию правило идёт в `User-Agent: *`.
- `getRules($rule, $section = '*')` — вернуть все правила указанного типа.
- Константы: `RobotsFile::SECTION_RULE = 'User-Agent'`, `RobotsFile::SITEMAP_RULE = 'Sitemap'`.
- Отдельная секция под бота: передайте имя как `$section`, например
  `$robots->addRule(['Disallow', '/*?'], 'Yandex')` → правило попадёт в `User-Agent: Yandex`.

## Выбор API

- `\Bitrix\Seo\RobotsFile` (D7) — единственный поддерживаемый программный интерфейс для robots.txt
  в ядре 26.x. Используйте его и в коде, и понимая, что админ-страница `seo_robots.php` работает
  через этот же класс.
- Директиву `Sitemap:` можно не прописывать вручную: при генерации карты сайта, если в настройках
  записи `b_seo_sitemap` стоит `SETTINGS['ROBOTS'] = 'Y'`, генератор сам добавит `Sitemap:` в
  robots (`Generator::runIndex`). Тогда ручной `addRule([SITEMAP_RULE, …])` избыточен — выбирайте
  один источник истины, чтобы не плодить дубль-строки.
- Главное зеркало (`Host`) и `Crawl-delay` в современной практике задаются ограниченно: `Host`
  Яндексом не используется, приоритет отдан 301-редиректу на канонический домен (см. связку с SEF
  и редиректами). Не закладывайте `Host` как основной механизм склейки зеркал.
- Константа `EMPTY_DISALLOW_RULE` помечена `@deprecated` — не используйте приём с пустым `Disallow`
  для «разрешить всё»; для разрешения применяйте явный `Allow`.
- **`Clean-param` (только Яндекс) для незначащих GET** (`utm_*`, `sort`, `from`, `gclid`) — часто
  лучше `Disallow`: робот и не обходит дубль, и **передаёт его сигналы на основную страницу**
  (`Disallow` сигналы теряет). Синтаксис: `Clean-param: utm_source&utm_medium /catalog/`
  (несколько параметров через `&`, путь — необязательный). Google `Clean-param` **не понимает** —
  для него те же дубли закрывайте `canonical`/`noindex`. Программно — тем же `RobotsFile`:
  `$robots->addRule(['Clean-param', 'utm_source&utm_medium'])`. Детальная стратегия дедупликации —
  в рецепте [SEO-06](./06-duplicates-response-codes.md).

## Проверка

**Режим «только файлы» (без запущенного Битрикс):**
- Запись `RobotsFile` появилась в корне сайта: `test -f <DOCUMENT_ROOT>/robots.txt`.
- В файле есть секция `User-Agent: *` и ожидаемые `Disallow:`-строки; для dev — ровно
  `Disallow: /` и отсутствие `Sitemap:`.
- Строки разделены `\r\n` (формат, который пишет класс) — учитывайте при diff/grep.
- Скрипт-генератор синтаксически валиден: `php -l /local/php_interface/seo/robots_setup.php`.

**Режим «живой Битрикс»:**
- `https://<домен>/robots.txt` отдаёт 200 и актуальное содержимое.
- На dev-зеркале строка `Disallow: /` присутствует, `Sitemap:` отсутствует.
- На production: служебные пути закрыты, `Sitemap:` указывает на существующий
  `https://<домен>/sitemap.xml` (карта отдаёт 200).
- Повторный прогон скрипта не плодит дубли (сработал `$bCheckUnique`).
- Контрольная проверка синтаксиса в Яндекс.Вебмастере / Google Search Console (анализ robots.txt):
  нужные URL разрешены, служебные — запрещены.

## ⚠️ Риски

- **Перенос robots между средами.** Production-`robots.txt` на dev открывает тестовую копию
  индексации (дубли в выдаче); dev-`robots.txt` (`Disallow: /`) на production выбивает сайт из
  индекса целиком. Всегда ветвите по env-флагу и проверяйте `Disallow: /` именно там, где он нужен.
- **`/*?` закрывает полезные GET-страницы.** Маска `/*?` запрещает всё с параметрами, включая,
  например, страницы пагинации или умного фильтра, которые вы хотите индексировать. Соразмеряйте
  с canonical-стратегией: либо закрываем в robots, либо открываем и склеиваем через `rel=canonical`
  (но не одновременно — закрытый в robots URL поисковик не дойдёт до canonical).
- **Дубль `Sitemap:`.** Если карта генерируется с `SETTINGS['ROBOTS']='Y'` и вы дополнительно
  зовёте `addRule([SITEMAP_RULE, …])`, получите две строки. Оставьте один источник.
- **Мультисайт: чужой корень.** Неверный `$siteId` запишет файл в корень другого сайта/зеркала.
  Сверяйте `$siteId` с реальным `DOCUMENT_ROOT` сайта перед записью.
- **robots.txt не защищает данные.** `Disallow` — рекомендация для добросовестных ботов, а не
  ограничение доступа. Закрытие `/personal/` или `/bitrix/` в robots не заменяет проверку прав:
  приватные разделы защищайте авторизацией, а не строкой в robots.
- ⚠️ **Не блокируйте render-ресурсы.** `Disallow: /bitrix/` без `Allow` для `*.css`/`*.js` закрывает
  стили и скрипты ядра — Google не отрисует страницу, страдают индексация и оценка mobile-friendly.
  Всегда добавляйте `Allow: /bitrix/*.css` и `/bitrix/*.js`.
- **robots.txt ≠ деиндексация.** `Disallow` управляет обходом, а не присутствием URL в выдаче.
  Чтобы убрать уже доступную страницу из индекса — `noindex`/`X-Robots-Tag` (рецепт `06`), а не robots.
- **Кириллические домены** в `Sitemap:` указывайте в корректной форме (punycode для домена);
  относительный путь в директиве `Sitemap:` недопустим — только абсолютный URL.

# Рецепт SEO-06: Дубли страниц и коды ответа (soft-404, Clean-param, noindex)

Под-скилл: `1c-bitrix-cms-seo` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x, модули `main` / `iblock` / `seo`.

## Цель
Убрать из индекса дубли и «мусорные» URL и заставить сайт отдавать **правильные HTTP-коды**. Три рычага: (1) починить **soft-404** — по умолчанию Битрикс на несуществующих URL и удалённых элементах отдаёт `200 OK` вместо `404`, и поисковик копит «пустышки» в индексе; (2) схлопнуть **дубли по GET-параметрам** (utm, sort, фильтр, пагинация, sessid) через `rel=canonical`, `Clean-param` (Яндекс) и `noindex`; (3) закрыть служебные/мусорные страницы корректным способом под каждый поисковик. Итог — в индексе только канонические страницы, удалённый товар отдаёт 404, краулинговый бюджет не тратится на дубли.

## Когда применять
- Яндекс.Вебмастер показывает «Дубли страниц с незначащими GET-параметрами» (раздел «Диагностика») или Google Search Console — «Страница является копией. Канонической версии нет».
- Удалили/деактивировали товар, а его URL отдаёт `200` с пустой карточкой (а должен 404) — классический soft-404 Битрикса.
- В индекс попали `/?PAGEN_=`, `/?sort=`, `/?utm_*`, `/index.php`, версии со слэшем и без, в разном регистре, `/catalog/.../filter/clear/apply/`.
- Нужна воспроизводимая (версионируемая) настройка кодов ответа и правил дедупликации, а не разовая правка.

Если задача — только закрыть служебные разделы от обхода, это рецепт `02-robots.md`; здесь — про коды ответа и склейку дублей, где одного `robots.txt` мало.

## Шаги
1. **Почините soft-404 на уровне платформы.** В «Настройки → Настройки продукта → Настройки модулей → Главный модуль» **снимите** галку «Посылать в заголовке статус 200 на 404 ошибку». В корневой `/404.php` первыми строками (до подключения `header.php` шаблона) выставьте статус и константу. Проверьте, что в `.htaccess` есть `ErrorDocument 404 /404.php`.
2. **Включите 404 в компонентах детального просмотра.** В `bitrix:news`/`bitrix:catalog` (и одиночных `news.detail`/`catalog.element`) включите параметры «Устанавливать статус 404» (`SET_STATUS_404=Y`) и «Показ специальной страницы» (`SHOW_404=Y`). Тогда несуществующий/деактивированный элемент отдаёт 404, а не 200 с пустотой.
3. **Определите канонический URL** каждого типа страницы (см. рецепт `04`): база раздела без GET, со слэшем, нижний регистр, без `index.php`. На дубли-вариации ставьте `rel=canonical` на эту базу.
4. **Закройте незначащие GET-параметры под Яндекс через `Clean-param`** в `robots.txt` (utm-метки, `sort`, `from`, `gclid`, `PAGEN_` где уместно). `Clean-param` сильнее `Disallow`: робот и не обходит дубль, и **передаёт его сигналы на основную страницу** (`Disallow` сигналы теряет).
5. **Для Google и точечных случаев** используйте `canonical` (склейка) и/или `noindex` (исключение из индекса) — Google **не понимает `Clean-param`**. Мета-robots `noindex, follow` ставьте через свойство страницы `robots`; на не-HTML ответах (PDF, печать) — заголовок `X-Robots-Tag`.
6. **Найдите остаточные дубли** в Яндекс.Вебмастере («Индексирование → Страницы в поиске», «Диагностика») и GSC («Индексирование → Страницы»); отправьте канонические на переобход. Учтите задержки: алерт о GET-дублях в Вебмастере появляется через 2–3 дня, исключение из индекса по `Clean-param` — 1–2 недели.

## Рабочий сниппет

**`/404.php`** — первые строки до подключения шаблона сайта (отдаёт корректный 404):

```php
<?php
// Первые три строки — критичны для статуса 404 (иначе soft-404 = 200 OK).
include_once($_SERVER['DOCUMENT_ROOT'] . '/bitrix/modules/main/include/urlrewrite.php');
CHTTP::SetStatus("404 Not Found");
@define("ERROR_404", "Y");

require($_SERVER['DOCUMENT_ROOT'] . '/bitrix/header.php');
$APPLICATION->SetTitle("Страница не найдена (404)");
?>
<h1>Страница не найдена</h1>
<p>Возможно, она удалена или адрес введён с ошибкой. <a href="/">На главную</a>.</p>
<?php require($_SERVER['DOCUMENT_ROOT'] . '/bitrix/footer.php'); ?>
```

**`robots.txt`** — `Clean-param` для незначащих GET (Яндекс). Несколько параметров через `&`, опционально префикс пути:

```text
# utm-метки и клиентские параметры — на всём сайте
Clean-param: utm_source&utm_medium&utm_campaign&utm_term&utm_content&from&gclid&yclid&_openstat
# сортировка/режим показа — только в каталоге
Clean-param: sort&order&show_count /catalog/
```

Программно `Clean-param` пишется тем же `RobotsFile` (тип правила — строка в секции, см. рецепт `02`):

```php
use Bitrix\Seo\RobotsFile;
$robots = new RobotsFile($siteId);
$robots->addRule(['Clean-param', 'utm_source&utm_medium&utm_campaign']); // секция User-Agent: *
```

**`noindex` через свойство страницы** (служебная/мусорная страница, например результат поиска или фильтр, который не схлопывается canonical):

```php
// в шаблоне страницы/компонента
$APPLICATION->SetPageProperty('robots', 'noindex, follow');
// в header.php шаблона сайта — вывести метатег:
//   <?php $APPLICATION->ShowMeta('robots'); ?>
```

**`X-Robots-Tag`** для не-HTML ответов (печатная версия, выгрузки) — заголовком до вывода:

```php
header('X-Robots-Tag: noindex, follow', true);
```

## Выбор API
- **Код 404 страницы** — `CHTTP::SetStatus("404 Not Found")` + `@define("ERROR_404","Y")` (legacy, но это штатный и единственный путь для кастомной 404). Поведение «отдавать 200 на 404» управляется настройкой Главного модуля (`error_reporting`/`update_devsrv`? — нет: опция **«Посылать в заголовке статус 200 на 404 ошибку»**); по умолчанию её снимают.
- **404 элемента инфоблока** — параметры компонента `SET_STATUS_404` и `SHOW_404` (плюс `FILE_404` / `MESSAGE_404` для своей страницы/текста). Это штатный механизм комплексных компонентов; вручную статус в шаблоне выставлять не нужно.
- **Склейка дублей — три инструмента под разные поисковики:**
  - `rel=canonical` (`$APPLICATION->SetPageProperty('canonical', …)`) — понимают **и Яндекс, и Google**; основной кросс-поисковый способ (см. рецепт `04`).
  - `Clean-param` — **только Яндекс**, но эффективнее всего для незначащих GET: экономит краулинговый бюджет и передаёт сигналы на основную страницу.
  - `noindex` (мета `robots` / `X-Robots-Tag`) — жёсткое исключение из индекса; понимают оба, но в отличие от `Disallow` робот должен **дойти** до страницы (значит, не закрывайте её ещё и в robots).
- **`Disallow` vs `noindex` vs `canonical` — не дублируйте.** Закрытый в `robots.txt` URL поисковик не обойдёт и **не увидит** ни `canonical`, ни `noindex` на нём. Под склейку выбирайте один механизм на тип дубля.

## Проверка

Проверка рендера и структуры — по общему паттерну: [verification](../../operations.md) (CLI для структуры, браузер/preview для рендера; на dev-стенде).

**Режим «только файлы» (без живого Битрикса):**
- В `/404.php` первыми идут `urlrewrite.php`-include, `CHTTP::SetStatus("404 Not Found")`, `@define("ERROR_404","Y")` — до `bitrix/header.php`. `php -l /404.php` без ошибок.
- В `robots.txt` строки `Clean-param:` синтаксически валидны (≤ 500 символов на директиву; параметры через `&`; путь — необязательный второй аргумент).
- Где стоит `noindex` через свойство `robots` — этот же URL **не** закрыт `Disallow` в robots (иначе noindex не сработает).
- В настройках компонента детального просмотра в коде/экспорте есть `SET_STATUS_404=Y` и `SHOW_404=Y`.

**Режим «живой Битрикс»:**
- `curl -I https://site.ru/nesuschestvuyushchaya-stranica/` → `HTTP/1.1 404 Not Found` (не 200).
- `curl -I https://site.ru/catalog/element-udalyon/` (удалённый элемент) → `404`, а не `200` с пустой карточкой.
- Главная и существующие страницы по-прежнему отдают `200` (фикс не сломал нормальные коды).
- На странице с `?utm_source=…` в исходнике присутствует `rel=canonical` на базовый URL; в Яндекс.Вебмастере «Инструменты → Анализ robots.txt» `Clean-param` распознан без ошибок.
- В `<head>` мусорной страницы есть `<meta name="robots" content="noindex, follow">`; для печатной версии в ответе — заголовок `X-Robots-Tag: noindex`.
- Через 1–2 недели в Вебмастере/GSC число дублей по GET снижается; в «Страницах в поиске» остаются канонические URL.

## ⚠️ Риски
- ⚠️ **soft-404 — тихий и массовый.** Пока `/404.php` отдаёт 200, поисковик считает несуществующие и удалённые страницы «живыми» и копит их в индексе (раздувание, расход краулингового бюджета, каннибализация). Проверяйте код ответа `curl -I`, а не глаза́ми по картинке страницы.
- ⚠️ **`Clean-param` — это Яндекс, не Google.** Для Google те же дубли закрывайте `canonical`/`noindex`. Расчёт только на `Clean-param` оставит дубли в Google.
- ⚠️ **`Disallow` + `canonical`/`noindex` одновременно бесполезны.** Закрытую в robots страницу робот не обойдёт и директиву на ней не прочитает. Выберите: либо `Disallow` (не обходить), либо `canonical`/`noindex` (обойти и склеить/исключить).
- ⚠️ **`/404.php` без статуса = маскировка ошибки.** Файл может рендерить «красивую» страницу 404 и при этом отдавать 200 — самый частый недосмотр. Три строки сверху обязательны, даже если включена настройка Главного модуля (страховка).
- **Дубль «товар в нескольких разделах».** Один элемент, привязанный к нескольким разделам, доступен по нескольким URL с разными хлебными крошками. Лечится `canonical` на основной раздел (`DETAIL_PAGE_URL` от основного раздела) — см. рецепт `04`/`03`.
- **`index.php`, слэш, регистр.** `/`, `/index.php`, `/Catalog/` vs `/catalog/` — разные URL для поисковика. Нормализуйте 301-редиректом в `.htaccess`/nginx (рецепт `04`), не только canonical.
- **`/clear/apply/` умного фильтра** — дубль базовой категории; закрывается в robots, а не 301 (301 сломает кнопку сброса фильтра) — см. рецепт `07`.

# Рецепт SEO-10: hreflang, мультиязычность и мультирегиональность

Под-скилл: `1c-bitrix-cms-seo` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x, мультисайт/языки модуля `main`.

## Цель
Корректно показать поисковику языковые и региональные версии сайта: для **разных языков** — связать версии тегами `hreflang` (HTML / HTTP-заголовок / sitemap) с взаимными ссылками, self-reference и `x-default`; для **одного языка и разных городов** (мультирегиональность РФ) — не путать с `hreflang`, а вести отдельные URL и привязывать регион в Яндекс.Вебмастере. Итог — пользователь и бот попадают на нужную версию, версии не конкурируют как дубли, регион/язык определяется явно, а не «угадыванием по IP».

## Когда применять
- Сайт на нескольких языках (ru/en/…): мультисайт Битрикс, поддомены или папки `/en/`.
- Мультирегиональный магазин (один язык, разные города): поддомены/папки `#CITY_DIR#`, разные цены/наличие/контакты.
- В выдаче не та языковая версия, или региональные поддомены конкурируют между собой как дубли.

Если задача — просто склеить дубли в пределах одной версии, это рецепты `04`/`06`. Здесь — про связывание **разных** версий между собой.

## Шаги
1. **Сначала разведите сущности — это ключевое.** `hreflang` — для **языков** (и языка+страны: `ru`, `en`, `de-AT`), **не для городов одного языка**. «Москва/СПб/Казань» на русском — это **не** `hreflang`: для них делают отдельные URL (поддомен/папка) + привязку региона в Яндекс.Вебмастере + уникализацию контента (адрес, телефон, цены). Путаница «город = hreflang» — частая ошибка.
2. **Дайте каждой версии собственный URL.** Отдельные домены/поддомены/папки (в Битрикс — мультисайт и языки, либо мультирегиональные поддомены с макросом `#CITY_DIR#`). ⚠️ Не делайте locale-adaptive: один URL с подменой контента по IP/`Accept-Language`. Googlebot ходит без `Accept-Language`, а его IP воспринимаются как США → региональный/языковой контент он может не увидеть и не проиндексировать.
3. **Свяжите языковые версии `hreflang`.** Каждая версия ссылается на **все** альтернативы **и на саму себя** (self-reference); ссылки **взаимные** (A→B обязывает B→A); URL — абсолютные. Добавьте `x-default` для языкового переключателя/версии по умолчанию. Метод — один из трёх (не дублировать): HTML-теги в `header.php`, HTTP-заголовок `Link`, или блоки `xhtml:link` в sitemap.
4. **Для мультирегиональности (один язык) настройте регион, а не `hreflang`.** В Яндекс.Вебмастере для каждого поддомена/папки задайте регион («Информация о сайте → Региональность»). Контент региональных версий должен реально отличаться (НП-данные, цены, наличие), иначе это дубли. Canonical региональной версии — на себя (self-canonical), не на «головной» домен.
5. **Согласуйте с sitemap и robots.** Региональные/языковые URL должны быть доступны для обхода (не закрыты в robots), присутствовать в sitemap своими каноническими адресами. Макрос `#CITY_DIR#` в карте сам не разворачивается — карту под мультирегион собирают отдельно по сайтам/доменам (см. рецепт `01`).

## Рабочий сниппет

**HTML hreflang** в `header.php` шаблона сайта (внутри `<head>`), для языковых версий одной страницы:

```php
<?php if (!defined("B_PROLOG_INCLUDED") || B_PROLOG_INCLUDED !== true) die();
// $alternates: ['ru' => 'https://example.ru/catalog/item/',
//               'en' => 'https://example.com/en/catalog/item/']
// Готовится в шаблоне/компоненте из соответствия языковых версий.
foreach ($alternates as $lang => $url) {
    echo '<link rel="alternate" hreflang="' . htmlspecialcharsbx($lang)
       . '" href="' . htmlspecialcharsbx($url) . '">' . "\n";
}
// x-default — версия по умолчанию (языковой переключатель/главная локаль):
echo '<link rel="alternate" hreflang="x-default" href="'
   . htmlspecialcharsbx($alternates['ru']) . '">' . "\n";
?>
```

**hreflang в sitemap** (альтернатива HTML; каждый `<url>` перечисляет все версии, включая себя):

```xml
<url>
  <loc>https://example.ru/catalog/item/</loc>
  <xhtml:link rel="alternate" hreflang="ru" href="https://example.ru/catalog/item/"/>
  <xhtml:link rel="alternate" hreflang="en" href="https://example.com/en/catalog/item/"/>
  <xhtml:link rel="alternate" hreflang="x-default" href="https://example.ru/catalog/item/"/>
</url>
```

(корневой тег sitemap должен объявлять `xmlns:xhtml="http://www.w3.org/1999/xhtml"`)

## Выбор API
- **hreflang — это разметка/заголовки, а не модуль.** В Битрикс его собирают в шаблоне сайта (`header.php` / `AddHeadString`) из соответствия языковых версий (мультисайт, языки, привязка элементов-переводов) либо генерируют в sitemap. Готового «переключателя hreflang из коробки» нет — соответствие версий ведёте сами (через свойства/привязки или внешний модуль мультиязычности).
- **Три метода равнозначны для Google** (HTML / HTTP `Link` / sitemap) — выбирайте один. HTML удобен для небольшого числа версий; sitemap — для крупных мультирегиональных сайтов (не раздувает `<head>`); HTTP-заголовок — для не-HTML (PDF).
- **Мультирегион одного языка — не `hreflang`, а региональность.** Яндекс: регион сайта в Вебмастере. Google: city-уровня геотаргетинга нет (только страна — через `hreflang` с кодом страны или ccTLD); города различайте отдельными URL + уникальным контентом, не `hreflang`.
- **Canonical и hreflang не противоречат:** каждая версия — self-canonical; `hreflang` связывает версии между собой. Нельзя ставить canonical языковой версии на другую языковую версию — это «схлопнет» перевод (см. рецепт `04`).

## Проверка

Проверка рендера и структуры — по общему паттерну: [verification](../../operations.md) (CLI для структуры, браузер/preview для рендера; на dev-стенде).

**Режим «только файлы» (без живого Битрикса):**
- `hreflang`-теги абсолютные, с self-reference и `x-default`; коды языков валидны (`ru`, `en`, `de-AT`), не выдуманы (нет `hreflang="ru-msk"` под город).
- Выбран **один** метод (HTML *или* sitemap *или* HTTP), без дублирования.
- В sitemap с hreflang корневой тег объявляет namespace `xhtml`. `php -l` шаблонов без ошибок.
- Для мультирегиона: региональные версии — self-canonical (не на головной домен), контент реально различается.

**Режим «живой Битрикс»:**
- В исходнике страницы каждой версии — `hreflang` на все альтернативы и на себя; на парной версии ссылки взаимны.
- Яндекс.Вебмастер / Google Search Console: отчёт по `hreflang`/международному таргетингу без ошибок «нет обратной ссылки» / «нет self-reference».
- Региональные поддомены имеют разный регион в Яндекс.Вебмастере и не отмечены как дубли друг друга.
- `curl -I` региональной/языковой версии без `Accept-Language` отдаёт её собственный контент (не редирект на головную по IP).

## ⚠️ Риски
- ⚠️ **Город ≠ `hreflang`.** `hreflang` — для языков и стран, не для городов одного языка. Региональные поддомены связывают регионом в Вебмастере и уникальным контентом, а не `hreflang` (иначе теги невалидны и игнорируются).
- ⚠️ **Locale-adaptive (один URL, контент по IP) — антипаттерн.** Googlebot без `Accept-Language` и с «американским» IP не увидит другие языки/регионы. Нужны отдельные URL на версию.
- ⚠️ **Невзаимные/без self-reference hreflang игнорируются.** Если A ссылается на B, а B на A — нет, связка не работает. Каждая версия должна ссылаться на все альтернативы **и на себя**.
- ⚠️ **Canonical поверх hreflang «съедает» перевод.** Canonical языковой/региональной версии на другой домен убирает её из индекса. Каждая версия — self-canonical.
- **Дубли региональных версий.** Поддомены городов с одинаковым контентом — дубли. Уникализируйте НП-данные, цены, наличие, тексты.
- **`#CITY_DIR#` в sitemap не разворачивается** — карту мультирегиона собирают по доменам/сайтам отдельно (рецепт `01`).

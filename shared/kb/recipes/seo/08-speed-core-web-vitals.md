# Рецепт SEO-08: Скорость и Core Web Vitals (композит, кэш, картинки)

Под-скилл: `1c-bitrix-cms-seo` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x, модули `main` (Composite, кэш), `main`/`iblock` (картинки), `perfmon`.

## Цель
Поднять скорость и пройти Core Web Vitals (LCP / INP / CLS) штатными рычагами Битрикс: **композитный сайт** (мгновенная отдача статической части + асинхронная динамика), управляемый/HTML-кэш и внешнее хранилище кэша (memcached/Redis), CDN для статики, нарезка и облегчение картинок, чистый клиентский JS. Цель — быстрый «зелёный» сайт, который поисковики учитывают в ранжировании, без потери персонализации и корректности.

## Когда применять
- PageSpeed Insights / Lighthouse / Яндекс.Вебмастер показывают проблемы с LCP/INP/CLS (не проходят пороги) или низкий индекс производительности Битрикс.
- Симптомы медленной отдачи: высокий TTFB, тяжёлые страницы каталога, долгая первая отрисовка.
- Готовите запуск/редизайн и хотите заложить кэш-стратегию и оптимизацию картинок сразу правильно.

Если узкое место — конкретные медленные SQL/хиты, начните с **Монитора производительности** (perfmon, шаг 6): сначала измерьте, потом оптимизируйте.

## Шаги
1. **Включите композитный сайт.** «Настройки → Настройки продукта → Композитный сайт» (или на BitrixVM `bx-sites … -a composite --enable`). Композит кэширует статическую часть страницы (отдаётся мгновенно, ~десятки мс), а динамику (корзина, авторизация, счётчики) догружает фоновым AJAX-запросом. Резко улучшает LCP/FCP для анонимных посетителей.
2. **Вынесите персонализацию в динамические области.** Всё, что различается для пользователей (имя в шапке, число товаров в корзине, «недавно просмотренные», формы с CSRF-токеном), оберните в динамическую область композита — иначе закэшированная страница первого посетителя покажется всем (утечка чужой корзины/сессии). В шаблоне компонента: `$frame = $this->createFrame()->begin(); … $frame->end();`; вне компонента — `\Bitrix\Main\Composite\StaticArea::startDynamicArea()`.
3. **Настройте кэширование.** Управляемый кэш (`cache_dir`), HTML-кэш компонентов, разумный TTL (часто `86400` вместо `3600`). Хранилище кэша вынесите в **memcached/Redis** (`bitrix/.settings.php`, секция `cache`) — снимает нагрузку с диска. Используйте тегированный кэш, чтобы точечно сбрасывать кэш при изменении данных, а не «всё подряд».
4. **Подключите CDN.** «Настройки → Облако 1С-Битрикс → Ускорение сайта (CDN)» — статика (css/js/изображения) раздаётся с ближайшего к посетителю узла, разгружает сервер и ускоряет загрузку ресурсов.
5. **Оптимизируйте картинки (LCP + CLS + вес).**
   - Не выводите оригиналы «как есть» — нарезайте под нужный размер `CFile::ResizeImageGet($id, ['width'=>800,'height'=>800], BX_RESIZE_IMAGE_PROPORTIONAL)`.
   - Сжимайте и отдавайте **WebP/AVIF** (модули типа `dev2fun.imagecompress`/серверная конвертация — ядро автоконвертацию в WebP не делает).
   - **`loading="lazy"`** на изображения ниже первого экрана; LCP-картинку (герой/первое фото) — наоборот, грузить сразу (`fetchpriority="high"`, без lazy).
   - Всегда задавайте `width`/`height` (или `aspect-ratio`) — иначе изображения «прыгают» при загрузке и портят **CLS**.
6. **Измерьте и почистите бэкенд.** Модуль **«Монитор производительности» (perfmon)**: индекс производительности, тест скорости, разбор медленных SQL и долгих хитов, проверка настроек окружения. Поднимите PHP до поддерживаемой версии (**минимум 8.2 c 01.02.2026, рекомендуется 8.4+**) с включённым OPcache; учтите, что ядро с 24.0 — **UTF-8 only** (MySQL ≥ 8.0). Хранилище кэша/сессий — Redis/memcached. Обновите BitrixVM (актуальная 9.x несёт свежий стек nginx/Redis/PHP), отключите неиспользуемые модули.
7. **Снизьте INP (бывший FID).** INP меряет отзывчивость на действия пользователя — это про клиентский JS. Минифицируйте и объединяйте CSS/JS (главный модуль), убирайте лишний/дублирующий jQuery и тяжёлые сторонние скрипты, грузите некритичный JS с `defer`/`async`.
8. **Сожмите ответы и отдавайте предсжатую статику.** Включите объединение и минификацию CSS/JS (главный модуль) и создание **предсжатых** копий `.min.css`/`.min.js` — тогда nginx отдаёт готовые сжатые файлы, не тратя CPU на лету. На сервере включите **Brotli** для статики (gzip как fallback), **HTTP/2** (минимум) или HTTP/3, `expires` + `Cache-Control: public, immutable` для статики. Композитный кэш можно отдавать **через nginx**, а не PHP, — это заметно снижает TTFB под нагрузкой.
9. **Соблюдайте регламент деплоя.** После выката шаблонов/компонентов: очистить кэш компонентов → **сбросить вычисленные SEO-значения** (`clearValues()`, рецепт `03`) → прогреть ключевые листинги/карточки → пересобрать композитный кэш → проверить заголовки (код ответа, `canonical`, `robots`, `cache-control`) → проверить LCP/INP/CLS → пересобрать sitemap и отдать на переобход. Пропуск сброса SEO-кэша и проверки заголовков делает релиз «визуально успешным», но вредным для SEO.

## Рабочий сниппет

Динамическая область композита в шаблоне компонента (персональный блок не кэшируется со статикой):

```php
<?php
// /local/templates/<tpl>/components/bitrix/main.user.link/<tpl>/template.php (пример)
if (!defined("B_PROLOG_INCLUDED") || B_PROLOG_INCLUDED !== true) die();

$frame = $this->createFrame()->begin("");   // старт динамической области композита
?>
<span class="header-user">
    <?= $arResult["USER"]["NAME"] ?? "Войти" ?> · корзина: <?= $arResult["CART_COUNT"] ?? 0 ?>
</span>
<?php
$frame->end();                               // конец динамической области
```

Нарезка + lazy-load изображения списка (CLS-safe: заданы размеры):

```php
<?php
$src = CFile::ResizeImageGet(
    $item["PREVIEW_PICTURE"],
    ["width" => 400, "height" => 400],
    BX_RESIZE_IMAGE_PROPORTIONAL,
    true
);
?>
<img src="<?= $src["src"] ?>"
     width="<?= $src["width"] ?>" height="<?= $src["height"] ?>"
     loading="lazy" alt="<?= htmlspecialcharsbx($item["NAME"]) ?>">
```

Внешнее хранилище кэша в `bitrix/.settings.php` (memcached):

```php
'cache' => [
    'value' => [
        'type' => 'memcache',
        'memcache' => ['host' => '127.0.0.1', 'port' => '11211'],
        'sid' => $_SERVER['DOCUMENT_ROOT'] . '#01',
    ],
    'readonly' => false,
],
```

Сжатие и кэширование статики на nginx (Brotli + gzip fallback, длинный `Cache-Control`):

```nginx
# текстовые ответы: Brotli (если модуль собран) + gzip fallback
brotli on;  brotli_comp_level 5;
brotli_types text/css application/javascript application/json image/svg+xml;
gzip on;    gzip_comp_level 5;
gzip_types  text/css application/javascript application/json image/svg+xml;

# статика с хэшем в имени — кэшируем «навсегда»
location ~* \.(css|js|woff2?|jpe?g|png|webp|avif|svg)$ {
    expires 30d;
    add_header Cache-Control "public, immutable";
    access_log off;
}
```

## Выбор API
- **Композит — `\Bitrix\Main\Composite\Engine`** (управляется через настройки/`bx-sites`), динамические области — **`$this->createFrame()`** (метод компонента, D7) или **`\Bitrix\Main\Composite\StaticArea::startDynamicArea()`** вне компонента. Это и есть штатный механизм; руками собирать AJAX-догрузку не нужно.
- **Картинки — `CFile::ResizeImageGet`** (legacy, но штатный путь к нарезке; D7-обёртка для ресайза в публичке не нужна). WebP/AVIF и пакетное сжатие — модулем Marketplace или на уровне сервера; ядро не конвертирует автоматически.
- **Кэш — управляемый кэш + теги**; хранилище (`File`/`memcache`/`redis`) переключается в `.settings.php`. Для своих данных — `\Bitrix\Main\Data\Cache` / `TaggedCache`, чтобы инвалидировать точечно.
- **Измерение — perfmon** (Монитор производительности): индекс, тест, медленные запросы. Сначала измерение, потом оптимизация — иначе «оптимизируете» не то.
- **Core Web Vitals по рычагам:** LCP ← композит/CDN/быстрый сервер + приоритет hero-картинки; INP ← меньше и легче JS, `defer`; CLS ← размеры картинок/баннеров, `font-display`, без «прыжков» layout.
- **Транспорт и сжатие — на сервере, не в PHP.** Минификация/объединение и предсжатые `.min.*` — главный модуль; Brotli/gzip, HTTP/2-3, `Cache-Control`, отдача композита — nginx. ⚠️ HTTP/3 у nginx помечен experimental — включайте осознанно.

## Проверка

Проверка рендера и структуры — по общему паттерну: [verification](../../operations.md) (CLI для структуры, браузер/preview для рендера; на dev-стенде).

**Режим «только файлы» (без живого Битрикса):**
- Персональные блоки в шаблонах обёрнуты в `createFrame()` / `StaticArea` (не уедут в статический кэш композита).
- Картинки выводятся через `CFile::ResizeImageGet` с заданными `width`/`height` в `<img>`; ниже первого экрана — `loading="lazy"`. `php -l` шаблонов без ошибок.
- В `.settings.php` хранилище кэша указано корректно (если внедряется memcached/Redis).
- Правки лежат в `/local/`, дистрибутив не тронут.

**Режим «живой Битрикс»:**
- Композит включён: в исходнике страницы для анонимного посетителя присутствуют маркеры композита, персональные блоки приходят отдельным запросом (видно в Network как composite/AJAX).
- Под разными пользователями имя/корзина в шапке корректны (персонализация не «застряла» из кэша).
- PageSpeed Insights / Lighthouse: LCP, INP, CLS улучшились; в Яндекс.Вебмастере «Скорость загрузки» — рост.
- perfmon: индекс производительности вырос, в списке медленных запросов нет очевидных тяжёлых хитов.
- Картинки отдаются нарезанными (не оригиналы), при наличии — в WebP; CLS близок к 0 (layout не прыгает при загрузке изображений).

## ⚠️ Риски
- ⚠️ **Композит без динамических областей = утечка персонализации.** Если корзину/авторизацию/формы не вынести в `createFrame()`, закэшированная страница первого посетителя (с его корзиной и именем) покажется всем. Это и приватность, и битый UX. Всегда оборачивайте персональное.
- ⚠️ **Композит «глотает» серверные события на закэшированной отдаче.** При отдаче из композита PHP-логика в `OnProlog`/`OnPageStart` не выполняется — редиректы (в т.ч. 301 SEO-модулей) и условная логика могут не сработать. Критично для умного фильтра (рецепт `07`): держите редиректы там, где композит их не съедает (`.htaccess`/init до композита), и проверяйте коды ответа на боевом.
- ⚠️ **Формы с `bitrix_sessid()` под композитом.** CSRF-токен в статическом кэше протухает; формы, не вынесенные в динамику, начнут отбивать отправку. Формы — в динамическую область.
- ⚠️ **Переполнение кэша комбинациями.** Кэш фильтра/параметрических страниц растёт лавинообразно; ограничивайте кэшируемые варианты и чистите по тегам, а не копите бесконечно (см. рецепт `07`).
- **Оригиналы картинок убивают LCP и вес.** Вывод полноразмерных изображений в списках — частая причина тяжёлых страниц; нарезка обязательна.
- **CLS из-за картинок без размеров и поздних баннеров.** Без `width`/`height`/`aspect-ratio` и при вставке контента сверху layout «прыгает» — CWV проседает даже на быстром сервере.
- **Сначала измерить.** Глобальная «оптимизация всего» без perfmon/Lighthouse тратит время не туда; находите узкое место данными.
- ⚠️ **Не блокируйте CSS/JS в robots.txt.** Закрытые `/bitrix/*.css|*.js` ломают рендеринг страницы для Google (он не «увидит» вёрстку) → проседают и индексация, и оценка mobile-friendly. Открывайте render-ресурсы явным `Allow` (рецепт `02`).
- **Мобильная версия — тот же контент.** При mobile-first индексации мобильный шаблон должен нести тот же основной контент, `title`/`description`, `alt` и `robots`, что и десктоп; «урезанная» мобильная версия теряет в индексе.

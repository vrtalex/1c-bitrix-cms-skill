# Рецепт Security-04: Аудит безопасности проекта

Под-скилл: `1c-bitrix-cms-security` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x, модуль `security`.

## Цель
Провести воспроизводимый аудит безопасности проекта на Битрикс: пройти по чек-листу настроек проактивной защиты и правил безопасной разработки, прогнать статические проверки кода скриптом `../../../scripts/check-conventions.sh`, разобрать пометки `[REVIEW]` вручную и зафиксировать результат через раздел безопасности «Монитора качества». Итог — список подтверждённых нарушений с приоритетом и план их закрытия.

## Когда применять
- Перед сдачей проекта или выкладкой на боевой сервер — финальная проверка готовности.
- Приняли чужой проект (передача от другого подрядчика) и нужно оценить текущий уровень защиты.
- Регулярный плановый аудит работающего сайта — ориентир раз в месяц, плюс обязательно перед и после крупного обновления продукта.
- После инцидента или срабатывания веб-антивируса / журнала вторжений.

Если нужно только настроить отдельный механизм защиты (WAF, OTP, сессии) — это отдельные рецепты домена; здесь — сводная проверка, что всё на месте и в коде нет известных дыр.

## Шаги
1. **Сделайте бэкап перед аудитом.** Любые изменения настроек безопасности делайте после полной резервной копии сайта и БД — это база, к которой можно откатиться.
2. **Пройдите чек-лист настроек** (см. блок ниже). Сверяйтесь с Панелью безопасности: Настройки → Проактивная защита → Панель безопасности — она показывает текущий уровень и какие параметры понижают защиту.
3. **Прогоните статику по коду:** `sh ../../../scripts/check-conventions.sh <корень_проекта>`. Скрипт даёт два типа сигналов: `[FAIL]` — надёжные нарушения, `[REVIEW]` — эвристика для ручной проверки. ⚠️ Скрипт — best-effort grep: его проверки XSS/SQLi/CSRF видят только sink-и с пользовательскими данными в одной строке и пропускают ввод через промежуточную переменную, поэтому чистый прогон НЕ доказывает отсутствие уязвимостей. Для этих классов обязателен тест статического анализа уязвимостей Битрикс (шаг 5) и/или ручной просмотр потока данных.
4. **Разберите каждый `[REVIEW]` вручную.** Эвристика не выносит вердикт — например, обработку `$_POST`/`$_GET` без видимого `check_bitrix_sessid` нужно прочитать глазами: действительно ли это изменяющее действие без защиты от CSRF.
5. **Запустите Сканер безопасности и раздел безопасности «Монитора качества»** (Настройки → Инструменты → Контроль качества). Детальная работа с Монитором — в под-скилле `1c-bitrix-cms-quality`.
6. **Зафиксируйте результат:** список подтверждённых нарушений с приоритетом (критичные ⚠️ — в первую очередь), ответственный и срок. Запланируйте следующий аудит.

### Чек-лист аудита
**Обновления и резервные копии**
- [ ] Установлены все обновления безопасности; ядро на поддерживаемой версии (актуальная на момент сбора — 26.400.0).
- [ ] Автоматическое резервное копирование работает, восстановление периодически проверяется.

**Проактивная защита (уровень)**
- [ ] Уровень защиты в Панели безопасности не ниже целевого; ни один параметр не понижает уровень.
- [ ] Проактивный фильтр (WAF) включён, исключений нет.
- [ ] «Журнал вторжений за последние 7 дней» = 0; «Журнал заражений за последние 7 дней» = 0.
- [ ] Контроль активности, CAPTCHA при регистрации, защита админраздела по IP, контроль целостности — включены согласно целевому уровню.
- [ ] Сессии хранятся в БД, включена смена идентификатора сессии; OTP для администраторов.

**Правила безопасной разработки (код)**
- [ ] Права пользователя проверяются на сервере для всех чувствительных операций.
- [ ] Пользовательские данные экранируются при выводе: `htmlspecialcharsbx()` / `HtmlFilter::encode()` (HTML), `CUtil::JSEscape()` / `Json::encode()` (JS); входящий HTML очищается `CBXSanitizer`.
- [ ] ⚠️ URL в `href`/`src` из пользовательских данных проходят проверку схемы по белому списку (`http`/`https`/`mailto` через `Uri::getScheme()`); `javascript:`/`data:`/`vbscript:` и `//host` отклоняются — `htmlspecialcharsbx()` их НЕ блокирует.
- [ ] Изменяющие действия защищены **одновременно**: метод POST (`Context::getRequest()->isPost()`), `check_bitrix_sessid()` и серверная проверка прав; формы содержат `bitrix_sessid_post()`. ⚠️ `bitrix_sessid_get()` (токен в URL) для разрушающих действий не используется; `check_bitrix_sessid()` без проверки POST неполон (проходит на GET).
- [ ] ⚠️ Нет `eval(base64(...))`, нет жёстко зашитых секретов/паролей в коде.
- [ ] ⚠️ Ручной секрет-скан репозитория: пароли БД, API-ключи, токены (grep по `password`/`secret`/`api_key`/`token`/приватным ключам, проверка истории git) — секреты держать в `.settings_extra.php` вне публичного VCS.
- [ ] ⚠️ В VCS не закоммичены `bitrix/.settings.php` / `dbconn.php` с боевыми кредами; проверить вручную (.gitignore не гарантирует, что файл не попал ранее).
- [ ] ⚠️ В `dbconn.php` не используется `$DBDebug = true` (показ SQL-ошибок всем посетителям); допустимо `$DBDebug = code` (только администраторам).
- [ ] Числа приводятся к типу (`intval()`), строки в сыром SQL экранируются (`$DB->ForSql()` или `SqlHelper::forSql()`); приоритет — ORM (`DataManager`).

**Контроль целостности и поиск следов компрометации (IOC)**

Пост-эксплуатация массовых атак на Битрикс модифицирует файлы, исполняющиеся на каждом
запросе, и сбрасывает веб-шеллы. Сверяйте при аудите (детали и сигнатуры — в рецепте
`05-threat-intel`):

- [ ] Корневой `index.php` не подменён; нет инъекций кода в `prolog_after.php`
  (`/bitrix/modules/main/include/`), `bx_root.php` (`/bitrix/modules/main/`),
  `security_file_verifier.php` (`/bitrix/admin/`).
- [ ] Файл `/bitrix/.settings.php` на месте (его удаление — известный приём
  пост-эксплуатации).
- [ ] В списке «Агенты» (`/bitrix/admin/agent_list.php`) нет вредоносных агентов с
  `eval()` в теле.
- [ ] Нет веб-шеллов со случайными недиктионарными именами по шаблону
  `[0-9a-f]{12}.php` в `/bitrix/admin/`, в корне сайта и в `/bitrix/docs/`.
- [ ] Сверка контрольных сумм ядра пройдена (контроль целостности проактивной защиты);
  при обнаружении внедрённого кода данные целостности собираются заново.

**Аудит обработки ПДн (152-ФЗ)**

- [ ] В Настройки → Настройки продукта → Соглашения созданы соглашения с текстом
  согласия на обработку ПДн (отдельный документ, не политика конфиденциальности).
- [ ] Каждая форма, собирающая ПДн, привязана к соглашению через компонент
  `bitrix:main.userconsent.request` перед кнопкой отправки.
- [ ] Полученные согласия фиксируются в журнале «Полученные согласия» (контекстное меню
  соглашения).
- [ ] Нет преднажатых галочек согласия: галочка по умолчанию снята, согласие требует
  активного действия пользователя.

## Рабочий сниппет
Статический прогон по коду проекта (режим «только файлы», живой Битрикс не нужен):

```sh
# из каталога с рецептом — путь к скрипту относительный;
# в реальном запуске укажите корень проверяемого проекта
sh ../../../scripts/check-conventions.sh /path/to/project
```

Что ловит скрипт ([FAIL] влияет на код выхода, [REVIEW] — нет):

| Проверка | Тип | Что значит |
|---|---|---|
| `eval(base64(...))` вне ядра | `[FAIL]` | ⚠️ Признак внедрённого вредоносного кода в проекте. |
| Короткий тег `<?` в `/local` | `[FAIL]` | Риск выдачи исходника как текста при выключенном `short_open_tag`; использовать `<?php`. |
| Файлы в каталогах ядра | `[REVIEW]` | Кастом держать в `/local`, иначе обновление перетрёт правки. |
| `$_REQUEST/$_POST/$_GET` без `bitrix_sessid` | `[REVIEW]` | Прочитать глазами: изменяющее действие без защиты от CSRF? |
| Найден эталон контрольных сумм | `[REVIEW]` | Сверку целостности ядра выполнить отдельным шагом. |

Разбор `[REVIEW]` вручную — пример проверки CSRF в найденном файле:

```php
<?php
// Изменяющее действие ОБЯЗАНО проверять метод POST + sessid + права до выполнения.
// check_bitrix_sessid() проходит и на GET, поэтому проверка POST обязательна.
use Bitrix\Main\Context;

if (Context::getRequest()->isPost() && check_bitrix_sessid() && $USER->IsAdmin()) {
    $id = (int)$_POST['ID'];   // данные приводим к типу
    // ... выполняем действие
}
// Нарушение, если: нет проверки POST-метода; нет check_bitrix_sessid();
// токен передан через bitrix_sessid_get() (URL → утечка через Referer/логи);
// либо действие меняет данные без серверной проверки прав.
```

## Проверка
**Режим «только файлы»:**
- Скрипт `../../../scripts/check-conventions.sh` отработал; все `[FAIL]` устранены (код выхода 0).
- Каждая строка `[REVIEW]` разобрана и помечена: ложное срабатывание или подтверждённое нарушение.
- В коде нет `eval(base64(...))`, нет секретов в открытом виде, в `dbconn.php` нет `$DBDebug = true`.

**Режим «живой Битрикс»:**
- Панель безопасности (Настройки → Проактивная защита → Панель безопасности) показывает целевой уровень; счётчики журналов вторжений и заражений за 7 дней = 0.
- Сканер безопасности (Настройки → Проактивная защита) прогнан, критичные замечания закрыты.
- Раздел безопасности «Монитора качества» (Настройки → Инструменты → Контроль качества) пройден.
- Сформирован список подтверждённых нарушений с приоритетом и назначена дата следующего аудита.

## ⚠️ Риски
- ⚠️ **`$DBDebug = true` в `dbconn.php` раскрывает тексты SQL-ошибок всем посетителям** — это утечка структуры БД и запросов. Допустимо только `$DBDebug = code` (показ администраторам) или отключение.
- ⚠️ **`eval(base64(...))` и секреты в коде — прямой признак компрометации/утечки.** При обнаружении внедрённого кода после устранения обязательно соберите данные контроля целостности заново и проверьте журнал заражений.
- ⚠️ **«Зелёный» статический прогон не равен безопасности.** `check-conventions.sh` — best-effort эвристика на grep: его проверки SQLi/XSS/CSRF ловят только sink-и с пользовательскими данными в той же строке (inline-adjacent) и ПРОПУСКАЮТ ввод, прошедший через промежуточную переменную (`$x = $_GET['id']; ... query($x)`). Поэтому чистый прогон — НЕ доказательство отсутствия уязвимостей. Для классов XSS/SQLi/CSRF обязательны статический анализатор уязвимостей Битрикс (Монитор качества → тест статического анализа) и/или ручной просмотр потока данных. Скрипт также не видит логические дыры, отсутствие проверки прав внутри ветвлений и небезопасные include по пользовательским данным. Чек-лист и ручной разбор `[REVIEW]` обязательны.
- **Включение защиты админраздела по IP без добавления собственного IP заблокирует вас самих** — сначала внесите свой IP, потом включайте.
- **Разовый аудит устаревает.** Новые обновления, новый код и новые сотрудники меняют картину — без регулярности (раз в месяц + вокруг обновлений) ценность падает.
- **Изменения настроек без бэкапа необратимы при ошибке** — резервная копия до аудита, а не после.

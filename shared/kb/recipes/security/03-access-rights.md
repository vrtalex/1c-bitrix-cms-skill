# Рецепт Security-03: Права доступа

Под-скилл: `1c-bitrix-cms-security` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x, модуль `main`.

## Цель
Корректно проверять права доступа в собственном коде «1С-Битрикс: Управление сайтом»: разграничение по группам пользователей и задачам/уровням, проверка прав на модуль (`$APPLICATION->GetGroupRight`), на инфоблоки (символьные уровни D/R/W/X и `CIBlockRights`), на файлы и папки. Цель — серверная проверка прав на каждую чувствительную операцию по принципу наименьших привилегий, чтобы изменяющее действие не выполнялось без подтверждённого права текущего пользователя.

## Когда применять
- Пишете AJAX-обработчик, контроллер или страницу, выполняющую создание/изменение/удаление данных, и нужно убедиться, что у пользователя есть на это право.
- Разграничиваете доступ к разделу/функции по группам пользователей или по правам модуля.
- Программно создаёте инфоблок и задаёте групповые права на чтение/запись его элементов.
- Закрываете административную функцию: она не должна работать без аутентификации и проверки прав.
- Нужен принцип наименьших привилегий: контентному редактору — права на свой инфоблок, а не на админку целиком.

Для простой проверки «это администратор?» достаточно `$USER->IsAdmin()`. Программный путь ниже нужен, когда право зависит от группы, модуля или конкретного инфоблока.

## Шаги
1. **Идентифицируйте пользователя на сервере.** Всегда работайте через `global $USER` (legacy) или `\Bitrix\Main\Engine\CurrentUser::get()` (две поддерживаемые версии API). Не полагайтесь на скрытые поля формы или клиентскую валидацию — их подделывают.
2. **Выберите уровень проверки** под операцию:
   - администратор сайта → `$USER->IsAdmin()`;
   - членство в группе → `$USER->GetUserGroupArray()` (вернёт массив ID групп, включая группу «все пользователи» = 2);
   - право на модуль → `$APPLICATION->GetGroupRight("iblock")` (символ `D/R/W/X` или числовой уровень);
   - право на конкретный инфоблок → `CIBlock::GetPermission($iblockId)` / `CIBlockRights`;
   - операция → `$USER->CanDoOperation("operation_name")`.
3. **Проверяйте ДО действия.** Сначала аутентификация и право, затем валидация данных, и только потом изменение. Для изменяющих действий требуйте одновременно: метод **POST** (`Context::getRequest()->isPost()` или `$_SERVER['REQUEST_METHOD'] === 'POST'`), CSRF-токен `check_bitrix_sessid()` и серверную проверку права. ⚠️ `check_bitrix_sessid()` проходит и на GET, поэтому без явной проверки метода защита от CSRF неполна. ⚠️ Токен формы передавайте через `bitrix_sessid_post()` (скрытое поле), а не `bitrix_sessid_get()` — последний кладёт токен в URL и утекает через Referer/логи.
4. **Применяйте наименьшие привилегии.** Выдавайте группе ровно те права, что нужны (например `W` на один инфоблок), а не право администратора. Для повышения привилегий внутри доверенного серверного кода используйте флаг `"NOT_CHECK_PERMISSIONS" => true` точечно.
5. **Задавайте групповые права программно** при создании инфоблока — массив `GROUP_ID => символьный_уровень` в `CIBlock::Add`/`SetGroupPermissions`.

## Рабочий сниппет
Проверки прав в AJAX-обработчике/странице и программная выдача прав на инфоблок при его создании.

```php
<?php
require($_SERVER['DOCUMENT_ROOT'] . '/bitrix/modules/main/include/prolog_before.php');

use Bitrix\Main\Loader;
use Bitrix\Main\Context;

global $USER, $APPLICATION;

// 0. Изменяющее действие: метод POST + CSRF-токен сессии.
//    check_bitrix_sessid() проходит и на GET — проверку метода POST не опускать.
if (!Context::getRequest()->isPost() || !check_bitrix_sessid()) {
    die('Доступ запрещён: требуется POST с корректным sessid');
}

// 1. Аутентификация: без неё административных функций нет.
if (!$USER->IsAuthorized()) {
    die('Доступ запрещён: требуется авторизация');
}

// 2. Право на модуль iblock. Символьные уровни: D=Запрещено, R=Чтение,
//    W=Запись, X=Полный доступ (включая права). Сравнение: D < R < W < X.
if ($APPLICATION->GetGroupRight('iblock') < 'W') {
    die('Доступ запрещён: нет права записи в инфоблоки');
}

Loader::includeModule('iblock');

// 3. Право на КОНКРЕТНЫЙ инфоблок (а не на модуль в целом).
$iblockId = 5;
if (CIBlock::GetPermission($iblockId) < 'W') {
    die('Доступ запрещён: нет права на этот инфоблок');
}

// 4. Право на операцию (задачи/операции групп) и членство в группе.
if (!$USER->CanDoOperation('edit_php')) {
    // нет операции — продолжаем без привилегированной ветки
}
$groups = $USER->GetUserGroupArray();          // [1, 2, ...] ID групп текущего юзера
$isContentManager = in_array(6, $groups, true); // пример: группа «Контент-редакторы»

// --- Программная выдача прав на инфоблок при его создании ---
// Ключ массива GROUP_ID, значение — символьный уровень.
$ib = new CIBlock();
$newIblockId = $ib->Add([
    'NAME'         => 'Статьи',
    'CODE'         => 'articles',
    'IBLOCK_TYPE_ID' => 'content',
    'SITE_ID'      => ['s1'],
    'GROUP_ID'     => [
        '1' => 'X',   // Администраторы — полный доступ
        '6' => 'W',   // Контент-редакторы — запись (наименьшие привилегии)
        '2' => 'R',   // Все пользователи — только чтение
    ],
]);
if (!$newIblockId) {
    echo $ib->LAST_ERROR;
}
```

⚠️ Флаг `NOT_CHECK_PERMISSIONS` отключает проверку прав в API-вызовах (например в `CIBlockElement::GetList(..., ['CHECK_PERMISSIONS' => 'N'])` или эквивалентном параметре методов). Применяйте его ТОЛЬКО в доверенном серверном коде, где право пользователя уже проверено выше, и никогда — на основе данных из запроса.

## Проверка
- `var_dump($USER->IsAdmin())` под админом → `true`, под анонимом/редактором → `false`.
- `var_dump($USER->GetUserGroupArray())` возвращает непустой массив ID групп (для гостя — `[2]`).
- `var_dump($APPLICATION->GetGroupRight('iblock'))` под редактором возвращает уровень, ожидаемый для его группы (`R`/`W`), не `X`.
- `var_dump(CIBlock::GetPermission($iblockId))` отражает реально выданный группе уровень.
- Негативный тест: вызвать обработчик от пользователя без права — должен сработать `die(...)`, изменение НЕ выполниться.
- В админке: «Настройки → Пользователи → Группы пользователей → Доступ» — уровни модулей; на форме инфоблока вкладка «Доступ» — групповые права D/R/W/X.

## ⚠️ Риски
- ⚠️ **Изменяющее действие без серверной проверки права — прямой путь к несанкционированному доступу и потере данных.** Клиентская валидация (скрытые поля, JS) обходится; право проверяйте на сервере до каждого create/update/delete.
- ⚠️ **`NOT_CHECK_PERMISSIONS` / `CHECK_PERMISSIONS => 'N'` на основе пользовательского ввода = обход прав.** Флаг допустим только в доверенном коде, где право уже подтверждено; не передавайте его из запроса.
- ⚠️ **Административная функция без `IsAuthorized()` + проверки прав** открывает её любому посетителю. Аутентификация и проверка права — обязательны вместе.
- ⚠️ **`check_bitrix_sessid()` без проверки метода POST не закрывает CSRF полностью.** Токен проходит и на GET-запросе, поэтому для изменяющих действий требуйте одновременно `isPost()` + `check_bitrix_sessid()` + серверное право. Токен формы — через `bitrix_sessid_post()`; `bitrix_sessid_get()` кладёт его в URL (утечка через Referer/логи) и для разрушающих операций не годится.
- **Право на модуль ≠ право на объект.** `GetGroupRight('iblock')` даёт уровень по модулю; на конкретный инфоблок проверяйте отдельно через `CIBlock::GetPermission` / `CIBlockRights` (расширенные права на элементы/разделы).
- **Символьные уровни сравниваются как строки** в порядке `D < R < W < X` — сравнивайте именно символы (`< 'W'`), не смешивайте с числовыми уровнями групп модуля (`b_module_group_access`).
- **Статические вызовы `CUser::GetID()` и т.п.** в ядре 26.x выбрасывают предупреждение; используйте объект `global $USER` либо `CurrentUser::get()`.
- **Принцип наименьших привилегий.** Группе выдавайте минимально достаточный уровень (например `W` на один инфоблок), а не доступ к админке; администраторов держите в отдельной группе.
- **Группа «Администраторы» (ID 1) и `IsAdmin()`** обходят большинство проверок прав — тестируйте разграничение под непривилегированным пользователем, а не под админом.

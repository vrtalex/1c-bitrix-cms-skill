# Рецепт security/02 — secure-coding: безопасная разработка под БУС

> Подсистема: 1c-bitrix-cms-security. Ядро БУС 26.x. Две поддерживаемые версии
> API (D7 и legacy) — обе официальны.
> Основан на официальных правилах безопасной разработки вендора и изучении ядра «1С-Битрикс: Управление сайтом» (SM_VERSION 26.x).

## Цель

Дать готовые безопасные паттерны для кода, который принимает и выводит
пользовательские данные, чтобы закрыть пять классов уязвимостей:

1. **XSS** — экранирование любого вывода через `htmlspecialcharsbx()`; для
   `href`/`src` — дополнительно валидация схемы URL по белому списку.
2. **CSRF** — подпись и проверка изменяющих действий токеном сессии + проверка
   метода POST.
3. **SQL-инъекции** — ORM либо штатное экранирование вместо конкатенации.
4. **LFI / PHP including** — белый список вместо подстановки ввода в имена файлов.
5. **Права и секреты** — серверная проверка прав, хранение паролей вне VCS.

Итог — словарь «правильный инструмент под каждый контекст вывода/ввода» и
сниппеты, которые можно вставлять напрямую.

## Когда применять

- Выводите в HTML, атрибут, JavaScript или JSON данные, пришедшие от
  пользователя (GET/POST/cookie, поля БД, заполненные пользователем).
- Пишете обработчик действия (создание/удаление/изменение) — форму, AJAX,
  ссылку-действие.
- Формируете SQL вручную или передаёте пользовательские данные в фильтр ORM.
- Подключаете файл по имени из запроса или вызываете shell-команду.
- Открываете чувствительную операцию или храните пароль/секрет в коде.

## Шаги

1. **Определить контекст вывода.** HTML-текст → `htmlspecialcharsbx()`.
   HTML-атрибут → то же, но значение **обязательно в двойных кавычках**.
   JavaScript-строка → `CUtil::JSEscape()`. JS-объект/JSON →
   `\Bitrix\Main\Web\Json::encode()`. Смешанный контекст (`onclick="..."`) —
   экранировать одновременно под HTML и под JS. **URL в `href`/`src`** —
   `htmlspecialcharsbx()` НЕОБХОДИМ, но НЕ достаточен: дополнительно валидировать
   схему по белому списку (см. шаг 1a).
1a. **URL-контекст (`href`/`src`).** `htmlspecialcharsbx()` экранирует кавычки и
   угловые скобки, но НЕ блокирует опасные схемы — `javascript:`, `data:`,
   `vbscript:` в `href`/`src` исполняются как код при клике/загрузке. Перед
   выводом валидируй схему URL по **белому списку** (только `http`, `https`,
   `mailto`; для внутренних ссылок — относительный путь). Отклоняй
   `javascript:`/`data:`/`vbscript:`, а также `protocol-relative` (`//evil.tld`)
   и схему с ведущими пробелами/управляющими символами (`\tjavascript:`).
   Схему извлекай через `\Bitrix\Main\Web\Uri::getScheme()` (нормализует
   значение), пустую схему трактуй как относительный путь только при отсутствии
   `//` в начале. Лишь после проверки схемы оборачивай в `htmlspecialcharsbx()`.
2. **Не доверять пользовательскому HTML.** Если нужно вывести HTML от
   пользователя — пропустить через `CBXSanitizer` (белый список тегов/атрибутов),
   а не через экранирование. ⚠️ Не применять `htmlspecialcharsEx()` — он работает
   по чёрному списку и неполон.
3. **Защитить изменяющие действия от CSRF.** В форме — скрытое поле
   `bitrix_sessid_post()`. Перед выполнением действия проверять
   **метод POST** (`Context::getRequest()->isPost()` или
   `$_SERVER['REQUEST_METHOD'] === 'POST'`) **вместе** с `check_bitrix_sessid()`,
   серверной проверкой прав и валидацией данных. ⚠️ `check_bitrix_sessid()`
   проходит и на GET — без явной проверки метода токен в URL можно протащить
   через GET. ⚠️ `bitrix_sessid_get()` кладёт токен в URL (утечка через Referer и
   логи) — для разрушающих действий не использовать, только `bitrix_sessid_post()`.
4. **Закрыть SQL.** Предпочтительно — ORM (`DataManager`). Если сырой SQL: числа
   приводить к типу (`intval()`), строки экранировать `$DB->ForSql()` (результат
   в кавычках) или `SqlHelper::forSql()`; вставку/обновление — через
   `$DB->PrepareInsert()` / `$DB->PrepareUpdate()`. Конкатенацию ввода в SQL —
   не применять.
5. **Закрыть LFI и выполнение команд.** Имя подключаемого файла — только из
   белого списка; набор символов ограничить латиницей и цифрами; обрезать
   null-byte. Аргументы shell-команд — из разрешённого множества и через
   `escapeshellarg()` / `escapeshellcmd()`.
6. **Проверить права на сервере.** Чувствительные операции — `$USER->IsAdmin()`
   или права модуля/инфоблока на серверной стороне; клиентская валидация не
   считается защитой. Загрузку файлов — валидировать тип/расширение.
7. **Убрать секреты из кода.** Пароли БД и ключи — в `.settings.php` /
   `.settings_extra.php` (последний — вне публичного VCS), не в коммитах.
8. **Не полагаться на неинициализированные суперглобалы/переменные.** Всегда
   инициализируй переменную явным значением перед использованием и читай ввод
   только из конкретного источника (`$_GET`/`$_POST`/`$_REQUEST` с проверкой), а
   не из «возникшей» в области видимости переменной. Это класс register_globals-
   подобной инъекции переменных (CVE-2023-1719, CWE-665 — Improper Initialization).
   Держи модуль `main` пропатченным.
9. **Не хранить секреты интеграций в читаемых настройках.** Пароли и ключи
   внешних сервисов не класть в настройки, отдаваемые по HTTP-запросу
   (cleartext storage, CVE-2024-34891, CWE-312). При раскрытии секрета —
   ротировать его.

## Рабочий сниппет

Файл подключается из `init.php` либо вызывается под админом. Путь:
`/local/php_interface/include/secure_demo.php`. Ядро не трогаем.

```php
<?php
// /local/php_interface/include/secure_demo.php
use Bitrix\Main\Application;
use Bitrix\Main\Web\Json;

if (!defined('B_PROLOG_INCLUDED') || B_PROLOG_INCLUDED !== true) {
    die();
}

// --- 1. XSS: экранирование под каждый контекст ---
$name = $_REQUEST['name'] ?? '';

// HTML-текст и HTML-атрибут (атрибут ОБЯЗАТЕЛЬНО в двойных кавычках):
echo '<div title="' . htmlspecialcharsbx($name) . '">'
    . htmlspecialcharsbx($name) . '</div>';
// htmlspecialcharsbx экранирует двойные кавычки, но НЕ одинарные —
// одинарные кавычки для атрибутов с этой функцией не использовать.

// JavaScript-строка:
echo '<script>var u = "' . CUtil::JSEscape($name) . '";</script>';

// JS-объект / JSON:
$payload = ['name' => $name, 'id' => (int)($_REQUEST['id'] ?? 0)];
echo '<script>var data = ' . Json::encode($payload) . ';</script>';

// Смешанный контекст onclick: экранируем и под JS, и под HTML-атрибут:
$js = "showUser('" . CUtil::JSEscape($name) . "')";
echo '<a href="#" onclick="' . htmlspecialcharsbx($js) . '">open</a>';

// URL в href/src: сперва валидируем СХЕМУ по белому списку, потом экранируем.
// htmlspecialcharsbx НЕ блокирует javascript:/data:/vbscript: — нужна проверка схемы.
$rawUrl = (string)($_REQUEST['url'] ?? '');
$safeUrl = '#';                                    // безопасное значение по умолчанию
$allowedSchemes = ['http', 'https', 'mailto'];
if ($rawUrl !== '' && strncmp($rawUrl, '//', 2) !== 0) {  // отсекаем protocol-relative
    $uri = new \Bitrix\Main\Web\Uri($rawUrl);
    $scheme = strtolower($uri->getScheme());       // нормализованная схема
    if ($scheme === '' && strncmp(ltrim($rawUrl), '/', 1) === 0) {
        $safeUrl = $rawUrl;                         // относительный путь от корня
    } elseif (in_array($scheme, $allowedSchemes, true)) {
        $safeUrl = $rawUrl;                         // разрешённая схема
    }
    // иначе остаётся '#': javascript:/data:/vbscript:/ведущие пробелы отклонены
}
echo '<a href="' . htmlspecialcharsbx($safeUrl) . '">ссылка</a>';

// Пользовательский HTML — только через белый список CBXSanitizer (не статически):
$dirtyHtml = (string)($_REQUEST['comment'] ?? '');
$san = new CBXSanitizer();
$san->SetLevel(CBXSanitizer::SECURE_LEVEL_MIDDLE);
$cleanHtml = $san->SanitizeHtml($dirtyHtml);   // оставит только разрешённые теги
echo $cleanHtml;

// --- 4. SQLi: ORM или штатное экранирование ---
$conn = Application::getConnection();
$sql  = $conn->getSqlHelper();

// Числа — приведение к типу:
$id = (int)($_REQUEST['id'] ?? 0);

// Строки в сыром SQL — forSql, результат В КАВЫЧКАХ:
$login = $sql->forSql((string)($_REQUEST['login'] ?? ''));
$rows = $conn->query(
    "SELECT ID, LOGIN FROM b_user WHERE ID = " . $id
    . " AND LOGIN = '" . $login . "'"
);

// Вставка через PrepareInsert (а не ручная конкатенация значений):
global $DB;
$insert = $DB->PrepareInsert('b_my_log', [
    'NAME' => $name,                       // экранирование берёт на себя ядро
    'CREATED' => date('Y-m-d H:i:s'),
]);
$DB->Query(
    "INSERT INTO b_my_log (" . $insert['FIELDS'] . ")"
    . " VALUES (" . $insert['VALUES'] . ")"
);
```

Изменяющее действие с CSRF-токеном и проверкой прав:

```php
<?php
// форма
echo '<form method="post" action="">';
echo bitrix_sessid_post();                 // скрытое поле с токеном сессии
echo '<input type="hidden" name="id" value="' . (int)$item['ID'] . '">';
echo '<button name="action" value="delete">Удалить</button>';
echo '</form>';

// обработчик: sessid + права + валидация — все три проверки обязательны
global $USER;
if (
    $_SERVER['REQUEST_METHOD'] === 'POST'
    && check_bitrix_sessid()               // токен совпадает с bitrix_sessid()
    && $USER->IsAdmin()                     // серверная проверка прав
) {
    $id = (int)($_REQUEST['id'] ?? 0);      // валидация данных
    if ($id > 0) {
        // безопасно выполняем изменяющее действие
    }
}
```

Подключение файла по белому списку (защита от LFI):

```php
<?php
$allowed = ['report', 'invoice', 'summary'];   // явный белый список
$page = (string)($_REQUEST['page'] ?? '');

if (in_array($page, $allowed, true)) {          // строгое сравнение
    include __DIR__ . '/views/' . $page . '.php';
}
// Пользовательский ввод НЕ подставляем в имя файла напрямую:
// include $_REQUEST['page'] . '.php';  // так делать нельзя
```

## Выбор инструмента

| Контекст вывода/ввода | Инструмент | Версия API |
|---|---|---|
| HTML-текст, HTML-атрибут (в `"`) | `htmlspecialcharsbx()` / `HtmlFilter::encode()` | обе |
| URL в `href`/`src` | белый список схем (`http`/`https`/`mailto`) через `Uri::getScheme()` + `htmlspecialcharsbx()` | D7 |
| Строка в JavaScript | `CUtil::JSEscape()` | обе |
| JS-объект / JSON | `\Bitrix\Main\Web\Json::encode()` | D7 |
| Пользовательский HTML (white-list) | `CBXSanitizer::SanitizeHtml()` | обе |
| Строка в сыром SQL | `$DB->ForSql()` / `SqlHelper::forSql()` | legacy / D7 |
| Вставка/обновление SQL | `$DB->PrepareInsert/PrepareUpdate` | legacy |
| Запрос к данным (предпочтительно) | ORM `DataManager` | D7 |
| CSRF-токен формы/ссылки | `bitrix_sessid_post/get()` + `check_bitrix_sessid()` | обе |
| Аргумент shell-команды | `escapeshellarg()` / `escapeshellcmd()` | PHP |

Правила различения:

- **`htmlspecialcharsbx()` vs `htmlspecialcharsEx()`.** Берём первый: он
  экранирует все спецсимволы. Второй работает по чёрному списку (неполон) — для
  защиты от XSS не подходит.
- **Экранирование vs `CBXSanitizer`.** Если HTML выводить как текст — экранируем.
  Если нужно сохранить разметку пользователя — очищаем санитайзером по белому
  списку; экранирование тут не подходит, оно сломает разметку.
- **Двойные кавычки для атрибутов обязательны.** `htmlspecialcharsbx()` не
  экранирует одинарные кавычки, поэтому атрибут в `'...'` остаётся уязвимым.
- **ORM vs сырой SQL.** Предпочтительно ORM. ⚠️ Но ORM не панацея: не передавать
  непроверенные пользовательские массивы в `select`/`filter`/`order`, не
  использовать `SqlExpression` / `ExpressionField` с пользовательскими данными,
  не пропускать в фильтр ключи на «~» (отключают преобразование значения).

## Проверка

**Режим «только файлы»** (есть код, нет живого ядра):

- Grep по выводу: каждый `echo`/`?>...<?` с пользовательскими данными обёрнут в
  `htmlspecialcharsbx` / `CUtil::JSEscape` / `Json::encode`; HTML-атрибуты — в
  двойных кавычках.
- Каждый `href=`/`src=` с пользовательским URL проходит проверку схемы по белому
  списку (не только `htmlspecialcharsbx`); `javascript:`/`data:`/`vbscript:` и
  `//host` отклоняются.
- Найти `htmlspecialcharsEx(` — кандидаты на замену.
- Каждый изменяющий обработчик содержит `check_bitrix_sessid()` и серверную
  проверку прав; формы — `bitrix_sessid_post()`.
- В сыром SQL нет конкатенации `$_REQUEST`/`$_GET`/`$_POST` без `intval`/
  `forSql`/`PrepareInsert`.
- `include`/`require` не принимают пользовательский ввод в путь без белого
  списка.
- В коммитах нет паролей/ключей (искать в `.settings.php`, `dbconn.php`).

**Режим «живой Битрикс»**:

- Отправить запрос с `<script>alert(1)</script>` в поле — на странице вывод
  экранирован, скрипт не исполняется.
- Повторить изменяющий запрос без корректного `sessid` — действие отклонено.
- Кавычка `'` в строковом параметре поиска не ломает SQL и не даёт ошибки БД.
- Подстановка постороннего имени в параметр `page` подключения файла не
  подключает чужой файл (белый список отсекает).

## ⚠️ Риски

- ⚠️ **XSS при одинарных кавычках в атрибутах.** `htmlspecialcharsbx()` не
  трогает `'`; атрибут в одинарных кавычках остаётся уязвим к внедрению скрипта.
  Атрибуты — только в двойных кавычках.
- ⚠️ **`htmlspecialcharsEx()` не защищает.** Чёрный список пропускает векторы
  атаки. Для экранирования вывода — только `htmlspecialcharsbx()` /
  `HtmlFilter::encode()`.
- ⚠️ **XSS через схему URL в `href`/`src`.** `htmlspecialcharsbx()` НЕ блокирует
  `javascript:`/`data:`/`vbscript:` — пользовательский URL в ссылке/картинке
  исполняет код при клике/загрузке. Валидируй схему по белому списку
  (`http`/`https`/`mailto`) через `Uri::getScheme()`; отклоняй опасные схемы,
  `protocol-relative` (`//host`) и схему с ведущими пробелами/управляющими
  символами. Экранирование делать ПОСЛЕ проверки схемы, не вместо неё.
- ⚠️ **Изменяющее действие без `check_bitrix_sessid()` и проверки метода.** Делает
  возможной CSRF: чужая страница с `<img src="site/action?delete=Y&id=123">` или
  поддельной формой выполнит действие от лица жертвы. `check_bitrix_sessid()`
  проходит и на GET, поэтому требовать одновременно POST-метод
  (`Context::getRequest()->isPost()`), `check_bitrix_sessid()` и серверную
  проверку прав. `bitrix_sessid_get()` для разрушающих действий не применять —
  токен в URL утекает через Referer/логи.
- ⚠️ **Конкатенация ввода в SQL — инъекция.** Прямая склейка `$_REQUEST` в запрос
  ведёт к чтению/изменению чужих данных. Только `intval` для чисел, `forSql` /
  `PrepareInsert` для строк или ORM.
- ⚠️ **Подстановка ввода в `include`/`system()`.** Открывает LFI (включение
  произвольного файла, в т.ч. через null-byte) и выполнение команд. Только белый
  список + ограничение символов; аргументы shell — через `escapeshellarg()`.
- ⚠️ **Секреты в коде/VCS.** Пароль БД или API-ключ в коммите — это утечка
  доступов. Хранить в `.settings_extra.php` вне публичного репозитория.
- ⚠️ **Неинициализированные суперглобалы/переменные.** Расчёт на «возникшую» в
  области видимости переменную открывает инъекцию переменных
  (register_globals-класс: CVE-2023-1719, CWE-665). Инициализируй каждую
  переменную явно и читай ввод только из проверенного источника; держи модуль
  `main` пропатченным.
- ⚠️ **Секреты интеграций в читаемых настройках.** Пароли/ключи внешних сервисов
  в настройках, доступных по HTTP, читаются администратором открытым текстом
  (cleartext storage: CVE-2024-34891, CWE-312). После раскрытия секрет —
  ротировать.
- **Права только на клиенте — не защита.** Скрытая кнопка не мешает отправить
  запрос напрямую. Проверять права на сервере перед каждой чувствительной
  операцией; загрузку файлов — валидировать по типу/расширению.

# Рецепт 06: Обмен с 1С (CommerceML 2.x) — каталог и заказы

Под-скилл: `1c-bitrix-cms-commerce` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x, модули `catalog` 25.x, `sale`, `iblock`.

> **Класс исполнения: E3** — необратимая/привилегированная операция: на проде по умолчанию не исполнять (инструкция человеку или изолированная копия). См. [«0. Право исполнять»](../../conventions.md).

## Цель
Настроить двусторонний обмен «1С ↔ сайт» по протоколу **CommerceML 2.x**: импорт каталога (товары, разделы, свойства, торговые предложения/SKU, цены, склады, остатки) из 1С в инфоблок и выгрузку/приём заказов. Понять протокол точки обмена `1c_exchange.php`, корректно спроектировать структуру под обмен (`XML_ID` как ключ синхронизации) и расширять обмен через события платформы, а не правкой ядра.

## Когда применять
- Запускаете интернет-магазин, где **1С — источник истины** по номенклатуре/ценам/остаткам, а заказы уходят в 1С на обработку.
- Проектируете структуру каталога и нужно заранее знать «контракт данных» (один инфоблок — товары, привязанный — SKU).
- Отлаживаете обмен: импорт «прошёл, но данные не появились», обрывы по таймауту, дубли товаров, права на точку входа.
- Расширяете поведение обмена (доп. обработка после импорта) без потери обновляемости.

Если магазина нет (корп-сайт без каталога из 1С) — обмен не нужен. Для базовой структуры инфоблока см. `../02-create-iblock.md`.

## Шаги

### Каталог (импорт из 1С)
1. **Спроектируйте структуру под CommerceML.** Один инфоблок — «Каталог» (товары), второй (привязанный) — «Торговые предложения / SKU». Если у товаров есть вариации (цвет/размер) — обязательно включайте торговые предложения (`USE_OFFERS=Y`); свойство-связка SKU → товар создаётся автоматически при импорте с офферами.
2. **Заведите пользователя обмена.** Отдельный пользователь сайта в группе «Обмен с 1С» с МИНИМАЛЬНЫМИ правами (только доступ к обмену) — НЕ администратор. Сильный уникальный пароль. В настройках интеграции укажите «Группы, пользователям которых разрешена выгрузка».
3. **Настройте импорт каталога** в админке (Магазин → Настройки → Интеграция с 1С): целевой тип инфоблока, список сайтов, торговые предложения, интервал шага, использование контрольных сумм для оптимизации.
4. **Со стороны 1С** укажите URL обмена `http://сайт/bitrix/admin/1c_exchange.php` (или историческую `1c_import.php`), логин/пароль пользователя обмена. Частоту запуска задаёт регламентное задание в 1С.
5. **Первый импорт.** Для пустого сайта дайте 1С сделать первый импорт — она сама создаст инфоблок, свойства, разделы, товары, цены, остатки через метаданные `Классификатора`.

### Заказы (выгрузка в 1С и приём изменений)
6. **Настройте выгрузку заказов** (вкладки выгрузки в «Интеграция с 1С»): условия выгрузки (только оплаченные / с разрешённой доставкой / со статуса), «Менять статусы заказов по информации из 1С», «Выгружать оплаты/отгрузки», «Создавать новые заказы и контрагентов из 1С».
7. **Сопоставьте статусы** заказа на сайте со статусами, которые ждёт 1С — она шлёт статус обратно. Сопоставьте способы оплаты и доставки с данными учётной системы.
8. **Настройте контрагентов** (вкладка «Экспорт в '1С:Предприятие'»): для физлиц минимум «Фамилия» или «Имя»; для юрлиц «Название», проверка уникальности приоритетно по ИНН.

## Рабочий сниппет
Обмен инициирует **1С**, а не сайт. Поэтому «рабочий код» здесь — не запуск обмена, а корректное проектирование (`XML_ID` совпадает с `Ид` из 1С) и расширение через события платформы. Положите обработчик в `/local/` (не в `bitrix/`), чтобы он пережил обновления.

Файл: `/local/php_interface/include/exchange_events.php` (подключается из `/local/php_interface/init.php`)

```php
<?php
use Bitrix\Main\Loader;

// ---------------------------------------------------------------------------
// 1. РАСШИРЕНИЕ ОБМЕНА БЕЗ ПРАВКИ ЯДРА.
//    После успешного импорта каталога из 1С — своя пост-обработка.
//    Событие модуля catalog: OnSuccessCatalogImport1C.
// ---------------------------------------------------------------------------
\Bitrix\Main\EventManager::getInstance()->addEventHandler(
    'catalog',
    'OnSuccessCatalogImport1C',
    'onCatalogImport1CDone'
);

// Событие catalog:OnSuccessCatalogImport1C передаёт ДВА аргумента:
// $arParams (параметры импорта) и $arErrors (накопленные ошибки/предупреждения).
// Принимаем оба — иначе потеряете диагностику и не отличите успех от частичной ошибки.
function onCatalogImport1CDone($arParams, $arErrors = [])
{
    // $arParams — сведения о завершённом импорте; $arErrors — список ошибок (пуст при успехе).
    // Здесь: пересчёт доп. полей, прогрев кэша разделов, нотификации.
    // НЕ меняйте здесь XML_ID элементов — это ключ синхронизации с 1С.
    \Bitrix\Main\Diag\Debug::writeToFile(
        ['params' => $arParams, 'errors' => $arErrors],
        '1c-import',
        'local/logs/exchange.log'
    );
}

// ---------------------------------------------------------------------------
// 2. ПРОГРАММНОЕ СОЗДАНИЕ ЭЛЕМЕНТА, СОВМЕСТИМОЕ С БУДУЩИМ ОБМЕНОМ.
//    XML_ID элемента ДОЛЖЕН совпадать с <Ид> товара в 1С.
//    Иначе следующий импорт из 1С создаст дубль, а исходный
//    деактивирует как «отсутствующий в выгрузке».
// ---------------------------------------------------------------------------
function addProductCompatibleWith1C(int $iblockId, string $xmlIdFrom1C, string $name): int
{
    if (!Loader::includeModule('iblock')) {
        return 0;
    }
    $el = new CIBlockElement();
    $id = $el->Add([
        'IBLOCK_ID' => $iblockId,
        'NAME'      => $name,
        'XML_ID'    => $xmlIdFrom1C,   // == <Ид> из 1С (первичный ключ синхронизации)
        'ACTIVE'    => 'Y',
    ]);
    if ((int)$id <= 0) {
        // Без обработки ошибки молчаливый дубль при следующем обмене.
        throw new \RuntimeException('Element add error: ' . $el->LAST_ERROR);
    }
    return (int)$id;
}
```

### Протокол обмена (для отладки)
Обмен — серия HTTP-запросов от 1С с параметрами `?type=` и `&mode=`; ответ — plain text, первая строка — статус (`success` / `progress` / `failure`).

Импорт каталога (`type=catalog`):

```
checkauth  -> success\n<имя_сессии>\n<id_сессии>        (Basic-логин/пароль)
init       -> zip=yes\nfile_limit=204800                (лимит части файла, ~200 КБ)
file       -> success                                   (1С POST'ит куски XML/ZIP в /upload/)
import     -> progress\n<%>  ...  success               (пошаговый конечный автомат)
deactivate -> деактивация отсутствующих (только при ПОЛНОЙ выгрузке)
```

Заказы (`type=sale`): `checkauth` → `init` → `query` (сайт отдаёт XML заказов) → `file` (1С шлёт встречный XML с изменениями) → `import` (приём) → `success`. Ветка `type=listen` — long-poll, чтобы 1С узнала о новом заказе почти мгновенно.

## Выбор API
Обмен — это **протокол точки входа `1c_exchange.php`**, который вызывает штатные компоненты и классы; писать его «руками» не нужно — он включается настройками. Если задача в расширении или интеграции, выбор такой:

- **Каталог — legacy-классы `C…` (`CIBlockCMLImport` / `CIBlockCMLExport`).** Это ядро обмена каталогом и рекомендованный путь; D7-обёртки для импорта каталога нет, и это нормально. Оркестрацию шагов ведёт компонент `bitrix:catalog.import.1c` поверх этих классов.
- **Заказы — гибрид.** Экспорт заказов — `CSaleExport::ExportOrders2Xml()`. Приём входящих заказов зависит от версии схемы: для схем `>= 2.10` задействуется D7-namespace `\Bitrix\Sale\Exchange\*` (`ManagerImport`, `OrderImport`, `ImportOneCSubordinateSale`), для более ранних — связка `CSaleOrderLoader`. Это две поддерживаемые версии API; для заказов ветка определяется автоматически по `ВерсияСхемы` в XML.
- **Расширение поведения — только события платформы** (`OnSuccessCatalogImport1C`, события модулей `catalog`/`sale`) и наследование компонентов в своё пространство имён. Правка файлов ядра затирается обновлением.

Идентификация во всех ветках — по `Ид` (`XML_ID`), а не по имени/коду: `GetElementByXML_ID`, `GetSectionByXML_ID`, `GetPropertyByXML_ID`. Это краеугольный камень — повторный импорт по `XML_ID` обновляет, а не дублирует.

## Проверка

**Режим «только файлы» (без живого Битрикса):**
- Обработчики обмена лежат в `/local/php_interface/`, а не в `bitrix/` (переживут обновление).
- В коде программного создания элементов **задан `XML_ID`** (совпадающий с `Ид` из 1С), есть `Loader::includeModule('iblock')` и обработка ошибок (`LAST_ERROR`).
- Расширение сделано через `addEventHandler` на событие платформы, а не правкой файлов в `bitrix/modules/`.
- Кодировка обмена — `windows-1251` по умолчанию; для ручного чтения примеров XML используйте `iconv -f WINDOWS-1251 -t UTF-8`.

**Режим «живой Битрикс»:**
- Запустите обмен со стороны 1С — финальный ответ `success` (импорт) / `success Импорт успешно завершен`.
- Журнал обмена: админка `bitrix/admin/exchange_log.php`; отчёты импорта — `/upload/1c_catalog/Reports`. Файлы каталога приходят в `/upload/1c_catalog/`, заказов — в `/upload/1c_exchange/`.
- Каталог: товары/разделы/цены/остатки появились в инфоблоке; у элементов проставлен `XML_ID`; торговые предложения привязаны к товарам.
- Заказы: тестовый заказ выгрузился в 1С; статус, изменённый в 1С, вернулся на сайт.
- Для отладки можно включить сохранение XML (`BX_CATALOG_IMPORT_1C_PRESERVE`), чтобы файлы не удалялись перед следующим обменом.

## ⚠️ Риски
- ⚠️ **`XML_ID` — первичный ключ синхронизации.** Потеря или смена `XML_ID` (на сайте или в 1С) ведёт к дублям и пересозданию элементов: старый деактивируется как «отсутствующий», новый создаётся заново. При программном создании всегда проставляйте `XML_ID`, равный `Ид` из 1С.
- ⚠️ **Режим «выгружать только изменённые» НЕ деактивирует** отсутствующие товары — этап `deactivate` выполняется только при полной выгрузке. Если ожидаете автоскрытие снятых с продажи позиций, нужна периодическая полная выгрузка.
- ⚠️ **Обнуление остатков при импорте офлайн-заказов** (включён количественный учёт без резервирования): поставьте «Количество контейнеров в пакете» = 1, чтобы онлайн- и офлайн-заказы шли в разных файлах обмена.
- ⚠️ **Несовпадение `ВерсияСхемы`** конфигурации 1С и ожиданий сайта — частая причина «обмен прошёл, но заказы не импортировались»: для заказов ветка `>= 2.10` идёт через D7, иначе — через `CSaleOrderLoader`.
- **502 / белый экран** при импорте — нехватка памяти/времени: увеличьте `memory_limit` (минимум 512 МБ) и `max_execution_time`, уменьшите «Интервал одного шага» до 1–5 сек.
- **Состояние обмена хранится в сессии** (`$_SESSION`). Балансировщик без общей сессии между нодами или несохранение cookie между шагами прерывают обмен.
- **Права и точка входа.** Обмен идёт под учёткой из группы обмена через `1c_exchange.php`; неверные права → `failure`. Проверьте отсутствие лишнего редиректа на HTTPS и доступность точки.
- **Ограничения редакции/тарифа по ценам.** В младшей редакции разрешён один тип цены и один склад; выгрузка нескольких типов цен прервёт обмен. Дубли типов цен с одинаковым названием также прерывают обмен.
- **Грабли контрольных сумм.** При кастомных изменениях в 1С товар может не попасть в выгрузку изменений, если контрольная сумма не пересчитана — пересчитайте её в 1С после правок.
- **Производительность по типам цен.** Больше ~50 типов цен через выгрузку «по типам цен» резко замедляет импорт (декартово произведение цен и количеств).

### ⚠️ Безопасность точки обмена `1c_exchange.php`
- ⚠️ **Только HTTPS.** Авторизация обмена — Basic (логин/пароль в заголовке). По HTTP креды перехватываются в открытом виде. Точка `1c_exchange.php` должна быть доступна исключительно по HTTPS.
- ⚠️ **Минимальные права пользователя обмена.** Сильный уникальный пароль, группа «Обмен с 1С», НЕ администратор. Утечка этих кредов = запись в каталог/заказы.
- ⚠️ **Ограничьте доступ по IP**, если адрес 1С статичный — пустите к `1c_exchange.php` только known-IP (на уровне веб-сервера/файрвола).
- ⚠️ **`/upload` — запрет выполнения PHP/.htaccess.** 1С POST'ит части XML/ZIP в `/upload/1c_catalog/` (и `1c_exchange/`). Каталог приёма не должен исполнять PHP и не должен допускать загрузку/переопределение через `.htaccess` (см. `../../conventions.md` — защита `/upload`).
- ⚠️ **Парсинг и авторизация — ядро (`CIBlockCMLImport` и компонент обмена).** Своими руками их не правьте; держите ядро БУС обновлённым — уязвимости разбора CommerceML закрываются обновлениями платформы.

# 04. Приёмка по чек-листу: готовность к сдаче

## Цель

Свести воедино обязательные пункты приёмки проекта на «1С-Битрикс: Управление сайтом» (ядро 26.x) в один чек-бокс-лист и подтвердить готовность к сдаче. Чек-лист объединяет результаты Монитора качества, статической проверки конвенций (`check-conventions`), требования безопасности, производительности и регламент выкладки. Назначение — дать однозначный ответ «можно сдавать / нельзя сдавать» перед передачей заказчику.

## Когда применять

- Перед сдачей проекта заказчику или перед релизом на прод.
- При финальной самопроверке интегратором/разработчиком.
- В точке приёмки QA, когда нужно зафиксировать, что обязательные критерии закрыты.
- Перед запуском нагрузочного тестирования и публичным запуском.

Применять после того, как прогнаны автотесты Монитора качества, исправлены замечания и пройдены ручные тесты (рекомендованный порядок Монитора: автотесты → отчёты и вкладка «Рекомендации» → исправления → перезапуск автотестов → ручные тесты).

## Шаги

1. **Запустить Монитор качества.** Настройки → Инструменты → Монитор качества. Добиться, чтобы все 26 обязательных тестов были в статусе «Пройден». Если обязательный тест переведён в «Пропущен» — у него обязан быть комментарий разработчика; без комментария пропуск обязательного теста недопустим.
2. **Прогнать статическую проверку конвенций.** Запустить `check-conventions` (см. `../../operations.md`). Не должно остаться ни одного `[FAIL]`; все критичные `[REVIEW]` должны быть разобраны и сняты (исправлены либо обоснованы комментарием).
3. **Закрыть блок безопасности.** Проактивная защита не ниже «Стандартный», экранирование пользовательского ввода, проверка `sessid` в формах, статический анализ уязвимостей кода пройден.
4. **Закрыть блок производительности.** Автокеширование включено, композитный режим работает где применимо, нет долгих страниц, оценка конфигурации не ниже «Удовлетворительная».
5. **Удалить тестовые и демо-артефакты.** Тестовые учётки, тестовые страницы и домены, дефолтные демо-шаблоны, демо-ключ — удалены/заменены боевыми.
6. **Подтвердить эксплуатационную готовность.** Бэкап снят, доступы переданы, продукт обновлён до актуальной версии, обновления безопасности применены.
7. **Заполнить сводный чек-лист ниже.** Каждый обязательный пункт должен быть отмечен. ⚠️ Незакрытый обязательный пункт = проект не готов к сдаче.

## Рабочий чек-лист приёмки

### A. Монитор качества (обязательно)

- [ ] Все 26 обязательных тестов Монитора качества — статус «Пройден».
- [ ] Каждый «Пропущен» среди обязательных тестов имеет комментарий разработчика с обоснованием.
- [ ] Автотесты (14 проверок) перезапущены после исправлений, замечания во вкладке «Рекомендации» разобраны.
- [ ] Заполнена информация об интеграторе и о техподдержке (группа «Сдача проекта»).
- [ ] Лицензионный ключ активирован (боевой, не демо).

### B. Конвенции и статпроверка (обязательно)

- [ ] `check-conventions` выполнен; в выводе нет ни одного `[FAIL]`.
- [ ] Все критичные `[REVIEW]` сняты: исправлены либо снабжены обоснованием.
- [ ] Кастомный код размещён в `/local`, ядро `/bitrix/modules/` не модифицировано (целостность ядра).

### C. Безопасность (обязательно)

- [ ] Модуль «Проактивная защита» включён минимум на уровне «Стандартный».
- [ ] Административные учётки защищены: повышенный уровень группы, двухфакторная авторизация (OTP) для администраторов.
- [ ] Права групп выданы по принципу наименьших привилегий.
- [ ] Пользовательский ввод экранируется до вывода и до SQL (`htmlspecialchars`, фильтрация, кодирование); данные из HTTP-запроса считаются небезопасными по умолчанию.
- [ ] Формы и операции изменения данных защищены проверкой `sessid` (`check_bitrix_sessid()`).
- [ ] Пройден тест статического анализа уязвимостей кода (XSS, SQL-инъекции, выполнение PHP-кода, инъекции команд, HTTP Response Splitting, File Inclusion).
- [ ] Вывод ошибок посетителям отключён (`$DBDebug = false`, режим «Не выводить» в Главном модуле).
- [ ] Подключение к БД настроено не под root, пароль БД стойкий (8+ символов, разные регистры, цифры, спецсимволы).
- [ ] Включён журнал системных событий и Web-антивирус.

### D. Производительность (обязательно)

- [ ] Автокеширование компонентов включено («Авто+Управляемое» или «Авто»).
- [ ] Композитный режим включён для применимых страниц.
- [ ] PHP сконфигурирован оптимально (Панель производительности → «Конфигурация»).
- [ ] Платформа в состоянии «Битрикс (оптимально)» на вкладке «Битрикс».
- [ ] Оценка конфигурации не ниже «Удовлетворительная»; под кешем нет долгих страниц.
- [ ] `/bitrix/php_interface/init.php` не содержит запросов к БД и других ресурсоёмких операций.

### E. Очистка тестовых и демо-данных (обязательно)

- [ ] Удалены тестовые учётные записи (`test`/`123456` и подобные).
- [ ] Удалены тестовые страницы, тестовые домены, демо-контент.
- [ ] Удалены/заменены дефолтные демо-шаблоны там, где используется боевой шаблон.
- [ ] Демо-ключ не остался на проде; на dev/stage — некоммерческий ключ разработки, на prod — боевой ключ.
- [ ] Удалены служебные скрипты переноса (`restore.php` и т. п.) с сервера.

### F. Эксплуатация: бэкап, доступы, обновления (обязательно)

- [ ] Снята актуальная резервная копия (файлы + дамп БД).
- [ ] Доступы (админка, БД, хостинг, домен/DNS) собраны и переданы заказчику безопасным каналом.
- [ ] Продукт обновлён до актуальной версии перед сдачей.
- [ ] Применены обновления безопасности.

### G. Функциональные ручные проверки (обязательно)

- [ ] Протестирован поиск, проверены битые ссылки.
- [ ] Проверено поведение форм при повторной отправке (F5 / повторный сабмит).
- [ ] Права доступа проверены в трёх состояниях: неавторизованный, авторизованный, администратор.
- [ ] Сайт повторно протестирован под включённым кешем.

### H. Соответствие РФ-законодательству (152-ФЗ, обязательно для сайтов, собирающих ПДн)

- [ ] Уведомление в Роскомнадзор подано **до начала обработки ПДн** (требование действует с 01.09.2022; подаётся на Портале персональных данных РКН).
- [ ] Согласие на обработку ПДн оформлено **отдельным документом**, а не зашито в оферту/ToS/договор (требование действует с 01.09.2025).
- [ ] Галочка согласия **не преднажата**: согласие требует активного утвердительного действия, молчание/бездействие согласием не считается.
- [ ] Каждая форма, собирающая ПДн, привязана к соглашению через `bitrix:main.userconsent.request`; полученные согласия пишутся в журнал «Полученные согласия».
- [ ] На сайте присутствует cookie-баннер с активным действием-принятием; используемые cookie описаны в политике конфиденциальности.
- [ ] Локализация ПДн: сервер/БД сайта и внешние сервисы (CRM, аналитика), получающие данные форм, размещены на инфраструктуре в РФ (152-ФЗ ст. 18 ч. 5 / 242-ФЗ).

### I. Рекомендуемое (необязательно, для сложных/нагруженных проектов)

- [ ] Карта сайта настроена (`bitrix:main.map`).
- [ ] Документация заполнена: настройки проекта, параметры компонентов, обработчики событий, агенты/SEO-URL.
- [ ] БД-изменения переносятся миграциями (sprint.migration / intervolga.migrato), а не правками вручную.
- [ ] Удалены неиспользуемые файлы, проверены сторонние подключения.

## Проверка

- В Мониторе качества: все 26 обязательных тестов зелёные, у пропущенных есть комментарии — формально проект готов к сдаче по чек-листу.
- `check-conventions` завершается без `[FAIL]`; список `[REVIEW]` либо пуст, либо каждый пункт помечен как разобранный.
- Контрольный прогон: открыть несколько ключевых страниц под включённым кешем и в роли неавторизованного посетителя — ошибки не выводятся, страницы отдаются быстро.
- Все чек-боксы блоков A–H отмечены (блок H — для сайтов, собирающих ПДн). Если хотя бы один обязательный пункт не отмечен — приёмка не пройдена.

## ⚠️ Риски

- ⚠️ **Нельзя сдавать проект с проваленными обязательными пунктами.** Хотя бы один незакрытый обязательный тест Монитора качества или неснятый критичный `[REVIEW]` означает, что проект не готов к сдаче. Это провал приёмки.
- ⚠️ **Демо-ключ на проде** приводит к отказу сайта по истечении срока. Перед сдачей убедиться, что ключ боевой.
- ⚠️ **Отсутствие бэкапа перед релизом** делает невозможным безопасный откат — критично перед нагрузочным запуском.
- Приёмка только по автотестам недопустима: автотесты собирают данные, но конечное решение и ручные тесты остаются за разработчиком.
- Точный состав и коды тестов Монитора качества (QD/QSEC/QP/QS/QC/QE) сверяйте непосредственно в инструменте на ядре 26.x — набор тестов меняется между версиями.

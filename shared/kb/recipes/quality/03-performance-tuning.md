# Рецепт 03 (quality): Тюнинг производительности под нагрузку

Под-скилл: `1c-bitrix-cms-quality` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x, модуль `perfmon` (Монитор/Панель производительности), для кластера — модуль «Веб-кластер».

## Цель
Довести проект до приёмочного уровня производительности под целевой нагрузкой и зафиксировать это измеримо. Рычаги: технология «Композитный сайт», стратегии кэша (автокэш компонентов, управляемый, тегированный), фасетный индекс каталога/умного фильтра, отложенные функции, БД-тюнинг, веб-кластер, CDN. Контроль — Монитор и Панель производительности с явными нормами и нагрузочным тестом.

## Когда применять
- Финальная приёмка сайта/магазина: нужно подтвердить, что под характерной нагрузкой время отклика держится в норме.
- Проект тормозит под нагрузкой, и нужно локализовать узкое место (backend-код, БД, кэш, окружение).
- «Большие объёмы данных»: каталог растёт, умный фильтр и списки деградируют, всплывают N+1 и тяжёлые выборки.
- Подготовка к росту трафика — оценка перехода на веб-кластер и распределённый кэш.

Если задача — спроектировать кэширование конкретного компонента с нуля, начните с правил кэша в `../../conventions.md` и подключите этот рецепт для замеров и приёмки.

## Шаги
1. **Снимите базовую метрику.** Настройки → Производительность → Панель производительности → «Тестировать производительность». Замер делайте **под характерной для сайта нагрузкой**; на малопосещаемом сайте сами открывайте разные страницы в течение всего измерения, иначе данные нерепрезентативны.
2. **Проверьте окружение** (даёт максимальный эффект за минимум усилий): OPcache включён; на собственном сервере `open_basedir` выключен; вся статика (css/js/img) отдаётся через nginx; драйверы оборудования оригинальные. Без OPcache страницы открываются заметно медленнее.
3. **БД → InnoDB и my.cnf.** В `dbconn.php`: `define("MYSQL_TABLE_TYPE", "InnoDB");`. Настройте `innodb_buffer_pool_size` (≈60–80% свободной памяти), включите `slow_query_log`, прогоните `mysqltuner.pl`. На «1С-Битрикс: Виртуальная машина» правки кладите в `/etc/mysql/conf.d/z_bx_custom.cnf`.
4. **Включите Композит.** Перед активацией очистите кэш компонентов. Для типового сайта — режим «Автокомпозит»; динамическими оставьте поиск, корзину, оформление заказа, личные блоки. nginx должен отдавать композитный кэш напрямую.
5. **Настройте кэш компонентов.** Для штатных компонентов — режим «Авто + Управляемое» (кэш сбрасывается при Add/Update/Delete элементов и разделов). Статичные данные — большой интервал (вплоть до недели/года); часто меняющиеся — управляемый кэш или короткий интервал.
6. **Включите управляемый и тегированный кэш каталога.** В `dbconn.php`: `define("BX_COMP_MANAGED_CACHE", true);`. Сброс по тегу — `CIBlock::clearIblockTagCache($iblockId)`. Для часто обновляемых больших массивов тегированный кэш не применяйте (см. Риски).
7. **Каталог: фасетный индекс.** Для умного фильтра используйте фасетный индекс (Инфоблоки 2.0). Пересоздавайте его вручную при добавлении/удалении свойства фильтра и перемещении разделов; полную переиндексацию — в период минимальной нагрузки.
8. **Вынесите кэш и сессии.** Для нагруженного проекта — кэш в Redis/Memcache, сессии в Memcache/Redis (а не в ядре БД); параметр `sid` кэша формируйте как `$_SERVER['DOCUMENT_ROOT'].'#01'`.
9. **CDN/масштабирование.** Статику — на сторонний CDN (встроенная функция CDN отключена начиная с версии модуля bitrixcloud 22.100.0). При росте нагрузки — модуль «Веб-кластер» (веб-ноды + балансировщик, master/slave БД, пул memcached).
10. **Повторите замер под нагрузкой** и проведите нагрузочное тестирование с паузами между обращениями. Сравните с нормами (см. чек-лист) и зафиксируйте результат в отчёте о приёмке.

## Чек-лист приёмки производительности
Нормы и контрольные точки для приёмки. Абсолютные баллы Монитора сами по себе ничего не значат — они нужны для поиска узких мест; сравнение идёт с эталоном «1С-Битрикс: Виртуальная машина» (VMBitrix).

| Что проверяем | Норма / приёмочный критерий |
|---|---|
| Среднее время отклика пустой страницы (Панель) | ниже **0,5 с** |
| Показатель производительности (Монитор) | хороший — от **100**; идеальный — **200+**; под пиковой нагрузкой не ниже **~10** (≈0,1 с/страницу) |
| Оценки подсистем (Конфигурация) | не ниже эталонной; если сервер ниже эталона — менять хостинг, если в норме, а сайт тормозит — backend-код проекта |
| Размер файла кэша | держать **≤ ~1 МБ**; больший оптимизировать |
| `init.php` | **без запросов к БД** (выполняется на каждый хит) |
| Замер | под характерной нагрузкой; Монитор включать ограниченно (≤1 ч; до недели в режиме «только медленные SQL») |
| Нагрузочный тест | с паузами между обращениями (5–20 с), показатель не падает ниже приемлемого |

Чек-лист рычагов (отметить применённое):
- [ ] OPcache включён; `open_basedir` выключен (свой сервер); статика — через nginx.
- [ ] Все таблицы → InnoDB; my.cnf настроен; `slow_query_log` включён; прогнан `mysqltuner.pl`.
- [ ] Композит включён (Автокомпозит); динамика — в `bxdynamic`-зонах; nginx отдаёт композит-кэш.
- [ ] Штатные компоненты на «Авто + Управляемое»; интервалы кэша подобраны по частоте обновления.
- [ ] `BX_COMP_MANAGED_CACHE` включён; тегированный кэш каталога; НЕ применять на часто меняющихся больших массивах.
- [ ] Кэш в Redis/Memcache, сессии вне ядра БД; `sid` кэша корректный.
- [ ] Каталог: фасетный индекс (Инфоблоки 2.0), индексы на поля фильтра, контроль порога объёма.
- [ ] Большие выборки: явный SELECT нужных полей, нет N+1, отказ от постраничной навигации на огромных списках.
- [ ] CDN (сторонний) для статики; веб-кластер при росте нагрузки.

## Большие объёмы данных и N+1
- **Выбирайте только нужные поля:** `GetList()` с явным `select` вместо `GetByID()` для каждого элемента; не тащите все свойства без ограничений.
- **Избегайте N+1:** один запрос со списком вместо запроса на элемент в цикле; в ORM подтягивайте связи через `runtime`/`with`, а не дозапросами в цикле.
- **Денормализация при больших объёмах** (партнёр): счётчики категорий и редко изменяемые агрегаты — в отдельное поле, пересчёт агентами (cron); свойства, по которым не фильтруют, хранить сериализованно; на 1 млн+ — вынос поиска/фильтра в внешний движок и кастомный кэш детальных страниц с выборочным сбросом.
- **Пороги каталога** (партнёр, не вендор): типы цен — 10–20 (каждый добавляет ~5–10% к времени генерации страниц фильтрации); Инфоблоки 2.0 — при 50 000+ товаров (лимит ~50 свойств на таблицу); деградация фасетного индекса начинается при ~10 млн записей в таблице фасетов.

## Отложенные функции (deferred)
Позволяют из тела страницы управлять элементами пролога (title, meta, навцепочка, CSS, кнопки панели), сохраняя кэшируемость шапки и меню: вывод откладывается до эпилога (`ShowTitle`, `ShowMeta`, `ShowNavChain`, `ShowCSS`). Исключение, работающее при кэшировании, — `SetViewTarget`/`EndViewTarget`/`ShowViewContent`. Ограничение: внутри `template.php`/`result_modifier.php` их применять нельзя (результат кэшируется), а `GetTitle()`/`GetProperty()` в теле страницы вернут значения ДО выполнения отложенных функций — условную логику на их результат не строить.

## Проверка
**Режим «только файлы» (без живого Битрикса):**
- В `dbconn.php` есть `MYSQL_TABLE_TYPE = InnoDB` и `BX_COMP_MANAGED_CACHE = true`.
- В `init.php` нет запросов к БД.
- Компоненты в шаблоне используют режим управляемого кэша; тяжёлых выборок и запросов к БД в `template.php`/`result_modifier.php` меню нет.
- В выборках больших списков — явный `select` нужных полей; нет запроса на элемент внутри цикла (N+1).

**Режим «живой Битрикс»:**
- Панель производительности: среднее время отклика ниже 0,5 с; оценки подсистем не ниже эталонных.
- Монитор, вкладка «Разработка»: нет некэшированных компонентов среди нагруженных страниц; нет файлов кэша заметно больше ~1 МБ.
- Композит работает: проверка в режиме инкогнито с несколькими переходами; динамические зоны подгружаются отдельным AJAX-запросом.
- Каталог: умный фильтр на фасетном индексе отвечает стабильно быстро; повторные запросы списка — из кэша.
- Нагрузочный тест с паузами: показатель производительности не опускается ниже приемлемого уровня.

## ⚠️ Риски
- ⚠️ **Провал приёмки по нагрузке из-за некэшированных компонентов.** Типовая причина медленной генерации — компонент без кэша на нагруженной странице. Ищите на вкладке «Разработка» Монитора и включайте кэш до сдачи.
- ⚠️ **Тегированный кэш на часто обновляемом большом массиве — перегрузка БД.** Теги хранятся в таблице `b_cache_tag`; для часто меняющихся больших данных накладные расходы не оправданы и создают рост нагрузки на БД. При массовом импорте отключайте тегированный кэш (`CIBlock::DisableTagCache`), грузите данные, затем включайте и чистите одним вызовом.
- ⚠️ **Запросы к БД в `init.php` — деградация на каждом хите.** `init.php` выполняется на каждый запрос; запросы к БД здесь умножают нагрузку линейно по трафику.
- ⚠️ **Частый массовый сброс большого кэша — пики нагрузки на проде.** Раскладывайте пользовательский кэш по подпапкам и избегайте короткоживущих ключей (например, `date()` без округления), иначе кэш перезаписывается слишком часто.
- ⚠️ **Деградация фасета на сверхбольшом каталоге.** При приближении к ~10 млн записей фасета тяжёлые JOIN замедляют фильтр; заранее минимизируйте свойства SKU, типы цен и склады, добавляйте индексы на поля фильтра.
- **Замер без нагрузки нерепрезентативен.** Без посетителей открывайте страницы сами в течение всего измерения; абсолютные баллы сравнивайте только с эталоном, не разные редакции между собой.
- **Запросы к БД и тяжёлые вычисления в `template.php`/`result_modifier.php` меню** плодят файлы кэша и переполняют акселератор; выносите их в `menu_ext.php`.
- **Встроенный CDN отключён** (с версии bitrixcloud 22.100.0) — закладывайте сторонний CDN, не рассчитывайте на штатную функцию.
- **Веб-кластер: запрет `localhost`.** В `/bitrix/.settings.php` на всех веб-серверах адрес главной БД указывать прямым IP; `localhost` использовать нельзя.

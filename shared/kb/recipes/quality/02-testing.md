# Рецепт Quality-02: Тестирование проекта на «1С-Битрикс»

Под-скилл: `1c-bitrix-cms-quality` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x.

## Цель
Выстроить тестирование проекта на Битрикс из двух слоёв: приёмочного (вендорский «Монитор качества» + чек-лист сдачи) и автоматического (PHPUnit/Codeception поверх инициализированного ядра, прогон в CI). Итог — повторяемая процедура, после которой проект можно сдавать клиенту с зафиксированным отчётом, а кастомный код покрыт тестами, которые гоняются на каждый коммит.

## Когда применять
- Перед сдачей проекта или релизом — нужен прозрачный приёмочный отчёт для клиента (Монитор качества).
- Пишете технически сложный кастомный функционал в `/local` (ООП-классы, низкоуровневые утилиты, расчёты) — нужны unit-тесты.
- Настраиваете CI и хотите ловить интеграционные проблемы автоматически на каждый push.
- Принимаете чужой проект и оцениваете, чем его регрессия покрыта.

Граница ответственности вендора: у платформы **нет встроенного фреймворка unit-тестов** и нет официального bootstrap для PHPUnit в дистрибутиве. Вендор описывает виды тестов и CI как рекомендации (курс «Разработка и эксплуатация высоконагруженных проектов»), а для приёмки даёт собственный инструмент — Монитор качества. Интеграция PHPUnit/Codeception — практика сообщества и партнёров.

## Шаги
1. **Определите слой тестирования под задачу.** Типовой проект (инфоблоки + штатные компоненты) держится на приёмочном слое — Монитор качества и чек-лист сдачи. Сложный кастом в `/local` требует ещё и автотестов; их объём может приблизиться к объёму проекта, поэтому покрывайте прицельно: API, расчётные модули, утилиты.
2. **Пройдите приёмочный чек-лист сдачи** (см. блок ниже) — поиск, защита от повторной отправки по F5, роли доступа, карта сайта, кеш включаемых областей, немодифицированное ядро, удаление неиспользуемых файлов.
3. **Пройдите «Монитор качества»** — основной официальный приёмочный инструмент. 26 обязательных тестов + 40 необязательных, из них 14 автоматических. Сгруппированы по этапам внедрения: интеграция дизайна → данные → компоненты → производительность → хостинг → безопасность → сдача. Порядок прохождения свободный.
4. **Закройте обязательные тесты или поставьте комментарий разработчика.** Обязательный тест пропускается только с комментарием (например, функционал отсутствует в ТЗ). Автоматический тест собирает данные, но решение всё равно за разработчиком. Статус «Пройдено успешно» ставит тестировщик/менеджер, не программист (партнёр Sibirix).
5. **Поднимите PHPUnit для кастома** (если есть сложный код): bootstrap инициализирует ядро, `phpunit.xml` указывает на него, классы лежат в `/local/classes`, тесты — в `/local/tests` (сниппеты ниже).
6. **Для сценарных и браузерных проверок добавьте Codeception** (партнёр/сообщество): functional-тесты гоняют код без веб-сервера (эмуляция GET/POST), acceptance — реальный браузер через WebDriver/Selenium. Вендор относит автотесты компонентов к функциональному тестированию (Selenium-бот эмулирует действия пользователя).
7. **Встройте прогон в CI.** Вендор формулирует три требования к CI: система контроля версий с частыми коммитами, модульное тестирование (PHPUnit/SimpleTest, опционально), синхронизация версий Bitrix Framework между тестовым и боевым серверами. Конкретный инструмент вендор не навязывает; на практике — Jenkins (партнёр Intervolga) или GitLab CI с docker-образом Codeception.

### Чек-лист приёмки (сдача проекта, вендор)
- [ ] **Поиск** — ссылки в результатах поиска по разным инфоблокам валидны.
- [ ] **F5 / повторная отправка** — обновление страницы не отправляет данные повторно.
- [ ] **Роли доступа** — корректная работа в трёх ролях: гость, авторизованный, администратор.
- [ ] **Карта сайта** — соответствует структуре контента (для каталога — каталожная версия).
- [ ] **Дефолтные шаблоны** — стандартные макеты исключены из публичной части.
- [ ] **Оформление контента** — стили применяются в визуальном редакторе.
- [ ] **Кеш включаемых областей** — динамическая информация не кешируется.
- [ ] **Резервное копирование и обновления** — настроены.
- [ ] **Автокеширование компонентов** — включено.
- [ ] **Ядро системы** — не модифицировано, соответствует оригиналу.
- [ ] **Неиспользуемые файлы** — удалены; тестовые данные удалены после выкладки.
- [ ] **Сторонние подключения** — проведён аудит безопасности.
- [ ] ⚠️ **Нагрузочное тестирование** — для крупных проектов обязательный этап сдачи: определяет эксплуатационные пределы на выбранном оборудовании (этап «производительность» Монитора).

## Рабочий сниппет

**`/local/tests/bootstrap.php`** — инициализация ядра для тестов (отключаем проверку прав и агенты, ошибки в stderr):

```php
<?php
// Bootstrap прогоняется ТОЛЬКО из CLI (phpunit). Каталог /local/tests НЕ должен
// отдаваться веб-сервером: bootstrap определяет NOT_CHECK_PERMISSIONS=true и при
// HTTP-доступе превратился бы в публичный эндпоинт с отключённой проверкой прав.
if (PHP_SAPI !== 'cli') { http_response_code(404); exit; } // CLI-only: иначе публичный привилегированный эндпоинт
$_SERVER["DOCUMENT_ROOT"] = dirname(dirname(__DIR__));
define("LANGUAGE_ID", "pa");
define("NO_KEEP_STATISTIC", true);     // не подключать модуль статистики
define("NOT_CHECK_PERMISSIONS", true); // отключить проверку прав на время тестов
define("NO_AGENT_CHECK", true);        // не запускать агенты
define("LOG_FILENAME", 'php://stderr'); // ошибки в stderr вместо файловых логов
require($_SERVER["DOCUMENT_ROOT"]."/bitrix/modules/main/include/prolog_before.php");
```

⚠️ Каталог `/local/tests` (вместе с `bootstrap.php`) должен быть закрыт от
веб-сервера — он рассчитан только на запуск из CLI через PHPUnit. Закройте
HTTP-доступ deny-правилом, например для Apache в `/local/tests/.htaccess`:

```apache
Require all denied
```

Для Nginx — `location ^~ /local/tests/ { deny all; return 404; }`. Без этого
оставленный в webroot bootstrap c `NOT_CHECK_PERMISSIONS=true` становится
публичным эндпоинтом с отключённой проверкой прав.

**Тест с подключением модуля.** ⚠️ Обязательно исключите `DB` из бэкапа глобальных переменных: PHPUnit по умолчанию делает backup/restore глобалов между тестами и тем самым рвёт глобальный коннект Битрикса `$GLOBALS["DB"]` — без этого падает первый же SQL-запрос:

```php
<?php
final class SimpleTest extends \PHPUnit\Framework\TestCase
{
    protected $backupGlobalsBlacklist = ['DB']; // не трогать коннект Битрикса

    public function testModuleLoad(): void
    {
        $this->assertTrue(\CModule::IncludeModule('iblock'));
    }

    /** @dataProvider leftoversProvider */
    public function testLeftovers(string $in, int $expectedTime, float $expectedQty): void
    {
        // ... проверка расчётной логики кастомного класса
    }
}
// Отладка SQL внутри теста: $GLOBALS["DB"]->debug = true;
```

**Структура и автозагрузка** (кастом в `/local`, ядро не трогаем):

```
/local/classes/MyProject/Product.php
/local/classes/MyProject/Requisites/Leftover.php
/local/tests/LeftoverTest.php
/local/phpunit.xml      bootstrap="tests/bootstrap.php", testsuite=tests, suffix Test.php
/local/composer.json    psr-4 "": "./classes/" ; scripts.test: "phpunit"
```

Запуск: `composer test` (или `phpunit --bootstrap local/tests/bootstrap.php local/tests`). Автозагрузчик кастомных классов также регистрируется в `init.php` через `spl_autoload_register()`.

**Шаги CI-сборки** (ориентир, партнёр Intervolga, Jenkins; трудозатраты на тесты ≈ 1/10 времени разработки):

1. Создание копии проекта (изоляция от внешних факторов).
2. Синтаксический анализ (ошибки, копипаст).
3. Модульное тестирование (PHPUnit).
4. Функциональное тестирование в основных браузерах.
5. Email-уведомления с результатами.

## Проверка
- **Приёмка:** все 26 обязательных тестов «Монитора качества» закрыты или имеют комментарий разработчика; статус «Пройдено успешно» проставлен тестировщиком/менеджером; отчёт отправлен в архив. Чек-лист сдачи пройден.
- **Unit:** `composer test` (или прямой вызов `phpunit`) завершается без ошибок; в тестовых классах с SQL присутствует `$backupGlobalsBlacklist = ['DB']`; `CModule::IncludeModule()` нужных модулей возвращает `true`.
- **CI:** pipeline стартует на каждый push; версия Bitrix Framework на тестовом и боевом серверах совпадает; результат прогона приходит ответственному.
- **Нагрузка:** для крупного проекта зафиксированы эксплуатационные пределы по итогам нагрузочного теста.

## ⚠️ Риски
- ⚠️ **Провал приёмки без Монитора качества.** Если обязательные тесты не закрыты и не прокомментированы, проект формально не сдан: обязательный тест пропускается только с комментарием разработчика, а финальный статус ставит не программист. Сдача «на словах» оставляет риски на стороне исполнителя.
- ⚠️ **Пропущенное нагрузочное тестирование крупного проекта** — выход на боевой трафик без знания эксплуатационных пределов оборудования; отказ под нагрузкой обнаружится у пользователей, а не на стенде.
- ⚠️ **Глобальный `$DB` сломан в тестах.** Без `$backupGlobalsBlacklist = ['DB']` PHPUnit рвёт коннект между тестами — падение на первом SQL-запросе маскируется под «нестабильные тесты».
- ⚠️ **Bootstrap тестов c `NOT_CHECK_PERMISSIONS=true`, отданный по HTTP — публичный эндпоинт без проверки прав.** Каталог `/local/tests` рассчитан только на CLI-запуск через PHPUnit; в webroot он отдаётся веб-сервером. Закройте `/local/tests` deny-правилом (`Require all denied` для Apache, `deny all` для Nginx) и держите CLI-only — иначе любой неаутентифицированный посетитель инициализирует ядро с отключённой проверкой прав.
- **Рассинхронизация версий.** Версии Bitrix Framework на тестовом и боевом серверах должны совпадать — иначе зелёный CI не гарантирует поведение в продакшене.
- **Модификация ядра проваливает приёмку.** Тест «ядро не модифицировано» — обязательный; весь кастом держите в `/local`, иначе и обновление перетрёт правки, и тест нельзя закрыть.
- **Переинвестирование в unit-тесты.** Объём кода тестов может превысить объём проекта; для типового сайта на штатных компонентах базовая гигиена — приёмочный слой, а автотесты — прицельно на сложный кастом.
- **Часть тестов Монитора релевантна только в продакшене** (политики безопасности БД, удаление тестовых данных после выкладки): планируйте их на этап после выкладки, а отсутствие функционала в ТЗ помечайте комментарием, иначе тест не закрыть (партнёр Sibirix).

# Рецепт 06 — Вывод контента на публичной странице

> Основан на изучении ядра «1С-Битрикс: Управление сайтом» (SM_VERSION 26.x) и официальной документации вендора.

## Цель

Показать данные инфоблока (новости, статьи, услуги) на публичной странице сайта:
правильно «обернуть» страницу в шаблон сайта (`header.php`/`footer.php`) и вывести
список через готовый компонент `bitrix:news.list` с кэшированием.

## Когда применять

- Нужен список элементов инфоблока на конкретной странице (главная, раздел, лендинг).
- Контент уже смоделирован инфоблоком (см. рецепт `05-query-elements`) — выводим,
  а не пишем свой SQL.
- Нужен **один** список → берём `bitrix:news.list` напрямую.
  Нужен **целый раздел** (список + детальная + поиск + RSS на ЧПУ) → комплексный
  `bitrix:news` (см. рецепт `08-complex-component-sef`); здесь даём краткую сноску.

## Шаги

1. **Создай публичный файл-страницу** в document root. Страница в Битрикс — это реальный
   `.php`-файл между двумя `require`. Расположение файла = URL страницы:
   - `/news/index.php` → URL `/news/`
   - `/about/index.php` → URL `/about/`
   Не клади страницы в `/bitrix` и `/local` — это служебные зоны.
2. **Открой страницу** строкой `require($_SERVER["DOCUMENT_ROOT"]."/bitrix/header.php");`
   Это подключает пролог ядра и `header.php` шаблона сайта (asset-таргет `TEMPLATE`,
   затем граница рабочей области `RestartWorkarea()`).
3. **Поставь title/мета** через `$APPLICATION` (это единственный путь для title — D7-аналога
   нет). `news.list` при `SET_TITLE=>"Y"` ставит title сам из имени инфоблока.
4. **Вызови компонент** `$APPLICATION->IncludeComponent("bitrix:news.list", ".default", [...])`
   в рабочей области (между header и footer).
5. **Закрой страницу** строкой `require($_SERVER["DOCUMENT_ROOT"]."/bitrix/footer.php");`
   Подключает эпилог и `footer.php` шаблона (закрывает HTML, выводит подвал).
6. **Согласуй `DETAIL_URL`** с реальным местом детальной страницы (см. «Кратко про news.detail»).

## Рабочий сниппет

Файл: `/news/index.php` (в document root; для кастома компонент-шаблона —
`/local/templates/<site_template>/components/bitrix/news.list/.default/template.php`).

```php
<?php
require($_SERVER["DOCUMENT_ROOT"]."/bitrix/header.php");
/** @var CMain $APPLICATION */

$APPLICATION->SetTitle("Новости");
?>

<h1>Новости компании</h1>

<?php
$APPLICATION->IncludeComponent(
    "bitrix:news.list",
    ".default",
    [
        "IBLOCK_TYPE"  => "news",       // тип инфоблока
        "IBLOCK_ID"    => 1,            // ID активного инфоблока (число) или CODE (строка)
        "NEWS_COUNT"   => 12,           // размер страницы; при <=0 движок берёт 20

        // Сортировка: дописывается ID => DESC, чтобы порядок был детерминирован
        "SORT_BY1"     => "ACTIVE_FROM",
        "SORT_ORDER1"  => "DESC",
        "SORT_BY2"     => "SORT",
        "SORT_ORDER2"  => "ASC",

        // Какие поля и свойства тянуть в выборку и шаблон
        "FIELD_CODE"   => ["PREVIEW_PICTURE", "DATE_ACTIVE_FROM"],
        "PROPERTY_CODE"=> ["AUTHOR"],   // коды свойств этого инфоблока

        // URL детальной страницы — должен совпасть с местом news.detail
        "DETAIL_URL"   => "/news/#ELEMENT_ID#/",

        // Кэш: "A" = Авто (управляется настройками сайта). По тегам инфоблока
        // стандартный компонент сбрасывает кэш при изменении элементов из коробки.
        "CACHE_TYPE"   => "A",
        "CACHE_TIME"   => "36000000",   // верхняя граница; реальный сброс — по тегу
        "CACHE_GROUPS" => "Y",          // учитывать группы пользователя в ключе кэша

        // Заголовок и хлебные крошки — ставятся ВНЕ кэша, на каждом хите
        "SET_TITLE"        => "Y",
        "ADD_SECTIONS_CHAIN"=> "Y",

        // Пагинация
        "DISPLAY_BOTTOM_PAGER" => "Y",
        "PAGER_TITLE"          => "Новости",
        "NAV_PAGE_COUNT"       => 1,    // элементов навигации (см. .parameters.php)

        // Фильтры выборки
        "CHECK_DATES" => "Y",           // показывать только с активной датой
    ]
);
?>

<?php
require($_SERVER["DOCUMENT_ROOT"]."/bitrix/footer.php");
```

**Кратко про `news.detail`.** Детальную карточку выводит парный компонент `bitrix:news.detail`.
Чтобы ссылки из списка работали, в `/news/index.php` через `if (...)` обычно различают режим
списка и детали по GET-переменной (`ELEMENT_ID`), либо детальную кладут отдельным файлом
`/news/detail.php`, согласовав `DETAIL_URL`. Минимальный вызов:

```php
$APPLICATION->IncludeComponent("bitrix:news.detail", ".default", [
    "IBLOCK_TYPE" => "news",
    "IBLOCK_ID"   => 1,
    "ELEMENT_ID"  => (int)($_REQUEST["ELEMENT_ID"] ?? 0),
    "FIELD_CODE"  => ["DETAIL_TEXT", "DETAIL_PICTURE"],
    "PROPERTY_CODE"=> ["AUTHOR"],
    "SET_TITLE"   => "Y",
    "ADD_ELEMENT_CHAIN" => "Y",
    "CACHE_TYPE"  => "A",
    "CACHE_TIME"  => "36000000",
    "CACHE_GROUPS"=> "Y",
]);
```

Для целого раздела на ЧПУ удобнее один комплексный `bitrix:news` в SEF-режиме — он сам
разрулит подпути `section`/`detail`/`search`/`rss` (см. рецепт `08-complex-component-sef`).

## Выбор API (для этой задачи)

- **Вывод — через готовый компонент, а не свой запрос.** `bitrix:news.list` уже делает
  выборку (`CIBlockElement::GetList` в два прохода: короткий запрос ID с навигацией, затем
  полный по ID c `SetUrlTemplates`), кэш по тегам инфоблока, SEO-метаполя, кнопки
  инлайн-редактирования. Свой SQL дублировал бы это и терял бы кэш/права.
- **Обвязка страницы — legacy `$APPLICATION` (`CMain`).** Подключение компонентов
  (`IncludeComponent`), title/мета (`SetTitle`/`SetPageProperty`), крошки (`AddChainItem`)
  идут только через `$APPLICATION` — это одна из двух поддерживаемых версий API, и для
  публичной HTML-страницы у неё нет D7-замены.
- **`IBLOCK_ID`: число или CODE.** Компонент ветвится `is_numeric($arParams["IBLOCK_ID"])`:
  число → поиск по ID; строка → по `CODE` + `SITE_ID` (инфоблок должен быть привязан к сайту).
  Для переносимости между копиями сайта CODE надёжнее числового ID.
- **Кэш по умолчанию — `CACHE_TYPE=>"A"`.** «A» = режим определяется настройками сайта;
  стандартный `news.list` под управляемым кэшем регистрирует теги инфоблока и сбрасывается
  при изменении элементов. `CACHE_TIME` — лишь верхняя граница TTL.
- **Кастом дизайна — копией шаблона, не правкой компонента.** Свой `template.php` клади в
  `/local/templates/<site_template>/components/bitrix/news.list/.default/`. Логику — рядом в
  `result_modifier.php` (внутри кэша) или `component_epilog.php` (вне кэша). Файлы в
  `/bitrix/components` не правь — перезапишутся обновлением.

## Проверка

**Режим «только файлы» (без живого Битрикс):**
- Страница лежит в document root, путь = ожидаемому URL; не в `/bitrix` и не в `/local`.
- Файл открывается `require(... "/bitrix/header.php")` и закрывается `require(... "/bitrix/footer.php")` — пара на месте.
- `IncludeComponent` вызван между ними; имя компонента — `"bitrix:news.list"`.
- В массиве параметров есть `IBLOCK_TYPE`, `IBLOCK_ID`, `CACHE_TYPE`; `DETAIL_URL`
  синтаксически согласован с местом детальной страницы.
- PHP-синтаксис: `php -l /path/to/news/index.php` (если есть PHP CLI).

**Режим «живой Битрикс»:**
- Открыть URL страницы — список выводится, пагинация и ссылки на детальную работают.
- Заголовок и хлебные крошки соответствуют (`SET_TITLE=>"Y"`, `ADD_SECTIONS_CHAIN=>"Y"`).
- Изменить элемент инфоблока в админке → после сброса тега список обновляется (проверка
  тегированного кэша).
- В режиме правки (админ авторизован) на элементах появляются кнопки инлайн-редактирования.
- Пустой список вместо ошибки → проверить, что `IBLOCK_ID` указывает на активный инфоблок
  текущего сайта и что у элементов проставлена активность/дата.

## ⚠️ Риски

- ⚠️ **Несогласованный `DETAIL_URL`** ведёт на несуществующую страницу (ложный 404). Шаблон
  URL в списке и реальное размещение детального компонента должны совпадать.
- ⚠️ **Title/мета/крошки — вне кэша.** Не переноси `SetTitle`/`AddChainItem` внутрь
  кэшируемого `$arResult` или в `result_modifier.php` — они «застынут» и покажут чужие
  значения. Их место — постобработка компонента / `component_epilog.php`.
- ⚠️ **Динамический параметр в `$arParams` отравляет кэш.** Текущее время, случайные значения
  в параметрах компонента делают кэш-ключ уникальным на каждый хит — кэш перестаёт работать.
- ⚠️ **`FILTER_NAME` — имя глобальной переменной, не сам массив.** Имя валидируется маской
  `^[A-Za-z_][A-Za-z01-9_]*$`. При непустом фильтре и `CACHE_FILTER != "Y"` кэш отключается
  (`CACHE_TIME=0`) — это ожидаемое поведение, не сбой.
- ⚠️ **Composite-кэш может отдать статический HTML, минуя PHP.** Если на странице есть
  динамика, зависящая от пользователя, оборачивай её в динамическую область, иначе один
  пользователь увидит закэшированный фрагмент другого.

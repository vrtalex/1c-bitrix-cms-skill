# Миграции структуры и данных между средами

> **Класс исполнения: E3** — необратимая/привилегированная операция: на проде по умолчанию не исполнять (инструкция человеку или изолированная копия). См. [«0. Право исполнять»](../../conventions.md).

## Цель

Переносить структуру и данные («1С-Битрикс: Управление сайтом», ядро 26.x) между
средами (dev → stage → prod) воспроизводимо: инфоблоки, свойства, HL-блоки,
пользовательские поля (UF), настройки модулей и магазина. Цель — чтобы релиз
повторялся командой из git, а не ручной донастройкой админки на каждой среде.

## Когда применять

- Готовите релиз и нужно перенести новые/изменённые инфоблоки, свойства, HL-блоки,
  справочники, UF-поля или настройки между средами.
- Версионируете структуру собственного модуля и хотите контролировать её через релизы.
- Поддерживаете несколько копий проекта (dev/stage/prod) и нужен единый источник
  правды по схеме данных в репозитории.

В ядре Bitrix Framework нет встроенного «migrate up/down» для прикладной схемы
(инфоблоки, HL, UF, настройки). Ядро даёт версионирование **модулей**
(`install/index.php` + `version.php` + `updater.php`) и ORM для собственных таблиц.
Перенос прикладных сущностей закрывают community-инструменты.

## Шаги

1. **Опирайтесь на символьные коды и XML_ID, не на числовые ID.** Числовой ID
   инфоблока/свойства/HL различается между средами; ссылайтесь на сущности по
   символьному коду (`CODE`) или `XML_ID`. Это базовое условие переносимости.

2. **Для версионирования структуры собственного модуля** — штатный механизм Bitrix:
   `install/index.php` (класс — наследник `CModule`) + `install/version.php` (номер и
   дата версии) + `updater.php` для изменений структуры между версиями. Поднимайте
   номер в `version.php` при каждом релизе модуля.

3. **Для переноса инфоблоков/HL/UF/настроек между средами** выберите ОДИН
   community-инструмент и придерживайтесь его:
   - «миграции как код» (up/down, билдеры, теги релизов) → `sprint.migration` (партнёр);
   - «снимок состояния + автотрекинг изменений» → `ws.migrations` (партнёр);
   - «export/import XML по XML_ID» → `intervolga.migrato` (партнёр).

4. **Храните артефакты в git.** Файлы миграций / XML-выгрузки / конфиги кладите в
   репозиторий — это и есть версионирование данных.

5. **Прогоняйте миграции из CLI** при деплое. Консольный запуск снимает таймаут
   веб-сервера на долгих операциях и встраивается в пайплайн без интерактива.

6. **Структуру таблиц пишите через ORM / `SqlHelper`** для переносимости между
   MySQL и PostgreSQL (см. ниже про граблю с прямым SQL).

## Рабочий сниппет/конфиг

**Вариант A — `sprint.migration` (миграции как код, up/down, теги релизов).**
Каждое изменение — отдельный файл-миграция в каталоге `Update/Migrations`
(имена вида `Version20150520000001.php`), реализует `up()` и `down()`. Билдеры
(`IblockBuilder`, `HlblockBuilder`, `UserTypeEntities` и др.) генерируют миграцию
по текущему состоянию объекта.

```bash
# создать миграцию вручную / через билдер
php migrate.php add --name=AddCatalogProps --desc="свойства каталога"
php migrate.php run IblockBuilder        # сгенерировать по состоянию инфоблока

# список и применение
php migrate.php ls --new
php migrate.php up                        # применить непрогнанные
php migrate.php up --add-tag=release001   # пометить пачку тегом релиза

# откат по тегу релиза
php migrate.php down --tag=release001
```

Конфиг: `migrations.cfg.php` (основной) и `migrations.{name}.php` (под окружение).
Версии миграций должны быть уникальны — иначе конфликт при копировании проекта.

**Вариант B — `intervolga.migrato` (export/import XML по XML_ID).**
Сущности выгружаются в XML и привязываются по XML_ID; каталог `/local/migrato/`
(конфиг + выгрузки) обязательно в git.

```bash
php run.php validate        # проверить корректность XML_ID
php run.php autofix         # автопочинка XML_ID
php run.php export          # снять состояние БД в XML
php run.php import          # применить XML в БД
php run.php log --fails     # только ошибки
```

В коде вместо числовых ID — публичный ID по XML_ID:

```php
$id = \Intervolga\Migrato\Data\BaseData::getPublicId($xmlId);
```

**Workflow релиза dev → prod (intervolga.migrato):**

- На prod однократно: установить модуль → настроить `config.xml` →
  `validate` + `autofix` → `export` → добавить `/local/migrato/` в git.
- На dev (после клона): `git pull` → `validate` → `autofix` (при необходимости)
  → `export` (снимок) → `git commit & push`.
- На prod при релизе: `validate` → `autofix` → `export` (контроль неучтённых
  изменений) → `git pull` (разрешить конфликты) → `import`.

**Запуск из cron / деплоя без интерактива** (консольные команды ядра):

```bash
0 3 * * * cd /path/to/root && php bitrix/bitrix.php update:modules --no-interaction
```

> `bitrix/bitrix.php` — CLI-обёртка, которую создаёт настройка Composer (или
> делают вручную: `require __DIR__ . '/modules/main/cli/bitrix.php';`). Из корня
> документа путь к ней — `bitrix/bitrix.php`, из каталога `bitrix` — просто
> `bitrix.php` (тот же файл). Подробнее — рецепты `../setup/04-cli-console.md` и
> `../update/04-apply-update.md`.

## Проверка

- `php migrate.php ls --installed` (sprint) или `php run.php log` (migrato) — все
  ожидаемые миграции/сущности в статусе «применено».
- На целевой среде сравните символьные коды/XML_ID и состав свойств инфоблоков с
  исходной средой — они должны совпадать.
- Для собственного модуля: установка/обновление через админку проходит без ошибок,
  номер в `version.php` соответствует релизу.
- Прогон миграции повторно не ломает данные (идемпотентность — см. риски).
- Проект на ORM/`SqlHelper` устанавливается и на MySQL, и на PostgreSQL без правок.

## ⚠️ Риски

- ⚠️ **Жёсткие числовые ID ломают перенос.** Визуальный редактор админки генерирует
  код с числовыми ID, которые на другой среде указывают на другие (или
  несуществующие) сущности — это ведёт к порче данных на проде. Пишите код вручную,
  ссылайтесь на символьные коды / XML_ID.
- ⚠️ **Идемпотентность обновлений модуля.** Одно обновление может примениться
  повторно — код `updater.php` обязан проверять «уже сделано», иначе дублирование
  или разрушение структуры. Новый код из самого обновления в `updater.php`
  недоступен (`Class not found`); он становится доступен только после применения.
- ⚠️ **Уникальность версий миграций.** При копировании проекта между машинами
  совпавшие версии вызывают конфликты и пропуск изменений; контролируйте уникальность.
- ⚠️ **Долгие миграции на проде.** Прогоняйте из CLI — иначе таймаут веб-сервера
  оборвёт операцию на полпути и оставит структуру в промежуточном состоянии.
  Перед `import`/релизом на проде делайте резервную копию БД.
- **Прямой SQL и переносимость.** Модули на ORM переносятся между MySQL и
  PostgreSQL без правок; прямой SQL требует адаптации (типы, экранирование
  идентификаторов через `SqlHelper::quote()`, функции). Пишите структуру через ORM.
- **Один инструмент на проект.** Не смешивайте sprint.migration / ws.migrations /
  intervolga.migrato в одном проекте — выберите один и придерживайтесь его.

# 01. Git-workflow и CI/CD для проекта на «1С-Битрикс»

## Цель

Поставить версионирование и выкладку Битрикс-проекта так, чтобы в репозитории
жил только собственный код проекта, а ядро, контент, кэши и конфиги с паролями
оставались на сервере. Настроить релизный поток dev → stage → prod через
центральный репозиторий, предсказуемые откаты и базовый пайплайн
build → tests → deploy. Цель — лёгкий репозиторий, воспроизводимые релизы и
быстрый откат при сбое на проде.

## Когда применять

- Заводите git на новом или существующем сайте на «1С-Битрикс».
- Нужен `.gitignore`, который не тянет в репозиторий ядро `/bitrix`, `/upload`
  и кэши.
- Выстраиваете передачу кода между средами (dev/stage/prod) и регламент
  отката.
- Готовите CI/CD-пайплайн (GitLab CI, Jenkins и т. п.).

Перед этим dev-среда должна быть подготовлена (`../setup/05-dev-setup.md`):
там настраиваются `.settings_extra.php`, блокировка почты `ONLY_EMAIL` и запрет
индексации на dev/stage. Этот рецепт описывает именно git и выкладку.

## Шаги

1. **Версионировать только свой код в `/local`.** Весь кастом проекта держат в
   `/local` (подпапки `activities`, `components`, `gadgets`, `modules`,
   `php_interface` с `init.php` и `user_lang`, `templates`). У `/local`
   приоритет над `/bitrix`: при одинаковых именах подключается версия из
   `/local`, и папка не перезаписывается при обновлениях ядра. Это и есть
   фундамент версионируемого проекта.

2. **Решить, что коммитить.** В репозиторий идут: `/local` целиком (компоненты,
   модули, шаблоны, `init.php`, файлы миграций); по историческим причинам —
   `/bitrix/php_interface/init.php` и `/bitrix/templates/`, если шаблоны живут
   в классическом месте (для свежих проектов их лучше размещать в
   `/local/templates`). Принцип: в репозитории только то, что реально
   разрабатываете. (партнёр)

3. **Исключить из git ядро, контент, кэши, конфиги.** Не коммитить: ядро
   `/bitrix`, загруженный контент `/upload`, кэши
   (`/bitrix/cache`, `managed_cache`, `stack_cache`, `html_pages`, `tmp`),
   конфиги с доступами (`.settings.php`, `/bitrix/php_interface/dbconn.php`),
   `.htaccess` и `robots.txt` (различаются по средам), логи/бэкапы/дампы
   (`*.log`, `*.bak`, архивы, `*.sql`), `sitemap*.xml` (генерируются
   автоматически), служебные `*.restore.php` и `bitrix/backup/`. Готовый
   `.gitignore` — в разделе ниже. (партнёр)

4. **Развести конфиги по средам.** Боевой `.settings.php` с доступами лежит на
   сервере и в git не попадает. Локальные переопределения (своя БД, отладка)
   выносят в `.settings_extra.php` — Битрикс мержит его поверх `.settings.php`.
   С версии главного модуля 24.100.0 (актуально для 26.x) `.settings.php` и
   `.settings_extra.php` можно размещать в `/local`. Файл `.settings_extra.php`
   тоже добавляют в `.gitignore` — каждый разработчик создаёт его локально.
   (партнёр)

5. **Выстроить релизный поток через центральный репозиторий.** Потоки
   dev → stage (test/prelive) → prod; передача кода между серверами — только
   через центральный репозиторий (например GitLab), а не напрямую с сервера на
   сервер. Ветка `master` отражает prod, `stage` — предпрод. Задачу накатывают
   в отдельную ветку (например `TICKET#`), затем сливают и тянут на нужную
   среду через `git pull`. (партнёр)

6. **Версионировать данные БД миграциями.** Файлы версионирует git, но
   структура и часть данных (инфоблоки, свойства, HL-блоки, настройки модулей)
   живут в БД. Их переносят между средами модулем миграций (артефакты в git), а
   не правками руками в админке прода. Детали — `02-migrations.md`. (партнёр)

7. **Выбрать способ выкладки.** Простой вариант — `git pull` на сервере в
   нужную ветку (тогда собранные ассеты приходится коммитить). Для нулевого
   простоя — атомарный деплой через симлинк `current` (см. ниже).

8. **Зафиксировать регламент отката.** На stage откат — `git reset --hard HEAD`
   (тестовая среда, история не критична). Свежая ошибка на проде —
   `git revert HEAD` (создаёт обратный коммит, сохраняет историю). Критический
   откат — `git reset --hard <HASH>` на заведомо рабочий коммит. При атомарном
   деплое откат = переключение симлинка `current` на предыдущий релиз. ⚠️
   `git reset --hard` на проде уничтожает незакоммиченные правки безвозвратно —
   сначала убедитесь, что на сервере нет ручных хотфиксов вне git.

9. **Собрать CI/CD-пайплайн.** Этапы build → tests → deploy: артефакт
   (собранный фронт, папка модуля миграций) собирается и передаётся на deploy.
   Триггеры: коммит в ветку test → выкладка на stage; тег/коммит в prod-ветку →
   выкладка на prod. Composer в Битрикс применим (автозагрузка, зависимости в
   `/local`). (партнёр)

## Рабочий сниппет/конфиг

Типовой `.gitignore` для проекта на Битрикс (партнёрская практика, адаптируйте
под структуру проекта):

```gitignore
# Ядро и пользовательский контент
/bitrix/
!/bitrix/php_interface/
!/bitrix/templates/
/upload/

# Кэши и временные файлы
/bitrix/cache/
/bitrix/managed_cache/
/bitrix/stack_cache/
/bitrix/html_pages/
/bitrix/tmp/

# Конфиги с доступами и окружением
/bitrix/.settings.php
/local/.settings.php
.settings_extra.php
/bitrix/php_interface/dbconn.php

# Различаются по средам — не версионировать
.htaccess
/robots.txt
/sitemap*.xml

# Логи, бэкапы, дампы, служебные скрипты
*.log
*.bak
*.sql
*.zip
*.rar
*.tar.gz
/bitrix/backup/
*.restore.php

# Системное / IDE
.DS_Store
Thumbs.db
/.idea/
```

Атомарный деплой (структура каталогов; партнёрская практика, например через
Deployer):

```text
project/
├── releases/          # пронумерованные релизы (build-артефакты)
│   ├── 20260626-1/
│   └── 20260627-1/
├── current -> releases/20260627-1   # симлинк на текущий релиз
├── shared/            # общие между релизами: upload, .settings.php, dbconn.php
│   ├── upload/
│   └── bitrix/.settings.php
└── .dep/              # метаданные деплоера: логи, lock
```

Выкладка: собрать новый релиз в `releases/`, проставить симлинки на `shared`,
затем переключить `current` (почти мгновенно). Откат — переключить `current`
на предыдущий релиз.

## Проверка

- `git status` в корне рабочей копии не показывает `/bitrix`, `/upload`,
  каталоги кэша, `.settings.php`/`dbconn.php` как неотслеживаемые или
  изменённые.
- `git ls-files | grep -E '^(bitrix/cache|upload)/'` ничего не возвращает.
- Размер репозитория остаётся небольшим (бинарь и контент не коммитятся).
- На чистой копии после `git pull` сайт поднимается, когда на месте оказались
  локальные `.settings.php`/`dbconn.php` и контент `/upload` (то, что вне git).
- Откат проверен на stage: `git revert`/переключение симлинка возвращает
  предыдущее рабочее состояние.

## ⚠️ Риски

- ⚠️ **Доступы в репозитории.** `.settings.php` и `dbconn.php` содержат логины,
  пароли БД и ключи. Если их закоммитили — это утечка: удалите из индекса
  (`git rm --cached`), добавьте в `.gitignore`, при попадании в удалённый
  репозиторий смените скомпрометированные пароли.
- ⚠️ **`git reset --hard` на проде** удаляет незакоммиченные изменения без
  возможности восстановления. Перед откатом убедитесь в отсутствии ручных
  правок на сервере; при наличии бэкапа держите его под рукой.
- ⚠️ **Ручные правки БД на проде вместо миграций** невозможно воспроизвести на
  других средах — расхождение схемы. Все структурные изменения вести
  миграциями (`02-migrations.md`).
- **`.gitignore` не настроен с самого начала** → в репозиторий попадают ядро,
  кэш, конфиги и дампы, репозиторий разрастается. Настраивайте до первого
  коммита.
- **Демо-ключ на проде** → сайт перестанет работать по истечении срока. На
  dev/stage — некоммерческий ключ разработки, на prod — боевой. (партнёр)
- **Числовые ID инфоблоков/свойств в коде** ломаются при переносе между
  средами — опирайтесь на символьные коды/`XML_ID`. (партнёр)
- **Письма и индексация с stage** — следите, чтобы на stage были включены
  `ONLY_EMAIL` и запрет индексации (`../setup/05-dev-setup.md`); поэтому
  `robots.txt` не версионируется.

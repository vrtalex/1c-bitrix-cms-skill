# Рецепт 03 — backup-restore: резервное копирование и восстановление «1С-Битрикс: Управление сайтом» (ядро 26.x)

> Подсистема: 1c-bitrix-cms-deploy. Ядро БУС 26.x. Две поддерживаемые версии API (D7 и legacy) — обе официальны.
> Штатные инструменты бэкапа/восстановления подтверждены вендорской документацией; шаги мастера `restore.php` реконструированы из его HTTP-форм.

> **Класс исполнения: E3** — необратимая/привилегированная операция: на проде по умолчанию не исполнять (инструкция человеку или изолированная копия). См. [«0. Право исполнять»](../../conventions.md).

## Цель

Создать восстановимую резервную копию сайта (дамп БД + файлы в `*.tar.gz`) и восстановить из неё сайт — на том же или новом окружении — через официальный браузерный `restore.php` или мастер восстановления из админки. Обеспечить, чтобы копия была полной, переносимой и безопасно хранимой.

## Когда применять

- Регулярный бэкап прод-сайта (ручной или по расписанию).
- ⚠️ Обязательная копия ПЕРЕД обновлением ядра/модулей, миграцией БД, переносом на другой хостинг.
- Перенос сайта на новое окружение (staging → prod, смена хостинга).
- Аварийное восстановление после сбоя диска, неудачного релиза или порчи данных.

## Шаги

### A. Создание резервной копии

1. **Выбрать способ создания.** Админка: `Настройки → Инструменты → Резервное копирование → Создание резервной копии` → выбрать место хранения (папка сайта / облако 1С-Битрикс / стороннее облако) → «Создать». CLI: `php -f <DOCROOT>/bitrix/modules/main/tools/backup.php`.
2. **Исключить лишнее из БД** (экспертные настройки): таблицы статистики, поискового индекса, журнала событий. Это уменьшает архив и ускоряет восстановление.
3. **Исключить лишние файлы:** кеш, временные данные, файлы по маске, файлы больше лимита и — обязательно — папку с резервными копиями (иначе рекурсия). Большой `/upload` имеет смысл архивировать отдельно: `tar -czvf upload.tar.gz ./upload`.
4. **Задать размер тома.** Многотомный архив: максимум одной части — 200 МБ, оптимально 100 МБ. Большой сайт даст много томов.
5. **Включить шифрование** (опция «Шифровать данные резервной копии»). Для облака 1С-Битрикс шифрование обязательно и не отключается. ⚠️ Запомнить/сохранить пароль шифрования — без него восстановление невозможно (см. Риски).
6. **Снизить нагрузку при создании** (по желанию): параметры «Длительность шага» и «Интервал» (сек) дробят процесс, разгружая сервер.

### B. Регулярное (автоматическое) копирование

1. Включить: `Настройки → Инструменты → Резервное копирование → Регулярное резервное копирование` — хранилище, расписание, ротация локальных копий. Минимальная частота — 1 день.
2. Для фоновых задач (включая автобэкап) скрипт `<DOCROOT>/bitrix/modules/main/tools/cron_events.php` должен запускаться по cron каждую минуту.
3. Контроль: `Настройки → Инструменты → Резервное копирование → Журнал резервного копирования` — статусы успеха/ошибок.

### C. Облачный бэкап 1С-Битрикс

1. Автозагрузка копий в облако доступна на коммерческих лицензиях с активной техподдержкой (кроме редакции «Первый сайт»); нужен модуль «Облако 1С-Битрикс».
2. Облако хранит максимум 3 последние копии на лицензию; старые удаляются автоматически. ⚠️ Не полагаться на облако как на единственный бэкап.
3. Все облачные копии хранятся только зашифрованными заданным паролем. Размещение зависит от страны покупки лицензии: Yandex Object Storage (РФ, Беларусь, Казахстан) или Amazon S3 (прочие страны).

### D. Восстановление через restore.php

1. ⚠️ Сделать резервную копию текущего состояния целевого окружения ДО восстановления — операция перезаписывает БД и файлы.
2. Скачать актуальный `restore.php` **только по HTTPS с официального домена 1С-Битрикс**, сверить контрольную сумму (или записать собственный хеш заведомо корректной копии); никогда не запускать копию, полученную по открытому `http://`. Положить в **корень** нового сайта. Владельцем файла должен быть системный пользователь сайта (`bitrix` в стандартном окружении).
3. Загрузить туда же ВСЕ части архива (для многотомного — все тома). ⚠️ Архив Битрикса содержит полный дамп БД (хеши паролей, ПД, секреты) и параметры подключения из `.settings.php`. Пока тома лежат в веб-каталоге, они доступны на скачивание любому. Закрыть их от HTTP (`deny from all` в `.htaccess`/`location` каталога) либо держать архивы **вне** веб-корня и указывать мастеру абсолютный путь; для архива, лежащего в веб-каталоге, желательно шифрование.
4. ⚠️ `restore.php` — неаутентифицированный эндпоинт: любой, кто откроет его, перезапишет весь сайт и БД. ДО открытия ограничить доступ к нему IP оператора (allow-list в конфиге веб-сервера или `.htaccess`) и/или basic-auth; снять ограничение только ПОСЛЕ удаления скрипта.
5. Открыть `https://новый-домен/restore.php` → «Далее» → выбрать архив-источник → дождаться распаковки. ⚠️ Форма передаёт параметры БД и логин администратора; никогда не отправлять их на публичный домен по открытому `http://`. Если TLS на новом хосте ещё не поднят — открывать `restore.php` через SSH-туннель на `localhost` либо за временным IP allow-list.
6. Ввести параметры новой БД (хост, логин, пароль, имя); при переносе на другой хостинг отметить «Создать базу данных, если не существует». Параметры пишутся в `.settings.php`.
7. Выбрать режим: **полное восстановление** (файлы + БД) или **только файлы** (БД не трогается — при частичной потере файлов).
8. ⚠️ Финальной кнопкой мастера удалить локальную копию архива и сам `restore.php`. Если мастер прерван/завершился с ошибкой — удалить безусловно вручную (та же очистка, что в E3):

   ```sh
   rm -f "<DOCROOT>/restore.php" "<DOCROOT>"/*.tar.gz
   ```

9. Снять временное ограничение доступа (allow-list/basic-auth) — только после того, как `restore.php` и архивы удалены.
10. Очистить кеш (`bitrix/cache/`, `bitrix/managed_cache/` или через админку), проверить доступность сайта и права.

Восстановление из админки (без скачивания скрипта): `Настройки → Инструменты → Резервное копирование → Список резервных копий → Восстановить`.

## Рабочий сниппет/конфиг

Создание копии из CLI и пошаговая загрузка томов при HTTP 413. Скелет для скрипта эксплуатации; подставить реальные пути и адрес окружения.

```sh
#!/bin/sh
# backup-and-stage.sh — снять копию из CLI и подготовить тома к restore.php.
set -eu

DOCROOT="/home/bitrix/www"          # корень сайта
BACKUP_DIR="$DOCROOT/bitrix/backup" # сюда ядро кладёт архивы (имя содержит дату)

# 1. Создать копию из консоли (использует настройки бэкапа сайта).
php -f "$DOCROOT/bitrix/modules/main/tools/backup.php"

# 2. Скопировать тома на внешний носитель/в облако: локальное хранилище
#    уязвимо к сбою диска — дублируем обязательно.
rsync -av "$BACKUP_DIR"/*.tar.gz /mnt/external/bitrix-backups/

# 3. При переносе: restore.php + все тома в корень целевого сайта,
#    владелец — системный пользователь сайта. restore.php брать только
#    по HTTPS с официального домена и сверять контрольную сумму.
#    Тома содержат дамп БД и секреты: закрыть каталог от HTTP
#    (deny from all) либо держать архивы вне веб-корня и указать мастеру
#    абсолютный путь. Ограничить доступ к restore.php по IP до открытия.
TARGET="/home/bitrix/www-new"
cp /path/to/restore.php "$TARGET/restore.php"
cp "$BACKUP_DIR"/*.tar.gz "$TARGET/"
chown bitrix:bitrix "$TARGET"/restore.php "$TARGET"/*.tar.gz
```

При загрузке томов через веб-форму на слабом хостинге может возникать `HTTP 413 Request Entity Too Large`. Тогда грузить тома порциями по 9–10 файлов, при необходимости уменьшая партию. Перед запуском мастера убедиться, что в корне лежат **все** тома архива.

## Проверка

```sh
# Архив(ы) созданы и не пустые
ls -lh <DOCROOT>/bitrix/backup/*.tar.gz

# Тома читаются как корректный gzip (целостность контейнера)
gzip -t <DOCROOT>/bitrix/backup/*.tar.gz && echo "архивы целы"
```

- Журнал резервного копирования (`Настройки → Инструменты → Резервное копирование → Журнал`) показывает статус «успешно».
- После восстановления: сайт открывается на целевом домене; в корне НЕ осталось `restore.php` и загруженных архивов.
- Восстановленная админка `/bitrix/admin/` пускает администратора; «Проверка системы» — без красных пунктов.
- Контрольное тестовое восстановление на отдельном окружении периодически подтверждает, что копии действительно восстановимы (не только создаются).

## ⚠️ Риски

- ⚠️ **Пароль шифрования бэкапа.** Забытый пароль делает восстановление невозможным: 1С-Битрикс не имеет доступа к содержимому и не восстанавливает пароль. Хранить пароль в безопасном месте отдельно от копий.
- ⚠️ **Бэкап перед обновлениями/миграциями/переносом.** Любое обновление ядра, миграция БД или восстановление поверх существующего сайта — только после свежей резервной копии.
- ⚠️ **Удалить служебные скрипты после восстановления.** Оставленный в корне `restore.php` — путь к перезаписи сайта посторонним. Удалять финальной кнопкой мастера вместе с архивами.
- ⚠️ **Облако хранит только 3 копии** и доступно не на всех редакциях — это дополнительный, а не единственный канал хранения. Локальные копии дублировать на внешний носитель.
- Автобэкап при ошибке повторяется только через 24 часа; вручную копию можно создать в любой момент. Следить за Журналом.
- Многотомный архив: грузить ВСЕ тома; при `HTTP 413` — партиями по 9–10 файлов.
- Владелец `restore.php` и архивов должен быть системным пользователем сайта (`bitrix`), иначе мастер не запишет файлы.
- Несовпадение кодировки архива и БД при восстановлении даёт `ERROR 1062` (дубли ключей) — привести `character_set_server` к кодировке архива.
- После переноса проверить права (`0755`/`0644`), параметры подключения в `.settings.php`/`dbconn.php` и очистить кеш.

## Воспроизводимое развёртывание базы (Режим 3)

Режим 3 из [../setup/03-installation.md](../setup/03-installation.md) — развёртывание из чистого base-бэкапа: единожды снятая копия полностью установленного сайта используется как эталон для последующих воспроизводимых развёртываний. Подходит для CI, DevOps-пайплайна, быстрого поднятия staging-копии без прохода установочного мастера.

### E1. Снять чистый base-бэкап

Выполняется один раз с уже завершённой полной установки — в бэкапе оказывается весь публичный слой (шаблоны, компоненты, контент), поэтому после restore визуальный редактор работает без дополнительных шагов.

```bash
# Снять копию штатным механизмом ядра (как в разделе A).
# backup.php использует сохранённые настройки бэкапа сайта —
# исключения таблиц (статистика, поиск) и файлов задаются там, а не флагами.
php -f "$DOCROOT/bitrix/modules/main/tools/backup.php"
```

Перед прогоном в настройках бэкапа сайта (`Настройки → Инструменты → Резервное копирование`) исключить из БД таблицы статистики и поискового индекса, из файлов — кеш и временные данные: это уменьшает архив и ускоряет restore. Альтернатива консоли — мастер «Создание резервной копии» в админке.

Записать sha256 полученного эталонного архива — он понадобится для проверки подлинности перед restore в E2:

```bash
sha256sum "$DOCROOT"/bitrix/backup/clean_base*.tar.gz > clean_base.sha256
```

⚠️ Base-архив должен быть **доверенной, самостоятельно снятой** копией полной установки, а не сторонним архивом из непроверенного источника.

Подробнее о штатном бэкапе: раздел [A. Создание резервной копии](#a-создание-резервной-копии) выше.

### E2. Развернуть базу на новой среде

1. Скачать `restore.php` только по HTTPS с официального домена 1С-Битрикс и сверить контрольную сумму (или хеш заведомо корректной копии); не запускать копию, полученную по открытому `http://`. Положить `restore.php` и архив бэкапа в **корень** целевого сайта; владелец — системный пользователь сайта (`bitrix`). Архив содержит дамп БД и секреты — закрыть его от HTTP (`deny from all` в `.htaccess` каталога) либо держать вне веб-корня и указать мастеру абсолютный путь.
2. ⚠️ До открытия `restore.php` сверить sha256 base-архива с записанным в E1 (`sha256sum -c clean_base.sha256`) — это подтверждает подлинность, а не только целостность контейнера.
3. ⚠️ Ограничить доступ к `restore.php` IP оператора (allow-list/basic-auth) ДО открытия — это неаутентифицированный эндпоинт, перезаписывающий весь сайт и БД.
4. Запустить мастер: `https://целевой-домен/restore.php` → выбрать архив → ввести параметры БД (хост, пользователь, пароль, имя). Параметры пишутся в `.settings.php` и `dbconn.php`.
5. Режим восстановления: **полное восстановление** (файлы + БД) — стандартный вариант для Режима 3.
6. После удаления скрипта (E3) снять ограничение доступа.

Детали прохода мастера `restore.php` — раздел [D. Восстановление через restore.php](#d-восстановление-через-restorephp) выше.

### E3. Удалить служебные скрипты и пройти гейт завершённости

⚠️ После завершения мастера удалить из корня сайта все установочные и служебные скрипты:

```bash
rm -f restore.php *.tar.gz bitrixsetup.php bx_1c_import.php bitrix_server_test.php
```

Оставленные скрипты обнаруживает Сканер безопасности вендора; `restore.php` в корне открывает возможность перезаписи сайта.

Гейт завершённости (проверка, что сайт полностью функционирует): см. раздел «Проверка» в [../setup/03-installation.md](../setup/03-installation.md).

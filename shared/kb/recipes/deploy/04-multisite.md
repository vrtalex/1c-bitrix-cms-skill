# 04. Мультисайт и мультиязычность

## Цель

Поднять и эксплуатировать несколько сайтов на одной копии «1С-Битрикс:
Управление сайтом» (ядро 26.x): выбрать модель (разные домены через
`DOCUMENT_ROOT` + симлинки на ядро, либо несколько сайтов по папкам одного
корня), корректно настроить определение текущего сайта, привязать к каждому
сайту инфоблоки, шаблоны и почтовые события, развести SEO (sitemap.xml /
robots.txt / зеркала). Мультиязычность здесь — частный случай мультисайта:
отдельные сайты с разными `LANGUAGE_ID` и раздельным контентом (встроенного
автоперевода нет).

Не путать три уровня «языка»:

- **`SITE_ID` / `LID`** (таблица `b_lang`) — идентификатор сайта (2 символа:
  `ru`, `en`, `s1`); им помечается весь контент, привязки, заказы.
- **`LANGUAGE_ID`** (региональные настройки сайта в `b_lang`) — формат
  даты/времени, кодировка, валюта, язык контента публичной части.
- **Язык интерфейса** (таблица `b_language`) — язык административной части,
  независим от публички.

## Когда применять

- Нужно обслуживать несколько брендов/доменов из одной админки и БД.
- Нужна языковая версия сайта (`/en/`, `en.site.ru`) с раздельной статистикой,
  валютами и доступом к заказам.
- Часть контента общая (один каталог инфоблоков), часть — посайтовая
  (меню, шаблоны, тексты, набор товаров).
- Поднимается второй сайт, а гостям выдаётся требование авторизации или
  чужой контент — нужно разобраться с `.access.php` и определением сайта.

## Шаги

1. **Выбрать модель.** Многосайтовость — это неограниченное число сайтов на
   **одной лицензии**: одно ядро (`/bitrix`), одна БД, одна админка (лимит на
   число сайтов есть только в младших редакциях «Первый сайт» / «Старт»).
   Варианты:
   - **Разные домены** — каждый сайт в своём `DOCUMENT_ROOT`, «Папка сайта»
     = `/`, ядро лежит в одном общем каталоге, остальные сайты обращаются к
     нему через **симлинки**.
   - **По папкам одного домена** — все сайты в подпапках одного корня
     (`/s1/`, `/en/`, `/shop/`), один `DOCUMENT_ROOT`, сайт различается по
     «Папке сайта».

2. **Понять алгоритм определения текущего сайта** (двухуровневый):
   1. **По доменному имени** — текущий домен сопоставляется с полем «Доменное
      имя». Сравнивается правая часть до точки, поэтому **все поддомены
      автоматически относятся к основному домену** (`www.example.com` и любой
      поддомен подойдут под `example.com`).
   2. **По «Папке сайта»** — если домены не заданы или совпадают, сайт
      определяется по пути в URL (путь относительно корня публички, **не**
      путь в файловой системе).
   Порты для разделения сайтов **не используются**. ID текущего сайта в
   публичной части — константа `SITE_ID`.

3. **Учесть сортировку для поддоменов.** Если второй сайт — поддомен первого
   (`en.site.ru`), его **индекс сортировки должен быть МЕНЬШЕ** (выше в
   списке), иначе система отдаст контент основного сайта (поддомен попадёт под
   правило основного домена).

4. **Поднять второй сайт на отдельном домене** (см. сниппет):
   - вынести ядро (`bitrix`, `upload`, `local`) в общий каталог;
   - в корне нового сайта создать симлинки на эти каталоги;
   - скопировать в корень нового сайта `index.php`, `.htaccess`, `404.php`
     (правила ЧПУ и страница 404 — с основного);
   - создать `.access.php` с правом чтения для гостей —
     **без него второй сайт потребует авторизации даже у гостей**.

5. **Или поднять сайт по папке** одного домена:
   - сайты — в отдельных непересекающихся подпапках (`/s1/`, `/s2/`);
   - «Доменное имя» — пусто (или одинаковое для всех);
   - «Папка сайта» — `/s1/`, `/s2/`;
   - «Путь к корневой папке веб-сервера» — пусто для всех;
   - дополнительная правка `httpd.conf` не требуется.

6. **Создать сайт в админке** (Настройки → Настройки продукта → Сайты →
   Список сайтов). Ключевые поля: **ID** (2 символа = `LID`), **Доменное имя**,
   **Папка сайта**, **Сортировка** (для поддомена — меньше основного),
   **Название сайта** (в письма), **URL сервера** (абсолютные ссылки в
   письмах), **E-mail по умолчанию** (`DEFAULT_EMAIL_FROM`), **Путь к корневой
   папке веб-сервера** (`DOCUMENT_ROOT`: для разных доменов — заполнить, для
   одного домена — пусто), **Язык / `LANGUAGE_ID`** и региональные настройки.

7. **Привязать инфоблоки.** Инфоблоки и компоненты — единое хранилище для всех
   сайтов; разделение делается на уровне выборок (условия по `SITE_ID`), разных
   шаблонов компонентов, разных меню и привязки самого инфоблока к сайтам.
   В форме инфоблока поле **«Сайты»** (таблица связи `b_iblock_site`) — отметить
   сайты, на которых инфоблок виден; для каждого сайта задать **URL-шаблоны**
   страниц (список / раздел / детальная) с макросами `#SITE_DIR#`,
   `#SERVER_NAME#`, `#IBLOCK_CODE#`, `#SECTION_CODE#`, `#ELEMENT_CODE#` — так
   получаются разные URL под разные сайты.

8. **Привязать шаблоны сайта.** У сайта может быть несколько шаблонов; система
   перебирает их **по возрастанию сортировки** и применяет первый, чьё условие
   выполнилось. Условия: по папке/файлу, по группам пользователей, по периоду
   времени, по параметру в URL. Сам набор шаблонов привязывается в настройках
   сайта (раздел «Шаблоны сайта»).

9. **Привязать почтовые события.** У почтового шаблона (`b_event_message`,
   `CEventMessage`) есть поля **«Сайт» (`LID`)** и **«Язык»**: один тип события
   может иметь **разные шаблоны для разных сайтов**. При отправке
   `CEvent::Send($eventName, $arLID, $arFields)` система ищет шаблоны с этим
   `EVENT_NAME` и **фильтрует по `LID`**. Для многоязычного сайта язык сайта и
   язык шаблона **должны совпадать**.

10. **Разнести SEO** (Маркетинг → Поисковая оптимизация):
    - **sitemap.xml** — при создании карты **выбирается сайт**, для каждого
      домена/сайта своя карта; автогенерация по инфоблокам добавляет элементы
      по правилам;
    - **robots.txt** — визуальный редактор директив (можно с правилами под
      разные ПС); для каждого сайта/региона свой файл;
    - **зеркала** — закрываются редиректом на основной домен в `.htaccess`,
      иначе дубли в поиске;
    - canonical / 404 / LastModified — SEO-инструменты включаются глобально
      или для отдельных страниц.

## Рабочий сниппет/конфиг

Структура для модели «разные домены» (общее ядро + симлинки):

```
/home/www/shared/   (ядро: bitrix, upload, local)
/home/www/site1/    (основной сайт)
/home/www/site2/    (второй сайт — симлинки на shared)
```

Симлинки в корне каждого дополнительного сайта (UNIX):

```bash
ln -s /home/www/shared/bitrix /home/www/site2/
ln -s /home/www/shared/upload /home/www/site2/
ln -s /home/www/shared/local  /home/www/site2/
```

`.access.php` в корне второго сайта — открыть чтение гостям:

```php
<?php $PERM["/"]["*"] = "R"; ?>
```

Поля формы сайта при модели «разные домены»:

| Параметр | site1 | site2 |
|---|---|---|
| Доменное имя | site1.ru | site2.ru |
| Папка сайта | / | / |
| URL сервера | www.site1.ru | www.site2.ru |
| Путь к корневой папке (DOCUMENT_ROOT) | /home/www/site1/ | /home/www/site2/ |

Разнесение кеша/шаблонов при HTML-кешировании на разных доменах
(в конфиге Apache соответствующего сайта) — плюс симлинки на
`/bitrix/php_interface` и `/bitrix/templates`:

```apache
SetEnv BX_PERSONAL_ROOT /bitrix_personal
```

## Проверка

1. Открыть каждый домен/папку в браузере под гостем: отдаётся **свой** контент,
   нет требования авторизации (если есть — проверить `.access.php`).
2. В шаблоне публички вывести `SITE_ID` и убедиться, что на каждом сайте он
   корректный (для поддомена — проверить, что сортировка меньше основного).
3. В форме инфоблока (поле «Сайты») сверить привязку и URL-шаблоны; открыть
   список/детальную на каждом сайте — ссылки ведут на правильный домен/папку.
4. Отправить тестовое письмо события на каждом сайте — приходит шаблон,
   привязанный к нужному `LID`, с корректным языком и адресом отправителя.
5. Запросить `/sitemap.xml` и `/robots.txt` на каждом домене — отдаются
   раздельные файлы; зеркала редиректятся на основной домен.
6. Проверить, что обновления ядра ставятся (симлинки целы, `/bitrix` не
   скопирован).

## ⚠️ Риски

- ⚠️ **Лицензия.** На несколько копий продукта нужен отдельный лицензионный
  ключ на каждую копию; многосайтовость в пределах одной копии (одно ядро +
  одна БД) — в рамках одной лицензии. Языковая версия как отдельный сайт той же
  копии **отдельной лицензии не требует**.
- ⚠️ **Не копировать `/bitrix`.** Дублирование каталога ядра нарушает условия
  лицензии и ломает механизм обновлений — только симлинки.
- ⚠️ **Проактивная защита.** Все домены нужно добавить в список разрешённых
  хостов, иначе сайт отвечает 403.
- ⚠️ **`.access.php`.** Без файла с правом `R` для гостей второй сайт закроется
  авторизацией — внешне выглядит как «сайт не работает».
- **Симлинки на shared-хостинге.** При `open_basedir` создание симлинков из PHP
  может быть заблокировано — нужна помощь хостера; на VPS возможен
  `mount --bind`.
- **HTML-кеш на разных доменах.** Без разнесения через `BX_PERSONAL_ROOT`
  кеш/шаблоны смешиваются между сайтами.
- **Мультиязычность.** Встроенной поддержки нескольких языков в одном элементе
  инфоблока нет: полноценный многоязычный сайт требует дублирования контента
  под каждый язык (или свойства-перевода) и синхронизации. Фразы публички
  переводятся модулем «Перевод» (`$MESS`-массивы); переключатель языка
  реализуется в шаблонах сайта с сохранением выбора. Альтернатива дублированию
  инфоблоков — собственный тип UF-поля «привязка к сайту/языку» и фильтрация
  выборок по нему; готовые свойства такого рода есть в Маркетплейс.

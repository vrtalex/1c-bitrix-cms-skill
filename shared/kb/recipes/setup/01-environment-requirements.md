# 01. Системные требования и окружение

## Цель

Подготовить серверное окружение для «1С-Битрикс: Управление сайтом» (ядро Main 26.x): корректные версии PHP/БД/веб-сервера, обязательные PHP-расширения, параметры `php.ini`, кодировку UTF-8, OPcache, права на файлы — и подтвердить соответствие официальными средствами проверки до установки продукта.

## Когда применять

- Перед первой установкой продукта на новый сервер или хостинг.
- При выборе/смене хостинга — чтобы заранее убедиться в пригодности площадки.
- При обновлении ядра до 26.x, когда поднялась минимальная планка PHP/БД.
- При диагностике проблем, связанных с конфигурацией окружения.

## Шаги

1. **Зафиксировать целевые версии ПО** (требования вендора, https://dev.1c-bitrix.ru/user_help/reqintro.php).
   - PHP **8.2.0** — минимум (требование с 01.02.2026). Рекомендуется **PHP 8.4+**. Пока PHP не поднят минимум до 8.2, система обновлений коробки блокирует установку любых обновлений.
   - БД: **MySQL 8.0+** (мастер установки требует с Main 24.0.200, кодировка БД по умолчанию — **utf8mb4**) или **MariaDB 10.x/11.x**; **PostgreSQL 11+** доступен только для редакции Энтерпрайз. Только UTF-8 (с Main 24.0.0 поддержка windows-1251 прекращена).
   - Веб-сервер: **Apache 2.0+** (с `.htaccess`) или **Nginx** (актуальная стабильная версия). В типовом окружении BitrixEnv/BitrixVM применяется связка Nginx (frontend) + Apache 2.4.x (backend).

2. **Установить обязательные PHP-расширения** (для работы ядра):
   GD + FreeType (генерация изображений, CAPTCHA), PHP XML/DOM (система обновлений), Zlib, OpenSSL, mbstring (строки в UTF-8), Hash, ZIP, и `mysqli` либо `pgsql` по типу БД. Расширение LDAP — при интеграции с Active Directory/LDAP.

3. **Привести `php.ini` к требованиям** (см. сниппет ниже). Ключевые пороги: `memory_limit` (≥128 Мб для «Старт», ≥256 Мб для «Бизнес»), `max_input_vars=10000`, `max_execution_time=300`, `post_max_size`/`upload_max_filesize=1024M`, `file_uploads=On`, `default_charset=UTF-8`.

4. **Обеспечить кодировку UTF-8.** С версии Main **24.0.0** продукт работает только в UTF-8 (см. раздел про конвертацию в ⚠️). Кодировка БД — **utf8mb4** (не utf8mb3); сортировка для MySQL 8.x/9.x — `utf8mb4_0900_ai_ci`, для MariaDB — `utf8mb4_unicode_ci`.

5. **Удалить настройку `mbstring.func_overload`.** ⚠️ С версии модуля Main **20.100.0** она не требуется и не поддерживается платформой; её наличие вызывает ошибку проверки системы. Настройку нужно убрать из `php.ini` (а не выставлять в `0` через виртуальную директиву).

6. **Настроить OPcache.** Выделить 32–64 Мб (до 128 Мб с учётом админ-раздела), задать каталог хранения скомпилированного кода, `opcache.max_accelerated_files=100000`, `opcache.revalidate_freq=0`. Отсутствие каталога для кэша — одна из самых частых ошибок настройки.

7. **Выставить права на файлы и папки.** По умолчанию **0644 для файлов** и **0755 для папок** (документированные значения вендора `BX_FILE_PERMISSIONS`/`BX_DIR_PERMISSIONS`). Пользователь веб-сервера должен иметь чтение и запись от корня сайта. Учтите: 0644/0755 доступны на чтение всем (world-readable), поэтому на неизолированном shared-хостинге соседний uid может прочитать конфиги с реквизитами БД. От соседей по площадке защищает именно изоляция (`open_basedir`, отдельный vhost, per-user PHP-FPM) либо более строгие права на секретонесущие файлы (см. ниже).

8. **Обеспечить локализацию ПДн на инфраструктуре в РФ.** Если сайт собирает
   персональные данные граждан РФ (формы, регистрация), первичные сбор, запись,
   систематизация, накопление и хранение должны вестись на серверах/БД, физически
   расположенных в России (152-ФЗ ст. 18 ч. 5 / 242-ФЗ). Это касается и БД самого
   сайта, и любых внешних сервисов (CRM, аналитика), получающих данные форм — выбор
   хостинга/площадки делается с учётом этого требования.

9. **Проверить соответствие** скриптом `bitrix_server_test.php` и формой «Проверка системы» в админке (см. «Проверка»).

## Рабочий сниппет/конфиг

`php.ini` — целевые значения для ядра 26.x:

```ini
; --- Версии и кодировка ---
default_charset = UTF-8

; --- Память и лимиты ---
memory_limit = 256M           ; ≥128M для «Старт», ≥256M для «Бизнес»
max_input_vars = 10000
max_execution_time = 300
post_max_size = 1024M
upload_max_filesize = 1024M
file_uploads = On

; --- Рекомендуемые ---
short_open_tag = On
allow_url_fopen = On

; --- Безопасность ---
expose_php = Off
enable_dl = Off
mail.add_x_header = Off

; --- Продакшн: ошибки в лог, не на экран ---
display_errors = Off
error_reporting = E_ALL
error_log = /var/log/php/error.log

; --- OPcache ---
opcache.enable = 1
opcache.max_accelerated_files = 100000
opcache.revalidate_freq = 0
; opcache.memory_consumption: 32–64M (до 128M с учётом админ-раздела)

; --- ⚠️ Удалить полностью, не оставлять даже =0 ---
; mbstring.func_overload   (не поддерживается с Main 20.100.0)
```

Константы прав на новые файлы/папки в `/bitrix/php_interface/dbconn.php`:

```php
define("BX_FILE_PERMISSIONS", 0644);
define("BX_DIR_PERMISSIONS", 0755);
```

Рекурсивное применение прав через SSH (FTP и SSH — под одним пользователем):

```bash
find . -type f -exec chmod 0644 {} \;
find . -type d -exec chmod 0755 {} \;
```

Опциональное ужесточение прав после установки/restore: секретонесущие конфиги стоит сделать недоступными для чтения посторонним — 0640 (группа = пользователь веб-сервера) либо 0600 (когда владелец файла == пользователь веб-сервера):

```bash
chmod 0640 bitrix/.settings.php bitrix/.settings_extra.php bitrix/php_interface/dbconn.php
```

## Проверка

1. **Скрипт `bitrix_server_test.php`** — загрузить на хостинг, открыть в браузере, нажать «Начать тестирование», затем **сразу удалить файл из webroot** (см. ⚠️ Риски — он раскрывает версии и пути сервера). Скрипт обновляется под актуальные требования; подходит для предварительной оценки площадки до установки продукта.
2. **Настройки → Инструменты → Проверка системы** (админка) — всесторонняя сверка с минимальными и рекомендуемыми требованиями: объём оперативной памяти, объём БД, объём диска под файлы проекта.
3. **Настройки → Производительность → Панель производительности** — параметр «Конфигурация» оценивает скорость отдачи страниц и корректность настроек PHP.
4. Точечно: `php -m` (наличие расширений), `php -i | grep -E 'memory_limit|max_input_vars|default_charset'` (значения), и убедиться, что `mbstring.func_overload` отсутствует в выводе.

## ⚠️ Риски

- **`mbstring.func_overload`** — наличие этой настройки приводит к ошибке проверки системы и некорректной работе ядра. Удалить полностью.
- **Конвертация windows-1251 → UTF-8** — перед миграцией существующего сайта на UTF-8 обязательно сделать резервную копию сайта и базы данных; конвертацию выполнять Мастером перехода на UTF-8 либо вручную. Новые установки в windows-1251 невозможны (поддержка прекращена).
- **Кодировка БД utf8mb3 вместо utf8mb4** — приведёт к потере символов вне BMP (эмодзи, часть нацалфавитов). Создавать БД сразу в utf8mb4.
- **Права 777 (world-writable)** — открытый доступ на запись для всех создаёт угрозу безопасности; вендор рекомендует значения по умолчанию 0644/0755. При этом 0644/0755 доступны на чтение всем: на неизолированном shared-хостинге соседний uid может прочитать конфиг с реквизитами БД. Защита — изоляция площадки (`open_basedir`, отдельный vhost, per-user FPM) либо ужесточение прав на секретонесущие файлы (см. ниже).
- **`display_errors = On` в продакшне** — раскрывает пути и детали окружения; на боевом сервере выводить ошибки только в лог.
- **Отсутствие каталога OPcache** — частая причина проблем с производительностью; каталог хранения скомпилированного кода должен быть настроен.
- ⚠️ **`bitrix_server_test.php` и любые phpinfo-пробы удалять из webroot сразу после проверки.** Загруженный для предварительной оценки площадки скрипт раскрывает версии PHP/расширений, серверные пути, директивы `php.ini` и состояние окружения — готовую разведданную для прицельной атаки. После прогона «Начать тестирование» немедленно удалите `bitrix_server_test.php` (и любой временный `phpinfo()`-файл) из публичной зоны; не оставляйте его доступным по HTTP.

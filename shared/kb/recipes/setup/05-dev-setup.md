# 05. Настройка dev-среды

## Цель

Подготовить среду разработки по правилам вендора «1С-Битрикс»: вести
разработку на отдельной копии в режиме «Установка для разработки», а не на
боевом сайте; на время работы отключить кэширование и автопроверку
обновлений; вынести локальные переопределения в `.settings_extra.php`;
заблокировать исходящую почту и индексацию поисковиками на dev. Цель — чтобы
изменения тестировались изолированно и не затрагивали продакшен и его
посетителей.

## Когда применять

- Разворачиваете рабочую копию для доработки сайта (своя БД, отладка).
- Получили дамп/копию прода и нужно превратить её в безопасную dev-площадку.
- Готовите площадку под командную разработку с git (см. `../../conventions.md`).

Перед этим шагом проект уже должен быть проинтроспектирован
(`../01-introspect-project.md`): известны редакция, версия ядра, путь к
`/local`, текущие настройки кэша.

## Контейнерный dev-стенд (env-docker)

Для небоевого использования (разработка, тестирование, код-ревью, проверка
безопасности) вендор рекомендует официальную контейнерную сборку
`bitrix-tools/env-docker` [[src]](https://github.com/bitrix-tools/env-docker).
Включает веб-сервер (nginx + apache/php-fpm), PHP 8.2–8.4, БД (MySQL/MariaDB или
PostgreSQL), кэш (Redis/Memcached), Sphinx, cron, push/pull.

Запуск:

```bash
docker compose up -d
```

Веб-интерфейс по умолчанию: HTTP `:8588`, HTTPS `:8589`.

Эта среда предназначена для разработки, тестирования, код-ревью и проверки
безопасности; прод-конфигурация настраивается отдельно (с усиленными
настройками). Рискованные и необратимые операции — обновления ядра/модулей,
восстановление из резервной копии, миграции данных — рекомендуется прогонять на
локальной копии в такой среде, а не на боевом сайте.

> **env-docker — это dev-стенд, а не готовый сайт.** Базовую установку на нём
> ведут через Режим 2 (мастер в браузере, [03-installation.md](03-installation.md) /
> [06-install-wizard.md](06-install-wizard.md)) или Режим 3 (restore из чистого
> base-бэкапа, [../deploy/03-backup-restore.md](../deploy/03-backup-restore.md)),
> после чего прогоняют гейт завершённости из
> [03-installation.md](03-installation.md). До зелёного гейта контент не строят.

## Шаги

1. **Снять рабочую копию, а не править прод.** Развернуть отдельную копию
   сайта (файлы + БД) на dev-сервере или локально. Разработка ведётся только
   на тестовой копии с включённым режимом «Установка для разработки»; править
   боевой сайт напрямую не следует. Перед любыми изменениями иметь свежий
   бэкап.

2. **Обезличить прод-дамп ПЕРЕД использованием (обязательно).** Если источник
   копии — дамп/копия прода, она несёт боевые ПДн и живые секреты. До того как
   копию открыли в браузере и начали ей пользоваться, выполнить санацию БД.
   Точный набор таблиц и колонок зависит от установленных модулей — приведённые
   ниже примеры адаптируйте под свою редакцию:
   - **Пароли/доступы.** Сбросить или рандомизировать хеши паролей и контрольные
     слова администраторов и пользователей (поля `PASSWORD`, `CHECKWORD` в
     `b_user`); либо урезать `b_user` до одного засеянного тестового админа.

     ```sql
     -- обнулить хеши и контрольные слова всем пользователям
     UPDATE b_user SET PASSWORD = MD5(RAND()), CHECKWORD = '';
     ```
   - **ПДн.** Обнулить/заменить персональные данные (email, телефоны, имена,
     адреса) в `b_user` и в таблицах заказов/CRM. Состав таблиц зависит от
     модулей: например `b_sale_order` / `b_sale_order_props_value` (модуль
     `sale`), `b_crm_*` (модуль `crm`).

     ```sql
     UPDATE b_user SET EMAIL = CONCAT('user', ID, '@example.test'),
       PERSONAL_PHONE = '', PERSONAL_MOBILE = '', NAME = '', LAST_NAME = '';
     -- значения свойств заказа (телефон, адрес, ФИО) — состав зависит от
     -- настроенных свойств; при сомнении проще очистить таблицу целиком:
     TRUNCATE TABLE b_sale_order_props_value;
     ```
   - **Секреты.** Ротировать или очистить живые секреты, не нужные dev-стенду, в
     `b_option` и `.settings.php`: ключи платёжных систем, SMS/email-API, пароли
     обмена с 1С, SMTP, OAuth.

     ```sql
     -- пример: очистить значения параметров платёжных систем
     DELETE FROM b_option WHERE MODULE_ID = 'sale' AND NAME LIKE '%SECRET%';
     ```

   Перепроверьте список под свои модули: имена таблиц/колонок различаются между
   редакциями и версиями. Только после обезличивания копию можно открывать и
   использовать как dev-площадку.

3. **Развязать копии по обновлениям.** Обновлять только одну копию из пары
   «локальная / публичная». Не переносить установку между серверами без учёта
   механизма привязки лицензии — иначе возможна ошибка синхронизации
   `ERROR_WRONG_CODE`.

4. **Отключить автопроверку обновлений на dev.** В настройках Главного модуля
   снять «Автоматически проверять наличие обновлений», чтобы dev-копия не
   обращалась к серверу обновлений и не пыталась тянуть апдейты (это задача
   прода). На боевом сайте опция остаётся включённой.

5. **Отключить кэширование на время разработки.** В разделе «Настройки →
   Настройки продукта → Автокэширование» выключить управляемый (managed),
   автоматический (auto) и HTML-кэш (composite). Это нужно, чтобы видеть
   результат правок сразу; в частности, `result_modifier.php` срабатывает
   только при отключённом кэшировании, а composite перехватывает запрос до
   ядра и может отдать страницу из статического HTML мимо вашей логики.

6. **Вынести локальные переопределения в `.settings_extra.php`.** Параметры,
   которые отличают dev от прода (своя БД, включённый debug), держать в
   `.settings_extra.php` — он переопределяет значения из `.settings.php` и не
   уезжает в репозиторий. С версии ядра 24.100.0 файл поддерживается и в
   `/local` (приоритет у `/local` через `Loader::getLocal`).

7. **Удержать исходящую почту (email containment).** Чтобы письма с dev-копии
   (заказы, уведомления, рассылки) не уходили реальным адресатам, задать
   константу `ONLY_EMAIL` в `dbconn.php` — весь исходящий e-mail
   перенаправляется на указанный адрес разработчика. Это перехват исходящих
   писем, а не обезличивание данных: адреса в БД остаются прежними, поэтому
   `ONLY_EMAIL` не заменяет санацию из шага 2.

8. **Закрыть dev-стенд от публичного доступа.** Стенд, на котором лежат любые
   производные от прода данные, ОБЯЗАН стоять за реальным контролем доступа:
   HTTP basic-auth, allow-list по IP на веб-сервере или приватная/VPN-сеть.
   `robots.txt` — это только запрет индексации, он не является средством
   конфиденциальности: на публично резолвящемся хосте кто угодно откроет копию
   прода напрямую по URL.

9. **Запретить индексацию dev поисковиками.** Положить в document root
   dev-копии `robots.txt`, запрещающий обход всего сайта, чтобы тестовая
   площадка не попала в поисковую выдачу. `robots.txt` носит рекомендательный
   характер для поисковых роботов и не ограничивает доступ — конфиденциальность
   обеспечивает только шаг 8.

10. **Проверить лицензию.** Убедиться, что на dev-копии не используется боевой
    лицензионный ключ одновременно с продом (см. ## ⚠️ Риски).

## Рабочий сниппет/конфиг

`php_interface/dbconn.php` (в document root копии) — перенаправление всей
почты и включение режима отладки на dev:

```php
<?php
// Весь исходящий e-mail на dev уходит только сюда (не реальным клиентам).
define('ONLY_EMAIL', 'dev@example.com');

// Режим отладки на время разработки — только на изолированной dev-копии.
$DBDebug = true; // лог SQL-запросов
```

> ⚠️ **Не задавать `BX_UTF` отдельно.** `BX_UTF` — это флаг кодировки UTF-8,
> часть согласованного UTF-стека (кодировка БД, файлов, `after_connect`/
> `SET NAMES`), а не переключатель отладки. Включать его допустимо только в
> рамках полной конвертации проекта в UTF-8; одиночное `define('BX_UTF', true)`
> на копии с другой кодировкой ломает вывод текста. Отладка на dev — это
> `$DBDebug = true` плюс блок `exception_handling.debug` ниже.

`.settings_extra.php` (в `/local/` или `/bitrix/`) — локальные
переопределения поверх `.settings.php`. Файл держать вне репозитория:

```php
<?php
return [
    'exception_handling' => [
        'value' => [
            'debug' => true,          // подробные ошибки на dev
            'handled_errors_types' => E_ALL & ~E_NOTICE & ~E_STRICT & ~E_DEPRECATED,
            'exception_errors_types' => E_ALL & ~E_NOTICE & ~E_STRICT & ~E_DEPRECATED,
        ],
        'readonly' => false,
    ],
    // При необходимости — своя БД для dev-копии:
    // 'connections' => ['value' => ['default' => [
    //     'host' => 'localhost', 'database' => 'dev_db',
    //     'login' => 'dev', 'password' => '...',
    // ]], 'readonly' => true],
];
```

> 🔒 **`debug`/`$DBDebug` недопустимы на публичном/боевом сайте.** При активном
> `exception_handling.debug = true` или `$DBDebug = true` анонимный посетитель
> увидит полные трейсы с абсолютными путями и сырыми SQL-запросами. Если
> отладочный конфиг или `dbconn.php`-переопределения уедут на прод через
> rsync/деплой, это утечка структуры проекта. Держать их вне репозитория и
> вне выкатки; перед отдачей убедиться, что на проде `debug = false`. Надёжнее
> включать отладку не статическим `define`, а по проверке окружения (хост/IP/
> переменная среды), чтобы случайная синхронизация не активировала её на проде.

`robots.txt` (в document root dev-копии) — полный запрет индексации:

```
User-agent: *
Disallow: /
```

## Проверка

- **Разработка на копии.** В «Настройки → Установка для разработки» включён
  соответствующий режим; правки делаются на копии, прод не трогается.
- **ПДн и секреты обезличены.** Если источник — прод-дамп, выполнена санация
  (шаг 2): хеши паролей сброшены, email/телефоны/имена в `b_user` и заказах/CRM
  заменены, живые ключи платёжек/SMS/обмена с 1С/SMTP в `b_option` и
  `.settings.php` ротированы или удалены.
- **Стенд закрыт от публики.** Доступ к dev-площадке ограничен basic-auth, IP
  allow-list или приватной сетью; не полагаемся на один `robots.txt`.
- **Автообновления отключены.** В Главном модуле снят флаг автоматической
  проверки обновлений; dev не инициирует обращения к серверу обновлений.
- **Кэш отключён.** В «Автокэширование» все три типа (managed/auto/HTML)
  выключены; правка в `result_modifier.php` или шаблоне видна сразу после
  перезагрузки страницы без сброса кэша.
- **Почта не уходит наружу.** Тестовое письмо (например, оформление заказа)
  приходит только на адрес из `ONLY_EMAIL`, реальные адресаты ничего не
  получают.
- **Debug активен только на dev.** На dev при искусственной ошибке выводится
  подробная информация (а не общая страница), что подтверждает применение
  `.settings_extra.php`. Перед отдачей на прод убедиться, что
  `exception_handling.debug = false`, `$DBDebug` не выставлен и dev-переопределения
  `dbconn.php`/`.settings_extra.php` удалены из выкатки.
- **Индексация закрыта.** `https://<dev-host>/robots.txt` отдаёт
  `Disallow: /`.

## ⚠️ Риски

- ⚠️ **Боевой лицензионный ключ на нескольких копиях.** Не держать активный
  лицензионный ключ прода одновременно на dev-копии. Перенос/дублирование
  установки без учёта привязки приводит к рассинхронизации
  (`ERROR_WRONG_CODE`) и проблемам с обновлениями. Для dev использовать
  отдельную копию по правилам вендора, обновляя только одну копию из пары.
- ⚠️ **Письма реальным клиентам с тестовой копии.** Без `ONLY_EMAIL`
  dev-копия с боевыми данными разошлёт уведомления, заказы и рассылки
  настоящим адресатам. Задавать `ONLY_EMAIL` сразу при развёртывании копии.
- ⚠️ **Боевые ПДн и секреты в dev-копии.** Прод-дамп несёт реальные пароли,
  персональные данные клиентов и живые ключи (платёжки, обмен с 1С, SMTP). Без
  обезличивания (шаг 2) их получают все, у кого есть доступ к стенду. Санацию
  выполнять до первого использования копии.
- ⚠️ **Открытый dev-стенд с прод-данными.** На публично резолвящемся хосте
  `robots.txt` не закрывает доступ — клон прода читается по прямому URL.
  Стенд с производными от прода данными держать за basic-auth, IP allow-list
  или в приватной/VPN-сети.
- ⚠️ **Отладка на проде.** `exception_handling.debug`/`$DBDebug`, утёкшие на
  боевой сайт через деплой, раскрывают анонимным посетителям трейсы с путями и
  SQL. Не включать отладку статическим `define`; перед прод-выкаткой проверять
  `debug = false`.
- ⚠️ **Правка на проде вместо копии.** Изменения на боевом сайте без бэкапа
  и тестовой копии ведут к простою и потере данных. При вынужденной правке на
  живом сайте заранее обеспечить доступ по FTP/SSH: административная часть
  может стать недоступной.
- **Кэш забыт включённым/выключенным.** Отключённый кэш скрывает реальную
  производительность; включённый — прячет свежие правки (особенно composite,
  который перехватывает запрос до ядра). Перед сдачей вернуть кэш к
  прод-настройкам и проверить поведение.
- **Индексация dev.** Без `robots.txt` тестовая копия попадёт в поисковую
  выдачу и может конкурировать с продом. Закрывать индексацию при
  развёртывании.

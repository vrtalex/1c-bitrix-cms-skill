# Рецепт 03 — installation: установка и восстановление «1С-Битрикс: Управление сайтом» (ядро 26.x)

> Подсистема: 1c-bitrix-cms-setup. Ядро БУС 26.x. Две поддерживаемые версии API (D7 и legacy) — обе официальны.
> Раздел «Установка без браузера» подтверждён на живом ядре 26.x; шаги мастера реконструированы из его HTTP-форм.

## Цель

Развернуть новый сайт из дистрибутива (создать `dbconn.php`/`.settings.php`, установить модули, получить стартовый сайт `s1`) и/или перенести готовый сайт через `restore.php`. Дать воспроизводимую headless-установку для CI/Docker — без интерактивного браузера.

## Когда применять

- Чистая установка на новое окружение (локально, staging, CI, Docker-образ).
- Автоматизация развёртывания: нужен скриптуемый прогон мастера в pipeline.
- Перенос/восстановление существующего сайта на другой хостинг или из резервной копии.
- Подготовка эталонного окружения под последующие рецепты (контент, шаблоны, настройки).

## Шаги

### A. Чистая установка из дистрибутива

1. **Понять дистрибутив.** Это преинсталляционный комплект: корневой `index.php` — лаунчер мастера (НЕ страница сайта), он стартует установку из модуля `main`. Редакция фиксируется в корневом `install.config` (имя продукта, `skipInstallModules`, `LANGUAGE_ID`, `INSTALL_CHARSET`). Публичная часть материализуется мастером из `bitrix/modules/<m>/install/` в document root и `/bitrix`.
2. **Подготовить окружение.** PHP с включённым `short_open_tag` (лаунчер это проверяет), доступная СУБД (MySQL), пустая БД и пользователь с правами на неё, корректные права на запись (папки `0755`, файлы `0644`; `/upload`, `bitrix/cache` могут требовать запись веб-сервером).
3. **Запустить мастер.** Открыть `/` (или `/index.php`) → пройти шаги: лицензионное соглашение → лицензионный ключ (демо или коммерческий) → проверка требований → параметры БД → установка модулей. Машина состояний: каждый шаг знает свой `CurrentStepID` и `NextStepID`.
4. **Результат мастера.** Создаются `bitrix/php_interface/dbconn.php` (параметры подключения) и `bitrix/.settings.php` (D7-конфиг), регистрируются и устанавливаются модули, создаётся стартовый сайт `LID="s1"`, `DIR="/"`.
5. **Постустановка.** Проверить вход в админку, запустить «Проверку системы», задать домен сайта (нужен для Инспектора сайтов), настроить регулярный бэкап.

### B. Установка без браузера (headless, для CI/Docker)

Мастер — пошаговая машина: POST на `/` (multipart/form-data) с полями `CurrentStepID`, `NextStepID`, `StepNext`. Последовательность подтверждена на живом ядре 26.x:

1. **welcome → agreement.** POST с `__wiz_agree_license=Y`.
2. **check_license_key.** Демо-режим: `__wiz_lic_key_variant=Y` + имя/фамилия/email. Для коммерческой лицензии передать ключ.
3. **requirements.** Проверка окружения (можно проскочить, если требования выполнены).
4. **create_database.** Существующие пользователь и БД: `__wiz_host`, `__wiz_create_user=N`, `__wiz_user`, `__wiz_password`, `__wiz_create_database=N`, `__wiz_database`. Шаг проверяет коннект и пишет `dbconn.php` + `.settings.php`.
5. **create_modules (AJAX-цикл).** POST с `__wiz_nextStep`/`__wiz_nextStepStage`; ответ содержит блок `[response]` с JS вида `ajaxForm.Post({nextStep, nextStepStage})`. Извлекать оттуда следующие `nextStep`/`nextStepStage` и повторять POST, пока модули не встанут. По завершении создаётся сайт `s1`.

Куки сессии (`PHPSESSID`/`BITRIX_SM_*`) сохранять между запросами — мастер ведёт состояние в сессии.

### C. Перенос/восстановление готового сайта (restore.php)

1. ⚠️ Сделать резервную копию текущего состояния целевого окружения ДО любых операций.
2. Скачать актуальный `restore.php` **только по HTTPS** с легитимного адреса вендора 1С-Битрикс; никогда не тянуть дистрибутив/`restore.php` по cleartext `http://` или с зеркал. Положить в **корень** нового сайта. Владельцем файла должен быть системный пользователь сайта (`bitrix` в стандартном окружении).
3. ⚠️ **Проверить целостность ДО запуска.** Сверить контрольную сумму/подпись скачанного `restore.php` и архива бэкапа с эталоном, опубликованным вендором, перед тем как открывать скрипт (`sha256sum restore.php` → сравнить с опубликованным значением; для бэкапа — сверить хеш с тем, что зафиксирован при создании копии). Известный пробел: вендор не всегда публикует официальную контрольную сумму `restore.php` — при её отсутствии минимизировать риск загрузкой строго по HTTPS с легитимного домена вендора и снятием/фиксацией собственного хеша сразу после скачивания, чтобы детектировать подмену.
4. Загрузить туда же все части архива (многотомный — грузить ВСЕ тома).
5. ⚠️ **Закрыть `restore.php` на время миграции.** Открывать мастер только по `https://новый-домен/restore.php` (НЕ cleartext `http://` — параметры БД и логин администратора иначе уходят открытым текстом). На окно миграции ограничить доступ к скрипту по IP (allow-list своего адреса в конфиге веб-сервера/`.htaccess`) и/или basic-auth; после переноса — удалить.
6. Открыть `https://новый-домен/restore.php` → выбрать архив-источник → дождаться распаковки.
7. Ввести параметры новой БД (хост, логин, пароль, имя). При переносе на другой хостинг отметить «Создать базу данных, если не существует».
8. Выбрать режим: полное восстановление (файлы + БД) или только файлы.
9. ⚠️ Финальной кнопкой мастера удалить локальную копию архива и сам `restore.php`; затем снять временное IP-/basic-auth-ограничение.
10. Очистить кеш (`bitrix/cache/`, `bitrix/managed_cache/` или через админку), проверить доступность сайта и права.

## Рабочий сниппет

Headless-прогон мастера через `curl` (раздел B). Скелет для CI/Docker; подставить реальные параметры окружения.

```sh
#!/bin/sh
# headless-install.sh — установка БУС без браузера. Запускать против поднятого PHP+СУБД.
set -eu

BASE="http://localhost"          # адрес поднятого окружения
JAR="$(mktemp)"                  # cookie jar: сессия мастера

DB_HOST="localhost"
DB_USER="bitrix"
DB_PASS="<СГЕНЕРИРУЙ_СИЛЬНЫЙ_ПАРОЛЬ>"   # плейсхолдер: подставить сильный пароль (например `openssl rand -base64 24`); НЕ коммитить
DB_NAME="bitrix"

post() { curl -fsS -c "$JAR" -b "$JAR" "$BASE/" "$@"; }

# 1. agreement
post -F "CurrentStepID=welcome" -F "NextStepID=agreement" -F "StepNext=Y" \
     -F "__wiz_agree_license=Y" >/dev/null

# 2. лицензионный ключ (демо)
post -F "CurrentStepID=agreement" -F "NextStepID=check_license_key" -F "StepNext=Y" \
     -F "__wiz_lic_key_variant=Y" \
     -F "__wiz_first_name=Admin" -F "__wiz_last_name=Admin" \
     -F "__wiz_email=admin@example.com" >/dev/null

# 3. проверка требований
post -F "CurrentStepID=check_license_key" -F "NextStepID=requirements" -F "StepNext=Y" >/dev/null

# 4. параметры БД (существующие пользователь и база) → пишет dbconn.php/.settings.php
post -F "CurrentStepID=requirements" -F "NextStepID=create_database" -F "StepNext=Y" \
     -F "__wiz_host=$DB_HOST" \
     -F "__wiz_create_user=N" -F "__wiz_user=$DB_USER" -F "__wiz_password=$DB_PASS" \
     -F "__wiz_create_database=N" -F "__wiz_database=$DB_NAME" >/dev/null

# 5. установка модулей (AJAX-цикл): из ответа берём nextStep/nextStepStage и повторяем
STEP="create_modules"; STAGE=""
for i in $(seq 1 200); do
  RESP="$(post -F "CurrentStepID=create_modules" -F "StepNext=Y" \
               -F "__wiz_nextStep=$STEP" -F "__wiz_nextStepStage=$STAGE")"
  # парсинг ajaxForm.Post({nextStep:'...', nextStepStage:'...'}) из [response]
  STEP="$(printf '%s' "$RESP"  | sed -n "s/.*nextStep:'\([^']*\)'.*/\1/p"      | head -1)"
  STAGE="$(printf '%s' "$RESP" | sed -n "s/.*nextStepStage:'\([^']*\)'.*/\1/p" | head -1)"
  [ -z "$STEP" ] && { echo "модули установлены, сайт s1 создан"; break; }
done

rm -f "$JAR"
```

Имена полей и формат `[response]` могут отличаться по сборкам ядра — перед прогоном на конкретной версии один раз снять реальные формы мастера (GET шага) и сверить `name` инпутов.

## Проверка

**Чистая установка:**

```sh
# Конфиги созданы мастером
ls -1 <DOCROOT>/bitrix/php_interface/dbconn.php <DOCROOT>/bitrix/.settings.php

# Стартовый сайт s1 в БД
mysql -u"$DB_USER" -p"$DB_PASS" "$DB_NAME" -e "SELECT LID, DEF, ACTIVE, DIR FROM b_lang;"

# Установленные модули
mysql -u"$DB_USER" -p"$DB_PASS" "$DB_NAME" -e "SELECT ID FROM b_module ORDER BY ID;"
```

- Открыть `/bitrix/admin/` → вход под созданным администратором.
- Админка → «Настройки → Инструменты → Проверка системы» → красных пунктов нет.
- Главная страница сайта отдаётся (не лаунчер мастера).

**Восстановление (restore.php):**

- Сайт открывается на новом домене; `restore.php` и архив удалены из корня.
- При проблемах входа из-за LDAP/AD или двухфакторки — временно отключить эти системы и сбросить пароль администратора.

## ⚠️ Риски

- ⚠️ **Бэкап перед операциями.** Перед установкой поверх существующего окружения и перед любым восстановлением — резервная копия. Восстановление перезаписывает БД и файлы.
- ⚠️ **Удалить служебные скрипты после.** `restore.php` и загруженные архивы удалить финальной кнопкой мастера. Оставленный в корне `restore.php` — путь к перезаписи сайта. На время миграции держать скрипт за HTTPS + IP-allow-list/basic-auth.
- ⚠️ **Целостность дистрибутива/`restore.php`.** Скачивать только по HTTPS с легитимного домена вендора и сверять контрольную сумму/подпись перед запуском. Официальной публикуемой контрольной суммы `restore.php` может не быть (известный пробел) — тогда снять собственный хеш сразу после загрузки и не запускать файл, полученный по cleartext `http://` или из непроверенного источника.
- ⚠️ **Пароль шифрования бэкапа.** Для облачных копий шифрование обязательно; забытый пароль делает восстановление невозможным — вендор не восстанавливает пароль.
- ⚠️ **Корневой `index.php` дистрибутива — инсталлятор, а не страница.** До завершения мастера публичных страниц нет; не путать с готовым сайтом.
- Многотомный архив: грузить ВСЕ части; при HTTP 413 — партиями по 9–10 файлов.
- Кодировка: несовпадение кодировки архива и БД даёт `ERROR 1062` (дубли ключей) — привести `character_set_server` к кодировке архива.
- Права/владелец: после переноса проверить `0755`/`0644`, владельца файлов, при необходимости `BX_FILE_PERMISSIONS`/`BX_DIR_PERMISSIONS` в `dbconn.php`.
- Кеш после переноса: очистить `bitrix/cache/` и `bitrix/managed_cache/`, иначе возможны рассинхронизации.
- Headless-мастер ведёт состояние в сессии: терять куки между POST нельзя, иначе шаг сбросится на начало.

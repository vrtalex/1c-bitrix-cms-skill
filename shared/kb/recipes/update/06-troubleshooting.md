# Диагностика и устранение проблем обновления

## Цель

Сопоставить наблюдаемый симптом сбоя обновления «1С-Битрикс: Управление сайтом» (ядро 26.x) с задокументированной причиной и фиксом. Каталог построен по схеме симптом → причина → фикс с точными текстами сообщений системы обновлений, чтобы не угадывать, а адресно восстанавливать прерванный или не стартующий апдейт.

## Когда применять

- Система обновлений выдаёт ошибку соединения, лицензии, распаковки или фатал PHP при установке/после установки.
- Апдейт «завис» на проценте, отдаёт 504, оборвался на тяжёлой операции БД.
- После установки появился `Fatal error`/`TypeError`, которого до обновления не было.
- Нужно форсировать перезаливку оригинальных файлов ядра после частично применённого апдейта.

## Шаги

1. **Снимите точный текст ошибки.** Код в квадратных скобках (`[110]`, `[SITE_LICENSE_VIOLATION]`, `[UUGZA073]`) и сигнатуру фатала PHP. По ним ищите запись в каталоге ниже.
2. **Перед любым ремонтом — свежий бэкап** (`Настройки → Инструменты → Резервное копирование`), если апдейт уже трогал БД. Восстановление доступно через `restore.php` даже без админ-доступа.
3. **Найдите запись в каталоге** (раздел «Рабочий сниппет/конфиг» — это сам каталог симптом → причина → фикс) и примените фикс.
4. **После фикса — повторите апдейт.** Система обновлений идемпотентна: она подхватывает оставшиеся непоставленные обновления. Прерванный апдейт восстанавливается повторным запуском, а не ручной чисткой.
5. **Если файлы ядра повреждены** — форсируйте перезаливку оригиналов через support-режим (см. запись «Перезаливка файлов» ниже), затем доберите авто-исправления структуры БД через «Проверку системы».
6. **Сбросьте кэш** (`Настройки → Настройки продукта → Автокеширование`) и пересоберите кэш меню/контента.

## Рабочий сниппет/конфиг

Каталог симптом → причина → фикс. Тексты сообщений и фиксы приведены дословно по источникам.

### Соединение и сеть

**Симптом:** `Ошибка соединения с сервером обновлений: [110] Connection timed out`.
**Причина:** заблокирован `fsockopen()` или исходящий порт 80 файрволом, либо нехватка RAM (типично на VPS 256 МБ OpenVZ).
**Фикс:** открыть исходящие соединения и сокеты на хосте; убедиться, что доступен сервер обновлений; при нехватке памяти поднять лимит PHP. Сервер обновлений по умолчанию — `www.1c-bitrix.ru`.
Источник (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=2697

**Симптом:** `php_network_getaddresses: getaddrinfo ... failed` при обращении к серверу обновлений.
**Причина:** не резолвится DNS / заблокирован исходящий запрос.
**Фикс (по практике):** проверить `allow_url_fopen=On`, настройки прокси в Главном модуле, доступность DNS (например резолв через `8.8.8.8`), отсутствие блокировки исходящих на хосте.
Источник (по практике, сообщество): https://dev.1c-bitrix.ru/community/forums/forum6/topic162372/

**Симптом:** `LICENSE_NOT_FOUND` при проверке обновлений.
**Причина (по практике):** система всё ещё смотрит на отключённый старый сервер обновлений (`www.bitrixsoft.com`), либо заблокированы исходящие соединения.
**Фикс (по практике):** `Настройки → Настройки продукта → Настройки модулей → Главный модуль → вкладка «Система обновлений»` — указать сервер `www.1c-bitrix.ru`; сверить `license_key.php` с ключом из личного кабинета; очистить `/bitrix/cache` и `/bitrix/managed_cache`.
Источник (сообщество): https://dev.1c-bitrix.ru/community/forums/forum6/topic162372/

### Лицензия

**Симптом:** `[SITE_LICENSE_VIOLATION] Превышено количество лицензированных сайтов`.
**Причина:** нет ни одного зарегистрированного/активного сайта ЛИБО число сайтов превышает лимит лицензии.
**Фикс:** `Настройки → Настройки продукта → Сайты → Список сайтов` — привести количество активных сайтов в соответствие с лицензией.
Источник (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=2697

**Симптом:** `[ERROR_WRONG_CODE] Проверка сайта не пройдена`.
**Причина:** текущее состояние установки не совпадает с зафиксированным при последнем обновлении. Система обновлений привязывается к состоянию инсталляции, чтобы один ключ не использовался для обновления нескольких живых копий.
**Фикс:** запускать апдейтер только на ОДНОЙ копии ключа (прод ИЛИ локальная копия, не обе одновременно). Для легальной dev-копии включить «Установка для разработки» (Главный модуль → «Система обновлений», доступно с 16.5.7). При миграции сервера — скопировать файлы и БД, переключить DNS, затем удалить старую установку.
Источник (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=2697

### OpCache и классы апдейтера

**Симптом:** `Class 'CUpdateExpertMode' not found` (или иной класс апдейтера) при запуске обновления.
**Причина:** `opcache.validate_timestamps=0` — OpCache отдаёт ранее закэшированные классы апдейтера после того, как апдейтер перезаписал их файлы.
**Фикс:** выставить `opcache.validate_timestamps=On` в `opcache.ini` на время обновления.
Источник (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=2697

> Примечание: ошибка вида `Class '...' not found` штатно возникает и внутри `updater.php`, если он ссылается на новый API из того же пакета — `updater.php` выполняется ДО копирования файлов в ядро. Это особенность механики пакета, а не сбой окружения (см. update/01).

### Распаковка пакета

**Симптом:** `[UUGZA06]` (неверный формат файла), `[UUGZA073]` (сбой на файле) или `[CL02] Ошибка распаковки пакета обновления`.
**Причина:** сбой штатного распаковщика на конкретном пакете/окружении.
**Фикс (по практике):** в `Настройки → Настройки продукта → Настройки модулей → Главный модуль → вкладка «Система обновлений»` включить опцию «Не использовать архиватор» и повторить установку. Перекачивать тот же пакет повторно при этой ошибке не требуется — меняется способ распаковки, а не сам пакет.
Источник (по практике, сообщество): https://rugeek.ru/articles/cl02-oshibka-raspakovki-paketa-problemy-s-obnovleniem-bitriks/ ; обсуждение на https://dev.1c-bitrix.ru/support/forum/forum6/topic59389/

### Фаталы PHP после обновления ядра

**Симптом:** `Fatal error: Declaration of Bitrix\Main\Diag\Logger::log($level, Stringable|string $message, array $context = []): void must be compatible with Psr\Log\LoggerInterface::log(...)` (репортилось после `main` 23.600.0+ и 23.650.100).
**Причина:** ядро подняло `psr/log` с типизированными аргументами, но в проекте грузится ВТОРОЙ, старый нетипизированный `psr/log` — из стороннего модуля, `php_interface` или `/local`. Системный PSR лежит в `/bitrix/modules/main/vendor/psr/`. Это транзитивный конфликт версий Composer стороннего пакета, а не дефект ядра.
**Фикс:** в `/local/composer.json` запинить `psr/log:^3.0` (а также `psr/container:^2.0`, `psr/http-message:^2.0`, `psr/http-client:^1.0`), затем в `/local` выполнить `composer update` и `composer install --no-dev`, убрав дубликат. Альтернатива — секция `"replace": {"psr/log":"*","psr/http-message":"*"}` в `composer.json`, чтобы уже идущие в `main` пакеты не грузились дважды.
Источник (vendor-community): https://dev.1c-bitrix.ru/community/webdev/user/568029/blog/44228/

**Симптом:** `[TypeError] Cannot assign null to property Bitrix\Main\Data\CacheEngine::$engine of type Redis|Memcache|Memcached` в `/bitrix/modules/main/lib/data/cacheengine.php` (после обновления `main` от 14.03.2025 на BitrixVM окружении ≤ 7.5.2).
**Причина (по практике):** ядро добавило `RedisCluster` как cache-engine, но union-тип свойства `$engine` на старом окружении не включал `\RedisCluster`.
**Фикс (по практике):** временный хотфикс — в объявлении свойства (строка 13 файла) дописать `\RedisCluster` в union: `protected static \RedisCluster|\Redis|\Memcache|null|\Memcached $engine = null;`. Постоянный фикс — обновить `main` (по источнику — до 25.100.100+; версию фикса проверить по changelog), затем откатить ручную правку.
Источник (по практике, один партнёрский источник): https://i-pusk.ru/blog/help-support/kriticheskaya-oshibka-bitriks-padenie-portala-posle-obnovleniya-yadra-ot-14-03-25-reshenie/

> Урок обоих кейсов: «двухшаговая» модель восстановления после редкого проблемного релиза — временный точечный патч, затем обновление до целевой версии и откат патча. Composer-зависимости обновлять одновременно с ядром и сначала проверять на стейдже.

### Долгий апдейт / 504

**Симптом:** установка «зависла» на проценте, отдаёт 502/504, в логах MySQL обрыв коннекта.
**Причина:** апдейтер выполняет шаги синхронно как AJAX-запросы. На тяжёлой операции БД (часто `CREATE [FULLTEXT] INDEX` на большой таблице, например `CREATE FULLTEXT INDEX IXF_VI_STATS_1 ON b_voximplant_statistic_index(CONTENT)`) превышается read-timeout reverse-proxy. Граница 504 — это таймаут NGINX/FastCGI (по умолчанию ~300 с), а не PHP.
**Фикс:** не прерывать — дать DDL завершиться (по практике видно через `SHOW PROCESSLIST` в MySQL), затем повторить апдейтер: он продолжит с оставшихся обновлений. Перед тяжёлым обновлением ядра поднять таймауты nginx (`proxy_connect_timeout`/`proxy_send_timeout`/`proxy_read_timeout`, и fastcgi-аналоги) и `max_execution_time` PHP. Настройка «продолжительность шага» в мастере управляет ТОЛЬКО чанкингом скачивания, а не шагом БД.
Источники (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=2693 ; (по практике): https://dev.1c-bitrix.ru/community/webdev/user/213927/blog/39820/

### Перезаливка файлов (support-режим вендора)

**Назначение:** инструмент вендора для форсированной перезаписи ОРИГИНАЛЬНЫХ файлов модулей и ядра после повреждения файлов или частично применённого апдейта.
**Как:** открыть `your_site/bitrix/admin/update_system.php?BX_SUPPORT_PROTOCOLnn=Y`, где `nn` — текущий день месяца БЕЗ ведущего нуля по дате сервера (`date('j')`; например 10-го: `BX_SUPPORT_PROTOCOL10=Y`). На очень старых версиях параметр был `BX_SUPPORT_MODE=Y`. Появится блок «Системная область» со списком системных модулей — оставить нужные и нажать «Перезагрузить все файлы».
**Границы:** перезаписывает ТОЛЬКО ФАЙЛЫ — БД, настройки сайта, кастомный код вне ядра и версии модулей не трогаются. Требует активную (непросроченную) лицензию: блок показан, но не выполнится без действующей лицензии. Для БД-стороны частично применённого апдейта — `Настройки → Инструменты → Проверка системы` (валидирует структуру, может авто-исправить COLLATION-несоответствия).
Источник (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=35&LESSON_ID=14020

## Проверка

**Режим «только файлы»** (без живого ядра):
- В `/bitrix/.settings.php` — `utf_mode => true`; в `/bitrix/php_interface/dbconn.php` — `define("BX_UTF", true);`.
- При фатале psr/log: в `/local/composer.json` присутствует пин `psr/log:^3.0`; нет дублирующей старой папки `psr/log` вне `/bitrix/modules/main/vendor/psr/`.
- Если применялся ручной хотфикс `cacheengine.php` — он откачен после обновления ядра (файл совпадает с оригиналом).
- В `php.ini`/`opcache.ini` на время апдейта `opcache.validate_timestamps=On`.

**Режим «живой Битрикс»**:
- `Настройки → Marketplace → Обновление платформы` показывает список обновлений без ошибок соединения/лицензии.
- `Настройки → Инструменты → Проверка системы` — нет красных пунктов и нет недоприменённых шагов обновления в финальном списке.
- После повтора апдейтера нет непоставленных рекомендуемых обновлений; версия ядра соответствует целевой.
- Кэш очищен; журнал ошибок не фиксирует новых фаталов; ключевые сценарии проходят.

## ⚠️ Риски

- **⚠️ Прерывание «зависшего» апдейта повреждает данные.** Применение может занять от минут до часов; дайте тяжёлой операции БД завершиться и повторите апдейтер, а не убивайте процесс.
- **⚠️ Запуск апдейтера с боевым ключом на копии → блокировка ключа** (`[ERROR_WRONG_CODE]`). Снятие — только через техподдержку. Для легальной dev-копии используйте «Установка для разработки».
- **⚠️ Ручные хотфиксы файлов ядра** (правка `cacheengine.php` и т.п.) — временная мера. Не оставлять их после обновления до целевой версии: иначе система обновлений будет ждать оригиналы файлов.
- **⚠️ Нет бэкапа перед ремонтом БД-стороны** — потеря данных при сбое авто-исправлений структуры. Бэкап обязателен.
- **support-режим перезаливки** трогает только файлы; наполовину применённые миграции БД им не чинятся — для них отдельно «Проверка системы».

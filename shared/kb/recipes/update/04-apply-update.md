# Применение обновления безопасно (ядро → модули → решения)

> **Класс исполнения: E3** — необратимая/привилегированная операция: на проде по умолчанию не исполнять (инструкция человеку или изолированная копия). См. [«0. Право исполнять»](../../conventions.md).

## Цель

Установить обновления «1С-Битрикс: Управление сайтом» (ядро 26.x) так, чтобы не потерять данные и не положить боевой сайт: свежий бэкап, прогон на копии/стейдже, правильный порядок, контроль кэша и конвертаций, перенос на прод после проверки.

## Когда применять

- Вышли обновления ядра/модулей в системе SiteUpdate, и их нужно установить.
- Готовится переход через мажор (24 → 25 → 26): обновляться надо поэтапно, через промежуточные версии, а не «прыжком».
- Перед апгрейдом окружения (PHP ≥ 8.2 с 01.02.2026, MySQL ≥ 8.0 с 01.09.2026): сначала довести ядро и модули до версии, совместимой с новым PHP/MySQL, и только потом переключать версию на сервере. Порядок («рукопожатие» версий) разобран в Шагах.
- Нужна детерминированная установка точных версий в CI/деплое (expert-mode JSON).

## Шаги

1. **⚠️ Свежий полный бэкап (файлы + БД) — обязательно.** `Настройки → Инструменты → Резервное копирование → Создание резервной копии`. Скрипт `restore.php` для восстановления берётся по ссылке внизу страницы «Список резервных копий». Откат из бэкапа теряет данные, накопленные после снятия копии (новые заказы, контент), поэтому план — не «откатить прод», а «обновить копию и перенести».
2. **Проверка системы.** `Настройки → Инструменты → Проверка системы → Начать тестирование`. Красные пункты исправить до обновления (память, права на запись «полная/папка загрузки/ядро», версии PHP/MySQL). Проверьте, что `mbstring.func_overload=0` — при включённой опции система обновлений не работает.
3. **Кодировка.** Сайт должен быть на UTF-8. Однобайтовые установки (cp1251) с 23.200.0 не обновляются — система блокирует апдейт до конвертации (мастер `/bitrix/admin/wizard_list.php`). Конвертацию выполняйте отдельной процедурой на копии, это не часть рядового обновления.
4. **Сначала на копии/стейдже — точная копия прода.** ⚠️ Не запускайте систему обновлений с боевым лицензионным ключом на копии (см. Риски). Каноническая последовательность вендора: тестовый сервер должен быть точной копией боевого — та же версия ядра, тот же PHP, та же СУБД (иначе апдейтер ведёт себя иначе и проверка на стейдже не показательна). Сначала обновляете и проверяете тест → затем на проде делаете бэкап → накатываете те же провалидированные обновления платформы → и только потом деплоите проверенный код/изменения БД. Доступ к серверу на время обновления ограничьте одним администратором, чтобы не запустить два установщика одновременно.
5. **Порядок установки строго: SiteUpdate → ядро (main) → системные модули → сторонние модули/решения и шаблоны.** `Настройки → Marketplace → Обновление платформы`. Кнопка «Установить рекомендуемые обновления» ставит всё разом; либо отметить модули вручную и «Установить обновления».
6. **Зависимости модулей.** Связанные модули (например Торговый каталог ↔ Валюты) выбирайте все вместе либо ни одного — иначе установка не пройдёт. В экспертном режиме можно задать целевую версию в колонке «Версия до», система доберёт промежуточные и проверит зависимости.
7. **Не прерывать процесс.** Если кажется, что «завис» — это шаговая загрузка (апдейтер гонит шаги AJAX-запросами; тяжёлый DDL вроде `CREATE INDEX` на большой таблице может идти минуты). Кнопка «Остановить установку» срабатывает ПОСЛЕ завершения текущего модуля, а не в середине, поэтому целостность не нарушается. При сбое (502/504, обрыв) штатное восстановление — просто повторный запуск: апдейтер возобновляет очередь с ещё не установленных обновлений.
8. **Рукопожатие версий при мажорной миграции окружения.** Когда апдейт идёт в связке с подъёмом PHP/СУБД, порядок строгий и взаимозависимый: (1) обновить ядро и ВСЕ модули до максимума на ТЕКУЩЕМ (старом) PHP; (2) поднять PHP инкрементально, по одной версии (5.6 → 7.0 → 7.4 → 8.0 → 8.1 → 8.2/8.3), с бэкапом и проверкой между шагами, версии не перепрыгивать; (3) на новом PHP повторно прогнать обновление Битрикса (дообновить ядро под новый PHP); (4) СУБД обновлять ПОСЛЕДНЕЙ. Коробочные обновления не тянутся, пока PHP не ≥ 8.2, но и PHP нельзя поднять, пока ядро недостаточно свежее — отсюда обязательное чередование. До перехода на PHP 8 ядро должно быть выше 22.x.
9. **⚠️ MySQL 8.0 — переход в один конец.** После апгрейда на MySQL 8.0 в меню BitrixVM нет пункта возврата на 5.7. ДО шага снимите полный дамп БД и предупредите команду: откат на 5.7 штатными средствами окружения недоступен. Серверы БД обновляйте по одному, а не «все разом».
10. **Приложения Marketplace — отдельно.** Сторонние решения обновляются через `Настройки → Marketplace → Обновление решений` (отдельно от платформы).
11. **Снять кэш после обновления.** `Настройки → Настройки продукта → Автокеширование` → очистить файлы кэша (выбрать все опции), включая композитный кэш.
12. **Проверить конвертации и сценарии.** Если модуль предложил конвертацию данных — выполнить и проверить журнал ошибок. Прогнать ключевые сценарии: навигация, формы, авторизация, для магазина — оплата.
13. **Перенос на прод.** После успешной проверки на копии переносите изменения на боевой (через `restore.php` или повторив установку на проде в окно обслуживания). Боевой сразу «вслепую» не обновляем.
14. **Мониторинг 3–5 дней** после релиза, затем финальный бэкап стабильного состояния.

## Рабочий сниппет/конфиг

CLI-обновление требует настроенного Composer + Symfony Console (точка входа `bitrix/modules/main/cli/bitrix.php`). В «голом» дистрибутиве CLI не запустится — сначала настройте Composer по официальной документации. Запуск из `DOCUMENT_ROOT/bitrix`.

Файл-обёртка (по шапке `bitrix.php`), путь в проекте — корень рядом с `/bitrix`:

```php
#!/usr/bin/php
<?php
// /bitrix.php (обёртка для CLI)
$_SERVER["DOCUMENT_ROOT"] = realpath(__DIR__);
require_once(__DIR__ . '/bitrix/modules/main/cli/bitrix.php');
```

Команды обновления (модуль main):

```bash
# обновить все доступные обновления модулей
php bitrix.php update:modules

# обновить только указанные модули (зависимости доберутся автоматически)
php bitrix.php update:modules -m main,iblock,ui

# детерминированно: пин точных версий (expert-mode) из JSON для CI/деплоя
php bitrix.php update:versions updates.json
```

Все update-команды требуют явного подтверждения — без подтверждения CLI ничего не устанавливает (вендор-блог, дословно: «команда без подтверждения устанавливать ничего не будет!»). JSON expert-режима валидируется через `\CUpdateExpertMode::isCorrectModulesStructure()`; список версий экспортируется/импортируется между стейджем и продом для синхронизации.

После изменения схемы/обновления модулей пересоберите ORM-аннотации:

```bash
php bitrix.php orm:annotate -m all
```

## Проверка

**Режим «только файлы»** (без живого ядра):
- В `/bitrix/.settings.php` подтвердить `utf_mode => true`; в `/bitrix/php_interface/dbconn.php` — `define("BX_UTF", true);` (нет строк cp1251).
- Свежий архив присутствует в `/bitrix/backup/`, имя содержит дату; пароль шифрования сохранён отдельно (без него восстановление невозможно).
- Для CLI: `composer.json` и обёртка `bitrix.php` на месте; иначе `bitrix.php` напечатает «Symfony Console is not installed».
- В `.settings.php` модуля — секция `console.commands` содержит `Update\ModulesCommand` / `Update\VersionsCommand` (команды доступны).

**Режим «живой Битрикс»**:
- `update:versions` показывает список обновлений без ошибок (адрес сервера обновлений — `www.1c-bitrix.ru`).
- «Проверка системы» — нет красных пунктов; PHP ≥ 8.2, MySQL ≥ 8.0.
- После установки в `Обновление платформы` нет непоставленных рекомендуемых обновлений; версия ядра соответствует целевой.
- Кэш очищен; ключевые сценарии (формы, авторизация, для магазина — оформление заказа) проходят.
- Инспектор сайтов / журнал ошибок не фиксирует новых сбоев в течение 3–5 дней.

## ⚠️ Риски

- **⚠️ Один лицензионный ключ на боевом и на копии → бан ключа.** Если систему обновлений запустить с боевым ключом на тестовой копии, сервер фиксирует несовпадение и блокирует ключ (ошибка `[ERROR_WRONG_CODE] Проверка сайта не пройдена`). Снятие — только через техподдержку. Используйте отдельный тестовый ключ/окружение, не запускайте SiteUpdate с боевым ключом на копии.
- **⚠️ Обновление боевого «вслепую» = простой и потеря данных.** Сначала копия/стейдж, затем перенос. Откат боевого из бэкапа теряет всё, что накоплено после копии.
- **⚠️ Нет свежего бэкапа перед апдейтом — потеря данных при сбое.** Бэкап обязателен и до обновления, и финальный после стабилизации.
- **⚠️ Прыжок через несколько мажоров.** Обновляйтесь поэтапно через промежуточные версии; авторские доработки и сторонние модули проверяйте на совместимость. В ядре 2026 удалены базовые классы `CAll*` и JS-расширения `core_fx` / `core_ls` — прямые обращения к ним вызовут ошибку.
- **Зафиксированные ранние версии в экспертном режиме** оставляют известные уязвимости — без необходимости не фиксируйте версии ниже актуальных.
- **Окружение раньше совместимости.** Не переключайте PHP/MySQL на сервере, пока ядро и модули не доведены до совместимой версии; после переключения повторно прогоните «Проверку системы» и доустановите обновления.
- **⚠️ MySQL 8.0 — без обратного пути через меню BitrixVM.** В меню BitrixVM нет пункта возврата на 5.7 после апгрейда на 8.0. Полный дамп БД снимать ДО шага; команду предупреждать заранее. PHP поднимать по одной версии, без перепрыгивания.
- **Не выключать «Загружать только стабильные обновления»** на боевых проектах; бета — только на тестовом окружении.

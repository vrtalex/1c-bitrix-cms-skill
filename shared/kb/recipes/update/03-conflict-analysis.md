# Рецепт Update-03: Анализ конфликтов и рисков перед обновлением

Под-скилл: `1c-bitrix-cms-update` • Платформа: «1С-Битрикс: Управление сайтом», ядро 26.x, модули `main`, `update_system` (SiteUpdate).

## Цель
Собрать офлайн (до запуска системы обновлений) полную карту рисков апдейта: найти модификации файлов ядра, переопределения компонентов/шаблонов в `/local`, кастом-модули с зависимостью от старого API, проверить совместимость PHP/СУБД и кодировки с целевой версией, оценить состояние лицензии и активные правки на проде. Итог — заполненный чек-лист и список конфликтов с приоритетом, который решает: можно обновляться сразу, нужна подготовка или сначала миграция окружения.

## Когда применять
- Перед любым обновлением ядра/модулей на проекте, который вы не обновляли давно или приняли от другого подрядчика.
- Перед переходом через несколько мажоров (24 → 25 → 26) — здесь конфликтов больше всего.
- Перед сменой PHP/СУБД на сервере (апгрейд платформы тянет требования к версии ядра).
- Когда непонятно состояние проекта: есть ли правки ядра, что лежит в `/local`, на какой версии PHP работает сайт.

Если нужно выполнить само обновление и откат — это отдельный рецепт домена; здесь только анализ ПЕРЕД ним.

## Шаги
1. **Зафиксируйте отправную и целевую точки.** Текущая версия ядра — `Настройки → Об установленном решении` или константа `SM_VERSION` в `bitrix/modules/main/classes/general/version.php`. Целевую версию задайте через Экспертный режим (см. ниже) — он покажет цепочку промежуточных версий и зависимости модулей.
2. **Сверьте файлы ядра с эталоном.** Любой изменённый файл в `/bitrix/modules/` или `/bitrix/components/bitrix/` делает результат автообновления непредсказуемым — гарантия обратной совместимости действует только для немодифицированного ядра. Контроль целостности: `Настройки → Проактивная защита → Контроль целостности` (живой Битрикс) либо сверка контрольных сумм офлайн (см. сниппет).
3. **Прогоните статику по проекту:** `sh ../../../scripts/check-conventions.sh <корень_проекта>`. Метка `[REVIEW] Файлы в каталогах ядра` — кандидаты на правки ядра; перенесите такой код в `/local` до обновления.
4. **Инвентаризируйте переопределения в `/local`.** Соберите список шаблонов компонентов, `result_modifier.php`, `component_epilog.php` и копий системных компонентов в `/local`, которые завязаны на разметку или `$arResult` штатной версии из `/bitrix`. После апдейта штатная версия может разойтись с переопределением — отметьте такие места для ручной сверки после обновления.
5. **Проверьте кастом-модули на старое API.** Для модулей в `/local/modules/` найдите вызовы, удалённые/изменённые в целевой ветке: классы `CAll*` (удалены в ветке 2026), JS-расширения `core_fx` и `core_ls` (функционал перенесён в `core`), а также конструкции, несовместимые с PHP 8 (см. чек-лист). Прямое обращение к удалённым сущностям приведёт к ошибке после апдейта.
   ⚠️ Основной класс конфликта — «ядро обновили, модуль — нет»: тиражный/платный модуль, оставшийся на старой версии под новым ядром, падает на вызове изменившегося API (`Call to undefined method ...` / `Class '...' not found`). Поэтому совместимость каждого тиражного/платного модуля с целевой версией ядра проверяют отдельно ДО апдейта, а связанные модули обновляют вместе.
6. **Не полагайтесь на порядок выполнения updater'ов между модулями.** Апдейты одного модуля ставятся строго последовательно по версиям, но когда в один заход обновляется несколько модулей, межмодульный порядок выполнения их `updater.php` вендором объявлен неопределённым — код не должен предполагать, что чужой updater отработал раньше. Связанные модули (каталог/валюта и т.п.) обновляют вместе; рекомендуемая практика — головной модуль первым. Это учитывают при оценке кастома, который рассчитывает на готовые данные/структуры другого модуля.
7. **Проверьте дубли Composer-пакетов `psr/*`.** Системные PSR-пакеты лежат в `/bitrix/modules/main/vendor/psr/`. Второй, более старый `psr/log` (или `psr/container` / `psr/http-message`), подтянутый сторонним модулем, из `php_interface` или из `/local`, после обновления ядра даёт фатал несовместимости сигнатур (`Declaration of ... must be compatible with Psr\Log\LoggerInterface::log(...)`). Профилактика: в `/local/composer.json` запинить `psr/log ^3.0` (а также `psr/container ^2.0`, `psr/http-message ^2.0`, `psr/http-client ^1.0`), выполнить в `/local` `composer update` и `composer install --no-dev`; альтернатива — `"replace": {"psr/log":"*", ...}`, чтобы пакеты из `main` не грузились дважды. Composer-зависимости обновляют одновременно с ядром и сначала тестируют на стейдже.
8. **Проверьте скрытые параметры компонентов в кастомных вызовах.** Обновление модуля может тихо (без ошибки) изменить поведение скрытого параметра компонента — кэширование, выборку, рендеринг — для кастомизированных вызовов. Регрессия проявляется как «поехавшее» поведение, а не как фатал. Для кастомных вызовов системных компонентов (`bitrix:menu` и т.п.) после апдейта явно сверяют значения параметров с актуальной документацией компонента целевой версии и помечают такие вызовы для ручной проверки.
9. **Проверьте совместимость окружения.** PHP ≥ 8.2 (с 01.02.2026 обновления ниже не ставятся; рекомендуется 8.3+, для нового ядра 8.4+); MySQL ≥ 8.0 (с 01.09.2026 ограничивается поддержка ниже; рекомендуется 8.4+). ⚠️ Порядок миграции: сначала довести ядро и модули до версии, совместимой с новой PHP/СУБД, и только потом переключать платформу на сервере.
10. **Проверьте кодировку.** С 24.0.0 поддерживается только UTF-8; сайт на cp1251 не обновляется (блокировка ещё с 23.200.0). Признаки однобайтовой установки — отсутствие `define("BX_UTF", true)` в `dbconn.php` и `utf_mode => true` в `.settings.php`. ⚠️ `mbstring.func_overload` должна быть `0` — при включённой опции система обновлений не работает. Граница смысла директивы — ядро 20.100.0: до 20.100.0 для UTF-8 требовался `func_overload=2`, начиная с 20.100.0 оно не требуется и не поддерживается (должно быть удалено / `=0`). При мультиверсионном прыжке через эту границу старое значение `func_overload=2` ломает работу — снять его до перехода.
11. **Оцените лицензию и купон.** Право на обновления действует ограниченный срок после регистрации лицензии. Купон льготного продления применяется заранее или в течение 30 дней после истечения — позже льгота недоступна. На «Ограниченной лицензии» / по истечении подписки новые версии (включая обновления безопасности) не загружаются, пока подписка не продлена.
12. **Зафиксируйте активные правки на проде.** Сверьте, что на боевом сайте нет несохранённых ручных правок страниц/шаблонов, не отражённых в `/local` и git: после обновления и переноса с копии они потеряются.
13. **Соберите вердикт.** Заполните чек-лист, выпишите конфликты с приоритетом (⚠️ критичные — в первую очередь) и решение: обновляться сразу / подготовить код и окружение / сначала миграция кодировки или PHP.

### Чек-лист «что проверить перед апдейтом»
**Ядро и модификации**
- [ ] Контроль целостности пройден; изменённых файлов в `/bitrix/modules/` и `/bitrix/components/bitrix/` нет.
- [ ] Структура таблиц БД ядра не менялась вручную.
- [ ] `[REVIEW]`-сигналы `check-conventions.sh` о файлах в каталогах ядра разобраны; кастом перенесён в `/local`.

**Переопределения в `/local`**
- [ ] Составлен список шаблонов компонентов / `result_modifier.php` / `component_epilog.php` / копий системных компонентов, зависящих от штатной версии.
- [ ] Помечены места, требующие ручной сверки после апдейта (разметка, ключи `$arResult`).

**Кастом-модули и API**
- [ ] В `/local/modules/` нет обращений к удалённым `CAll*`.
- [ ] Нет обращений к JS-расширениям `core_fx` / `core_ls`.
- [ ] Код проверен на совместимость с PHP 8 (см. ниже).
- [ ] ⚠️ Совместимость каждого тиражного/платного модуля с целевым ядром проверена (риск `Call to undefined method` / `Class not found`); связанные модули обновляются вместе.
- [ ] Кастом не полагается на порядок выполнения чужих `updater.php` (межмодульный порядок неопределён).
- [ ] Нет дублирующего `psr/log` (или `psr/container` / `psr/http-message`) вне `/bitrix/modules/main/vendor/psr/`; при наличии — пины в `/local/composer.json`.
- [ ] Кастомные вызовы системных компонентов (`bitrix:menu` и т.п.) помечены для ручной сверки скрытых параметров после апдейта.

**Окружение (целевая версия)**
- [ ] PHP ≥ 8.2 (рекомендуется 8.3+/8.4+); `short_open_tag = On`.
- [ ] MySQL ≥ 8.0 (рекомендуется 8.4+). Oracle/MSSQL не поддерживаются; PostgreSQL — только Enterprise (11+).
- [ ] ⚠️ Порядок: ядро/модули → совместимая с новой PHP/СУБД версия → только потом переключение платформы.

**Кодировка**
- [ ] Сайт в UTF-8: `define("BX_UTF", true)` в `dbconn.php`, `utf_mode => true` в `.settings.php`.
- [ ] ⚠️ `mbstring.func_overload = 0`.

**Лицензия и прод**
- [ ] Право на обновления активно; при истечении — оценена возможность купона (заранее или ≤ 30 дней после).
- [ ] Активных ручных правок на проде, не отражённых в `/local`/git, нет.
- [ ] ⚠️ Свежий полный бэкап файлов и БД сделан до любых действий.

### Проблемы кода при PHP 8 (искать в кастом-модулях/компонентах)
- Конструкторы с именем класса → заменить на `__construct()`.
- Нестатические методы, вызываемые статически → объявить `public static`.
- `count()` на не-массиве → проверять `is_array()` перед вызовом.
- Использование неопределённых констант → обернуть `defined()`.
- Неэкранированные ключи массива `$array[key]` → `$array['key']`.
- `each()` — удалена; заменить на `foreach`.

## Рабочий сниппет/конфиг (путь в /local)
Анализ выполняется офлайн и read-only — он ничего не меняет в проекте. Скрипт раскладки результата держите в `/local`, например `/local/tools/preupdate-conflicts.sh`.

```sh
#!/usr/bin/env bash
# /local/tools/preupdate-conflicts.sh — офлайн-карта рисков перед обновлением.
# Запуск: sh /local/tools/preupdate-conflicts.sh /path/to/project
set -eu
ROOT="${1:?укажите корень проекта (document root)}"

echo "== Текущая версия ядра =="
grep -m1 'SM_VERSION' "$ROOT/bitrix/modules/main/classes/general/version.php" || echo "version.php не найден"

echo "== Кодировка =="
grep -q 'BX_UTF' "$ROOT/bitrix/php_interface/dbconn.php" 2>/dev/null \
  && echo "BX_UTF найден (UTF-8)" || echo "BX_UTF не найден — проверить однобайтовую кодировку"
grep -q 'utf_mode' "$ROOT/bitrix/.settings.php" 2>/dev/null \
  && echo "utf_mode присутствует в .settings.php" || echo "utf_mode не найден в .settings.php"

echo "== Переопределения в /local (кандидаты на сверку) =="
find "$ROOT/local" \( -name 'result_modifier.php' -o -name 'component_epilog.php' \) 2>/dev/null
find "$ROOT/local/components/bitrix" -maxdepth 2 -type d 2>/dev/null  # копии системных компонентов

echo "== Старое API в кастом-модулях =="
grep -rIl --include='*.php' -E '\bCAll[A-Z][A-Za-z]+' "$ROOT/local/modules" 2>/dev/null || echo "обращений к CAll* не найдено"
grep -rIl --include='*.js'  -E 'core_fx|core_ls'       "$ROOT/local"          2>/dev/null || echo "обращений к core_fx/core_ls не найдено"

echo "== mbstring.func_overload (PHP) =="
php -r 'echo "func_overload=".ini_get("mbstring.func_overload")."\n";' 2>/dev/null || echo "php cli недоступен — проверить вручную"

echo "== Дубли Composer-пакетов psr/* вне ядра =="
# Системный PSR — в /bitrix/modules/main/vendor/psr/. Любые psr/* в /local или в стороннем модуле — кандидат на конфликт сигнатур.
find "$ROOT/local" "$ROOT/bitrix/php_interface" -path '*/psr/log/*' -o -path '*/psr/container/*' -o -path '*/psr/http-message/*' 2>/dev/null \
  | grep -v '/bitrix/modules/main/vendor/psr/' || echo "дублей psr/* вне ядра не найдено"
```

Сверка контрольных сумм ядра офлайн (если есть эталон версии): сравните дерево `/bitrix/modules/<m>/` с эталонным дистрибутивом той же версии; расхождения вне `cache`/`managed_cache`/`tmp` — кандидаты на правки ядра. Сам список изменённых файлов на живом сайте даёт `Контроль целостности`.

## Проверка
**Режим «только файлы»:**
- `check-conventions.sh` отработал; `[REVIEW]`-сигналы о файлах в каталогах ядра разобраны.
- Скрипт `preupdate-conflicts.sh` показал версию, кодировку, список переопределений в `/local` и обращений к старому API.
- Подтверждено: кастом в `/local`, не в `/bitrix`; UTF-8 включён; `mbstring.func_overload = 0` (при прыжке через ядро 20.100.0 старое значение `=2` снято).
- Код кастом-модулей просмотрен на совместимость с PHP 8 по списку выше.
- Нет дублирующего `psr/log` / `psr/container` / `psr/http-message` вне `/bitrix/modules/main/vendor/psr/`.
- Версии тиражных/платных модулей сверены с целевым ядром; связанные модули намечены к совместному обновлению.

**Режим «живой Битрикс»:**
- `Контроль целостности` (Настройки → Проактивная защита) не показывает изменённых файлов ядра.
- Экспертный режим (Настройки → Настройки продукта → Настройки модулей → Главный модуль → Система обновлений) включён; целевые версии выставлены, зависимости модулей разрешены без конфликтов.
- В блоке «Ответ сервера обновлений» лицензия активна и дата корректна.
- Сделан свежий полный бэкап (Настройки → Инструменты → Резервное копирование) до перехода к обновлению.

## ⚠️ Риски
- ⚠️ **Модификации файлов ядра делают автообновление непредсказуемым** и ведут к потере права на поддержку. Кастом — только в `/local`; найденные правки ядра перенести/переписать ДО апдейта.
- ⚠️ **Несовместимость окружения блокирует обновление.** PHP < 8.2 и MySQL < 8.0 (по датам ограничения) не дадут поставить даже обновления безопасности. Сначала ядро/модули до совместимой версии, потом переключение платформы — не наоборот.
- ⚠️ **Сайт на cp1251 не обновляется** (блокировка с 23.200.0). Конвертация в UTF-8 — отдельная сложная операция с обязательным бэкапом и тренировкой на копии; на крупном каталоге возможны ошибки структуры БД, исправляемые вручную SQL.
- ⚠️ **`mbstring.func_overload ≠ 0` ломает систему обновлений.** Отключить до старта апдейта.
- ⚠️ **Обновление с боевым ключом на копии = бан ключа** (`[ERROR_WRONG_CODE] Проверка сайта не пройдена`). Анализ — read-only и безопасен, но при последующем тестировании на копии используйте отдельный тестовый ключ/окружение; снятие бана — через техподдержку.
- **Переопределения в `/local` молча расходятся со штатной версией.** Они не вызывают ошибку установки, но после апдейта поведение/вёрстка могут «поехать» — без списка для ручной сверки регрессию легко пропустить. То же касается скрытых параметров системных компонентов в кастомных вызовах: их поведение может измениться после апдейта без ошибки.
- ⚠️ **«Ядро обновили, модуль — нет».** Тиражный/платный модуль на старой версии под новым ядром падает на изменившемся API (`Call to undefined method` / `Class not found`). Совместимость каждого такого модуля с целевым ядром проверяют отдельно; связанные модули обновляют вместе.
- ⚠️ **Дубль `psr/*` вне ядра ломает загрузку фаталом несовместимости сигнатур** после обновления ядра. Системные PSR — в `/bitrix/modules/main/vendor/psr/`; второй старый `psr/log` из стороннего модуля или `/local` устраняется пинами в `/local/composer.json` (или `replace`).
- **Межмодульный порядок выполнения `updater.php` не гарантирован.** Кастом, рассчитывающий, что чужой updater отработал раньше, может сломаться; связанные модули обновляют вместе, головной первым.
- **Активные правки на проде теряются при переносе с копии.** Всё, что не в `/local`/git, перед обновлением должно быть зафиксировано.
- **Прыжок через несколько мажоров рискованнее поэтапного.** Чем больше разрыв версий, тем выше шанс, что авторские доработки и сторонние модули перестанут работать.

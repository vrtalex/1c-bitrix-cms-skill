# Рецепт 09 — Безопасное тест-обновление на копии/стейдже

> Подсистема: 1c-bitrix-cms-update. Ядро БУС 26.x. Две поддерживаемые версии API (D7 и legacy) — обе официальны.
> Основан на изучении ядра «1С-Битрикс: Управление сайтом» (SM_VERSION 26.x), официальной документации вендора и практик деплоя (помечено).

## Цель

Развернуть копию боевого сайта на стейдже, прогнать на ней обновление и при этом: (1) не израсходовать слот обновлений лицензии и не получить блокировку ключа `[ERROR_WRONG_CODE]`; (2) «обезвредить» восстановленную копию, чтобы она не слала клиентам реальные письма и не дёргала боевые endpoint'ы. Результат — изолированная копия, на которой можно безопасно тестировать апдейт перед переносом на прод.

## Когда применять

- Перед обновлением ядра/модулей боевого сайта — обновление сначала прогоняется на копии (см. `./04-apply-update.md`).
- Развёрнута копия из бэкапа / перенос на новый хостинг / клонирование для тестов.
- Нужна постоянная dev-копия рядом с боевой на том же лицензионном ключе.

## Шаги

1. **Перед первым запуском системы обновлений на непубличной копии включите «Установка для разработки».** Путь: Настройки → Настройки продукта → Настройки модулей → Главный модуль → вкладка «Система обновлений» → опция «Установка для разработки». Так копия учитывается отдельно и не расходует слот обновлений лицензии. По лицензии допустимы две установки: одна публичная (боевая) и одна локальная/dev (недоступная из интернета).
2. **Соблюдайте правило «обновлять только одну из двух установок».** Систему обновлений с боевым лицензионным ключом запускайте только на ОДНОЙ копии (прод ИЛИ dev), не на обеих. Система обновлений привязана к состоянию установки, чтобы один ключ не использовался для обновления нескольких живых копий.
3. **Понимайте, что такое `[ERROR_WRONG_CODE]`.** Система обновлений «запоминает» состояние установки и привязана к нему; ошибка возникает, когда текущее состояние не совпадает с зафиксированным на момент последнего обновления. Срабатывает при запуске обновления с боевым ключом на клоне, после восстановления из бэкапа, после переноса/смены хостинга, смены IP или параметров хостинга. Это штатный механизм, не позволяющий обновлять несколько живых копий на одном ключе.
4. **Если ключ заблокирован — обращайтесь в техподдержку вендора.** Автоматического сброса привязки нет; снятие — только через техподдержку. Подготовьте: лицензионный ключ; что делали перед ошибкой (restore из бэкапа / перенос / смена IP/хостинга); общее число установок, включая локальные/dev-копии; домены, использующие этот ключ.
5. **Обезвредьте восстановленную/клонированную копию ДО любых тестов.** Очередь почты и агенты переживают восстановление — без обезвреживания копия будет слать реальные письма клиентам и гонять агенты на боевые endpoint'ы. Чек-лист (по практике):
   - закрыть копию от публичного доступа (Главный модуль → вкладка «Доступ»);
   - отключить отправку почты на уровне сервера/VM (заблокировать SMTP), чтобы письма физически не уходили;
   - отключить почтовые события/шаблоны (Настройки → Настройки продукта → Почтовые/Email-события) и почтовые ящики;
   - остановить/проверить агенты и обработчики cron, бьющие в боевые endpoint'ы, отключить интеграции с внешними боевыми сервисами;
   - выставить `robots.txt` → `Disallow: /`, чтобы копию не индексировали;
   - очистить весь кэш.
6. **Только после обезвреживания и включения «Установки для разработки» запускайте обновление** на копии по штатной процедуре (`./04-apply-update.md`): бэкап → проверка системы → ядро → модули → решения → сброс кэша → проверка сценариев.
7. **При переезде на новый сервер.** Скопируйте файлы и БД на новый сервер, переключите DNS, на старой установке обновления НЕ запускайте и удалите её после переключения. Не переносите продукт «туда-обратно» и не обновляйте обе копии.

## Рабочий сниппет

Проверка состояния подписки/доступа обновлений из кода (модуль `rest`) — полезно в пред-флайт-скрипте перед запуском апдейта на копии:

```php
<?php
use Bitrix\Main\Loader;
use Bitrix\Rest\Marketplace\Client;

if (Loader::includeModule('rest')) {
    $hasAccess  = Client::isSubscriptionAccess();      // доступ к обновлениям/решениям
    $finalDate  = Client::getSubscriptionFinalDate();  // дата окончания подписки
}
```

Минимальное «обезвреживание почты» на уровне проекта — отключить отправку через перехват события `OnBeforeMailSend` (страховка в дополнение к блокировке SMTP на сервере). Размещать в `/local/php_interface/init.php`:

```php
<?php
use Bitrix\Main\EventManager;

// Только на dev/staging-копии: глушим исходящую почту
if (defined('BX_STAGING') && BX_STAGING === true) {
    EventManager::getInstance()->addEventHandler(
        'main',
        'OnBeforeMailSend',
        static function (): bool {
            return false; // письмо не отправляется
        }
    );
}
```

`robots.txt` для закрытой копии (полный запрет индексации):

```text
User-agent: *
Disallow: /
```

## Проверка

**Режим «только файлы»** (без живого ядра):
- В копии `robots.txt` содержит `Disallow: /`.
- В `/local/php_interface/init.php` присутствует обработчик `OnBeforeMailSend` (или иной механизм глушения почты), привязанный к флагу стейджа.
- В конфигурации сервера/VM SMTP-отправка для копии заблокирована (нет рабочего sendmail/relay на боевой адрес).

**Режим «живой Битрикс»:**
- В Главном модуле включена опция «Установка для разработки»; копия закрыта на вкладке «Доступ».
- Почтовые события/ящики отключены; тестовая отправка письма не уходит наружу.
- Система обновлений на копии не выдаёт `[ERROR_WRONG_CODE]` (флаг dev-установки учтён) — а на боевом ключе одновременно обновляется только одна установка.
- Агенты/cron копии не обращаются к боевым внешним сервисам.

## ⚠️ Риски

- **⚠️ Запуск системы обновлений с боевым ключом на копии → `[ERROR_WRONG_CODE]` и блокировка ключа.** Снятие — только через техподдержку вендора. Перед обновлением на непубличной копии включайте «Установка для разработки» и обновляйте только одну из двух установок.
- **⚠️ Необезвреженная копия шлёт реальные письма и дёргает боевые endpoint'ы.** Очередь почты и агенты переживают восстановление. Сначала обезвредьте (доступ, SMTP, почтовые события, агенты, robots), затем тестируйте.
- **⚠️ Перенос «туда-обратно» и обновление обеих копий.** Рассинхрон состояния ведёт к блокировке ключа. При переезде обновляйте/держите активной только новую установку, старую удаляйте после переключения DNS.
- **Злоупотребление тумблером dev-установки.** Переключения опции логируются на стороне сервера лицензий; частое переключение, чтобы обновлять две живые копии, может привести к блокировке коммерческого ключа.
- **Незакрытая копия в индексе/доступе.** Без `Disallow: /` и закрытия доступа копия может попасть в поисковую выдачу и принимать реальный трафик/заказы.

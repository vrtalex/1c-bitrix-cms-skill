# Рецепт 08 — Обновление решений Marketplace, партнёрских и собственных модулей

> Подсистема: 1c-bitrix-cms-update. Ядро БУС 26.x. Две поддерживаемые версии API (D7 и legacy) — обе официальны.
> Основан на изучении ядра «1С-Битрикс: Управление сайтом» (SM_VERSION 26.x), официальной документации вендора и исходного кода business-редакции (модуль `rest`).

## Цель

Обновить то, что лежит вне ядра вендора: партнёрские (тиражные) модули, приложения Marketplace и собственные модули проекта. Цель — провести их по правильному каналу обновления, не перепутав с «Обновлением платформы», корректно отработать цепочку версий и контракт `updater.php`, и не потерять проектные доработки при обновлении ядра.

## Когда применять

- Установлены партнёрские/тиражные решения, для которых вышли обновления (отдельный канал от ядра).
- Используются приложения Marketplace, устанавливаемые через модуль `rest`, и их нужно обновить программно (деплой/CI).
- В проекте есть собственный модуль (`/local/modules/...`), для которого нужно собрать и применить обновление с `updater.php` / `version_control.txt`.
- Планируется обновление ядра, и нужно убедиться, что доработки переживут перезапись `/bitrix`.

## Шаги

1. **Определите происхождение каждого модуля и нужный канал.** Полный код партнёрского/тиражного модуля имеет формат `код_партнёра.код_модуля` (есть точка, например `mycompany.catalogpro`). Такие решения обновляются НЕ через «Обновление платформы», а через отдельный раздел. Три канала разнесены:
   - **«Обновление платформы»** (Marketplace → Обновление платформы) — ядро (`main`) и модули вендора через SiteUpdate.
   - **«Обновление решений»** (Marketplace → Обновление решений) — партнёрские/тиражные модули. Здесь же вкладка **«Активация купона»**: код от разработчика модуля для активации/продления решения.
   - **Приложения Marketplace** — через модуль `rest` (классы `Bitrix\Rest\Marketplace\*`).
2. **Партнёрские модули — через «Обновление решений».** Раздел «Сторонние обновления» предназначен для обновлений дополнительных модулей, разработанных партнёрами «1С-Битрикс». Вкладки: «Установка обновлений», «Список обновлений», «Активация купона». Если у решения есть код активации/продления — активируйте его на вкладке «Активация купона» до установки.
3. **Гейт совместимости с целевым ядром.** Если у платного/партнёрского решения нет совместимой версии под целевую версию ядра — остановитесь и обратитесь к разработчику решения. Частичный апдейт (ядро обновлено, модуль — нет) даёт `Call to undefined method` / `Class not found`. Варианты до получения совместимой версии: временно отключить модуль либо отложить обновление ядра.
4. **Приложения Marketplace — программно через модуль `rest`.** Опрос и установка/переустановка идут через `Bitrix\Rest\Marketplace\Client` и `Bitrix\Rest\Marketplace\Application` (см. сниппет). Это коробочный путь; не путать с облачным Market Битрикс24.
5. **Собственный модуль — обновление по цепочке версий.** Версия живёт в `install/version.php` (массив `$arModuleVersion` с ключами `VERSION` и `VERSION_DATE`), читается через `ModuleManager::getVersion($moduleName)`. Обновления ставятся друг за другом строго по возрастанию версии; каждое обновление несёт только дельту относительно предыдущего. Если в одном хите ставится несколько версий одного модуля — их апдейтеры исполняются последовательно по версиям.
6. **Контракт `updater.php` (соблюдать при сборке и проверке апдейтера):**
   - **Новый API текущего обновления внутри апдейтера недоступен** — обращение к новому классу этого же обновления даст `Class '...' not found`. Не вызывайте классы, которые поставляются тем же пакетом.
   - **Идемпотентность обязательна** — обновление может переустановиться несколько раз; код должен безопасно переживать повторный запуск.
   - Обновление выполняется независимо от того, установлен ли модуль, — наличие модуля проверяйте в самой логике апдейтера.
   - **Прервать установку** — присвоить `$errorMessage` (например, при отсутствии зависимости).
   - **Файлы копируйте через** `$updater->CopyFiles(src_rel, dst_rel)`; пути относительны от папки обновления, целевой путь раскрывается в `/local/...` или `/bitrix/...`.
7. **Межмодульные зависимости версий — `version_control.txt`.** Опциональный файл объявляет, при каких версиях смежных модулей ставить обновление. Формат строки `iblock,13.8.0` означает «ставить, если `iblock` версии ≥ 13.8.0 ИЛИ если `iblock` не установлен вообще». По строке на каждую зависимость.
8. **Учтите, что переживает обновление ядра.** Обновление перезаписывает каталоги вендора в `/bitrix` (`modules`, `components/bitrix`, `js`, `activities/bitrix`, `blocks/bitrix`, `wizards/bitrix`, `routes`). Проектные доработки и копии шаблонов компонентов держите в `/local` — это и есть штатное разделение «продукт ↔ проект», которое сохраняет кастомизации при обновлении ядра. Перед обновлением убедитесь, что доработок в каталогах вендора нет.
9. **Демо-данные.** Импорт демо-данных — это шаг мастера установки решения (`#SITE_DIR#`, `/xml/<язык>/`, `WizardServices::ImportIBlockFromXML`), а не обновления кода. Обновление решения по умолчанию демо-данные не переигрывает; повторный запуск установки решения, напротив, может затронуть демо-структуру — проверяйте это до повторного install.

## Рабочий сниппет

Опрос и установка приложений Marketplace в коробке через модуль `rest`. Классы и методы подтверждены по исходному коду business-редакции (`modules/rest/lib/marketplace/`).

```php
<?php
use Bitrix\Main\Loader;
use Bitrix\Rest\Marketplace\Client;
use Bitrix\Rest\Marketplace\Application;

if (!Loader::includeModule('rest')) {
    return;
}

// Сколько доступно обновлений приложений Маркетплейса
$num = Client::getAvailableUpdateNum();

// Список доступных обновлений (можно сузить кодами приложений)
$updates = Client::getUpdates(['partner.app_one', 'partner.app_two']);

// Доступное обновление по конкретному коду решения
$available = Client::getAvailableUpdate('partner.app_one');

// Дистрибутив установки/обновления решения по коду и версии
$install = Client::getInstall('partner.app_one', false);

// Установка / переустановка / удаление решения
Application::install('partner.app_one');          // $version=false → актуальная
Application::reinstall($installedId);
Application::uninstall('partner.app_one', false); // $clean=false — без удаления данных
```

Снятие версии собственного модуля для пред-/постпроверки апдейта:

```php
<?php
use Bitrix\Main\ModuleManager;

// Версия установленного модуля (из install/version.php)
$version = ModuleManager::getVersion('mycompany.catalogpro');

// Установлен ли модуль вообще
$installed = ModuleManager::isModuleInstalled('mycompany.catalogpro');
```

Минимальный `install/version.php` собственного модуля (ключи `VERSION` / `VERSION_DATE`; точный формат даты задаёт сборщик — не выдумывать):

```php
<?php
$arModuleVersion = [
    'VERSION'      => '1.2.0',
    'VERSION_DATE' => '2026-06-01 12:00:00',
];
```

## Выбор API

- **Коробка (БУС):** установка/обновление решений Marketplace — модуль `rest`, классы `Bitrix\Rest\Marketplace\Client` / `Application`. Версии модулей — `Bitrix\Main\ModuleManager`. Это путь для скриптов деплоя.
- **Облачный Market Битрикс24** (REST-приложения портала) — отдельная история и отдельная документация; на коробку «Управление сайтом» её API не переносится, хотя имена методов в коробочных `Client`/`Application` совпадают по смыслу.
- **Партнёрские/тиражные модули** штатно обновляются через админ-раздел «Обновление решений», а не программно через `rest`.

## Проверка

**Режим «только файлы»** (без живого ядра):
- У собственного модуля присутствует `install/version.php` с `$arModuleVersion['VERSION']` и `['VERSION_DATE']`.
- В пакете обновления есть `version.php`; при наличии — `description.*`, `updater.php`, `version_control.txt`. В `updater.php` нет обращений к новым классам этого же пакета обновления; копирование файлов — через `$updater->CopyFiles(...)`.
- Проектные доработки и копии шаблонов компонентов лежат в `/local`, а не в каталогах вендора (`/bitrix/modules`, `/bitrix/components/bitrix`, `/bitrix/js`, `/bitrix/activities/bitrix`, ...).
- Партнёрские модули опознаются по коду с точкой `код_партнёра.код_модуля`.

**Режим «живой Битрикс»:**
- `ModuleManager::getVersion('код.модуля')` возвращает ожидаемую версию после обновления.
- В разделе «Обновление решений» нет непоставленных обновлений для партнёрских модулей; для решений с купоном код активирован на вкладке «Активация купона».
- Для приложений Marketplace `Client::getAvailableUpdateNum()` равен 0 (или ожидаемому) после применения.
- Журнал ошибок не фиксирует `Class not found` / `Call to undefined method` после обновления.
- Ключевые сценарии модуля проходят; кэш очищен.

## ⚠️ Риски

- **⚠️ Платный/партнёрский модуль без совместимой версии под целевое ядро — блокер.** Не форсируйте обновление ядра, если решение под него ещё не выпущено: обратитесь к разработчику решения, временно отключите модуль или отложите обновление. Иначе — `Call to undefined method` / `Class not found`.
- **⚠️ Доработки в каталогах вендора теряются при обновлении ядра.** `/bitrix/modules`, `/bitrix/components/bitrix`, `/bitrix/js`, `/bitrix/activities/bitrix`, `/bitrix/blocks/bitrix`, `/bitrix/wizards/bitrix`, `/bitrix/routes` перезаписываются. Правка оригиналов означает потерю изменений после обновления и снятие модуля с поддержки. Кастомизация шаблона компонента — копированием в `/local`, не правкой оригинала.
- **⚠️ Новый API недоступен внутри `updater.php`.** Обращение к классу, поставляемому тем же обновлением, на хите обновления даст `Class '...' not found`. Откладывайте такие вызовы на последующие хиты/обновления.
- **Неидемпотентный апдейтер.** Обновление может выполниться повторно; не-идемпотентный код (двойная вставка данных, повторное переименование) приведёт к рассинхрону. Проверяйте текущее состояние перед изменением.
- **Перепутанный канал.** Установка партнёрского решения через «Обновление платформы» (или наоборот) не сработает — каналы и источники разные. Сверяйтесь с форматом кода модуля.
- **Повторный install решения вместо обновления** может затронуть демо-структуру (инфоблоки, публичку). Для обновления кода используйте канал обновления, а не повторную установку.

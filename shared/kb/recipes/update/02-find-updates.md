# Поиск доступных обновлений (ядро, модули, решения Marketplace)

## Цель

Определить, какие обновления доступны для проекта «1С-Битрикс: Управление сайтом» (ядро 26.x): обновления ядра и системных модулей через SiteUpdate, обновления сторонних решений Marketplace, а также собрать «что меняется» из истории версий вендора. Результат — список кандидатов на установку, который дальше идёт в анализ конфликтов (см. `./03-conflict-analysis.md`).

## Когда применять

- Перед плановым обновлением ядра/модулей — нужно увидеть полный перечень доступных пакетов.
- При подготовке деплоя/CI, когда требуется детерминированный список версий.
- При диагностике «почему обновления не приходят» (адрес сервера, лицензия, окружение).
- Когда нужно оценить объём изменений между мажорами (24→25→26) до фактической установки.

## Шаги

1. **Проверьте предусловия SiteUpdate** (Настройки → Настройки продукта → Настройки модулей → Главный модуль → вкладка «Система обновлений»):
   - адрес сервера обновлений — `www.1c-bitrix.ru` (поддерживается HTTPS);
   - опция «Загружать только стабильные обновления» включена (для боевых проектов оставляйте включённой);
   - на уровне PHP `mbstring.func_overload=0` — при включённом overload система обновлений не работает.

   **Канал обновлений по умолчанию — stable.** Пока флаг «Загружать только стабильные обновления» включён, инстанс получает только стабильные пакеты; beta/test не приходят. Чтобы получать предрелизные версии, флаг снимают (или включают beta на вкладке «Дополнительно»). Обратное — способ отложить рискованный релиз: держать stable-only включённым и/или через экспертный режим запинить целевую версию («Версия до»), а не ставить последнюю.

   **Для клонов/стейджа — флаг «Установка для разработки».** В этой же вкладке Главного модуля флаг «Установка для разработки» помечает копию как dev/тестовую: такая копия не занимает слот обновления боевой лицензии. Его выставляют на стейдже перед тем, как открывать там список обновлений (см. рецепт безопасного применения и отката в «Связано»).
2. **Откройте список обновлений ядра и модулей**: Marketplace (Настройки) → «Обновление платформы». Здесь отображаются доступные пакеты для ядра (main) и системных модулей. Кнопка «Установить рекомендуемые обновления» соответствует полному списку; вручную можно отметить отдельные модули.
3. **Включите экспертный режим** (если нужно увидеть целевые версии и зависимости): на той же странице появляется вкладка «Экспертный режим», где в колонке «Версия до» видны конкретные доступные версии каждого модуля и автоматически подсвечиваются зависимости (пример: `catalog 25.0.0` требует `ui 25.50.0`; `sale 25.0.100` требует `seo 24.700.0`).
4. **Проверьте обновления сторонних решений** отдельно: Настройки → Marketplace → «Обновление решений». Системные обновления и обновления приложений Marketplace — это два разных списка.
5. **Если в проекте настроен Composer + Symfony Console** — получите тот же список из CLI (удобно для CI/деплоя):
   - `php bitrix.php update:versions` — показ доступных версий обновлений;
   - `php bitrix.php list` — убедиться, что секция `update:*` доступна.
   CLI требует настроенного Composer; в «голом» дистрибутиве команды не запустятся (см. раздел «Проверка»).
6. **Прыжок через несколько мажоров планируйте поэтапно.** Обновления некоторых модулей имеют зависимости на обновления других модулей, поэтому прыжок через много мажорных версий (например 24 → 25 → 26) выполняется staged-обновлениями через промежуточные версии: сначала ставится «Обновление системы SiteUpdate», если предложено, затем перепроверка, затем ядро до промежуточной версии и только потом до текущей. Промежуточные версии фиксируются через экспертный режим (колонка «Версия до»).
7. **Соберите «что меняется»** из истории версий вендора (`dev.1c-bitrix.ru/docs/versions.php` и история изменений модулей) — это источник для оценки несовместимостей и планирования (например, в ветке 2026 удалены классы `CAll*` и JS-расширения `core_fx`/`core_ls`).
8. **Передайте полученный список** (модуль → доступная целевая версия) в анализ конфликтов и совместимости — `./03-conflict-analysis.md`.

## Рабочий сниппет/конфиг (путь в /local)

Экспертный режим поддерживает экспорт/импорт списка версий в JSON между сайтами — это удобно для синхронизации боевого и тестового стенда и для детерминированной установки в CI. Файл размещают в проекте, например `/local/updates/updates.json`:

```json
{
  "main":   { "version": "26.150.0" },
  "iblock": { "version": "26.0.0" },
  "catalog":{ "version": "25.0.0" },
  "ui":     { "version": "25.50.0" }
}
```

Применение точного списка версий (expert-mode) из CLI:

```bash
# из папки DOCUMENT_ROOT/bitrix
php bitrix.php update:versions                       # сначала смотрим, что доступно
php bitrix.php update:modules -i /local/updates/updates.json   # импорт точного списка (только при composer)
```

Структура JSON валидируется ядром (`\CUpdateExpertMode::isCorrectModulesStructure()`). `update:modules` интерактивно спрашивает подтверждение перед установкой; для неинтерактивного запуска используется стандартный флаг Symfony `-n`.

## Проверка

**Режим «только файлы» (без живого Битрикс):**
- Подтвердите наличие модуля обновлений и CLI-точки входа: файл `bitrix/modules/main/cli/bitrix.php` и команда `Update\VersionsCommand` в секции `console.commands` файла `bitrix/modules/main/.settings.php`.
- Проверьте формат экспортного JSON структурно (модуль → версия) — без обращения к серверу обновлений; фактическую валидацию делает `CUpdateExpertMode`.
- Соберите список целевых версий из истории версий вендора как офлайн-источник «что меняется».

**Режим «живой Битрикс»:**
- Откройте «Обновление платформы» — список доступных пакетов отрисовывается; проверьте блок «Ответ сервера обновлений» (дата, статус лицензии).
- Включите экспертный режим — убедитесь, что колонка «Версия до» заполнена и подсвечены зависимости.
- Откройте «Обновление решений» — отдельный список для приложений Marketplace.
- При настроенном Composer: `php bitrix.php update:versions` возвращает перечень версий, согласованный с веб-интерфейсом.

## ⚠️ Риски

- ⚠️ **Лицензия/купон.** Загрузка апдейтов требует активного права на обновления (действует 1 год после регистрации лицензии; продлевается купоном). На демо-коробке или при истёкшем периоде список доступных обновлений может быть пуст или ограничен. Это не блокирует подготовку: анализ конфликтов и планирование по истории версий выполняются всё равно (см. `./03-conflict-analysis.md`). Купон активируется на вкладке «Активация купона»; если после активации статус «лицензия истекла» — сверьте дату в блоке «Ответ сервера обновлений».
- ⚠️ **Бан лицензионного ключа.** Запуск системы обновлений с одним и тем же боевым ключом одновременно на копии и на проде приводит к блокировке ключа (ошибка `[ERROR_WRONG_CODE]`, снятие — через техподдержку вендора). Для тестового стенда используйте отдельный ключ/окружение и не запускайте проверку обновлений с боевым ключом на копии.
- **Зависимости модулей.** Некоторые обновления связаны (например, Торговый каталог ↔ Валюты). При выборочной установке нужно брать либо все связанные модули, либо ни одного — экспертный режим помогает увидеть эти связи на этапе поиска.
- **Окружение как фильтр доступности.** С 01.02.2026 ограничена поддержка PHP ниже 8.2, с 01.09.2026 — MySQL ниже 8.0. На несовместимом окружении новые версии могут не предлагаться к установке. Это учитывается при чтении списка.
- **Фиксация старых версий.** В экспертном режиме можно остановиться на зафиксированной версии, но это оставляет известные уязвимости; вендор рекомендует ставить свежие версии.

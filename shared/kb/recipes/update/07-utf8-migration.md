# Миграция кодировки cp1251 → UTF-8

## Цель

Перевести унаследованный сайт «1С-Битрикс: Управление сайтом» с однобайтовой кодировки cp1251 на UTF-8 (utf8mb4), не потеряв сериализованные данные и не получив «кракозябры». С 24.0.0 продукты поставляются только в UTF-8, и однобайтовые установки не обновляются до конвертации, поэтому миграция кодировки — обязательный предшественник дальнейших обновлений.

## Когда применять

- Сайт работает в cp1251 (`define("BX_UTF", true)` отсутствует, в `dbconn.php` есть `setlocale(... 'CP1251')`/`mb_internal_encoding("Windows-1251")`).
- Система обновлений блокирует апдейт до конвертации кодировки.
- Готовится переход на UTF-8 перед апгрейдом ядра/PHP/MySQL до актуальных версий.

## Шаги

1. **⚠️ Полный бэкап файлов и БД.** Миграция кодировки необратима без бэкапа; выполняйте на копии, перенос на прод — после проверки.
2. **Подготовьте сериализованные данные.** Байтовая конвертация «в лоб» НЕ пересчитывает префиксы длины PHP-сериализованных строк — такие колонки портятся (см. раздел про `b_bp_workflow_template`/`b_iblock_element_prop`). Эти данные конвертируйте отдельным проходом через инструменты, которые десериализуют → конвертируют → пересериализуют.
3. **Сконвертируйте файлы проекта.** `iconv -f cp1251 -t utf8` по шаблонам/компонентам/`php_interface`; сохранять БЕЗ BOM (BOM даёт лишние символы в теле составных страниц).
4. **Сконвертируйте БД напрямую через SQL** (не из админки — см. Риски): `ALTER DATABASE` + сгенерированные `ALTER TABLE ... CONVERT TO CHARACTER SET`. Целевая кодировка для актуального ядра — utf8mb4.
5. **Обновите конфиги** (`dbconn.php`, `.settings.php`, `after_connect*.php`, `php.ini`) — чеклист ниже.
6. **Уберите директиву `mbstring.func_overload`** — на границе ядра 20.100.0 её требование снято (см. ниже).
7. **Смените кодировку в Региональных настройках для КАЖДОГО языка** (`Настройки → Настройки продукта → Язык → Региональные настройки`).
8. **Пересоберите поисковый индекс** — строки старой кодировки не матчатся с UTF-8-запросами (см. ниже).
9. **Сбросьте весь кэш и перелогиньтесь**, прогоните ключевые сценарии.

## Рабочий сниппет/конфиг

**Сериализованные данные — ключевая ловушка.** Конвертация меняет кодировку байт-в-байт и НЕ пересчитывает длины PHP-сериализованных строк. Из-за этого ломаются сериализованные массивы (`a:2:{...}`): HTML/Text-свойства инфоблоков, настройки админ-отображения, шаблоны бизнес-процессов в `b_bp_workflow_template` (хранятся сериализованными И сжатыми), свойства в `b_iblock_element_prop`. Байтовый префикс длины перестаёт совпадать с числом байт UTF-8, значение остаётся неконвертированным. Дословная вендорская формулировка: шаблоны в `b_bp_workflow_template` хранятся сериализованными и сжатыми, поэтому стандартная конвертация кодировки БД их не затронет — нужен отдельный скрипт. Конвертировать такие колонки отдельным проходом через `CMain::ConvertCharsetArray()` (обёртка над iconv), который десериализует → конвертирует → пересериализует.
Источники (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=43&LESSON_ID=7495 ; https://dev.1c-bitrix.ru/api_help/main/reference/cmain/convertcharsetarray.php

```php
<?php
// Отдельный проход по сериализованной колонке: десериализация → конвертация → пересериализация.
// Запускать осознанно, на копии, с бэкапом. Пример обёртки над вендорным методом.
require_once($_SERVER['DOCUMENT_ROOT'] . '/bitrix/modules/main/include/prolog_before.php');

global $DB;
$res = $DB->Query("SELECT ID, VALUE FROM b_iblock_element_prop WHERE VALUE LIKE 'a:%'");
while ($row = $res->Fetch()) {
    $data = @unserialize($row['VALUE']);
    if ($data === false) {
        continue; // не сериализовано или повреждено — пропускаем
    }
    $converted = $GLOBALS['APPLICATION']->ConvertCharsetArray($data, 'windows-1251', 'UTF-8');
    $value = $DB->ForSql(serialize($converted));
    $DB->Query("UPDATE b_iblock_element_prop SET VALUE='" . $value . "' WHERE ID=" . (int)$row['ID']);
}
```

**Конвертация БД через SQL (рекомендация вендора).** Вендор предписывает конвертировать БД напрямую через SQL, а не из административной части:

```sql
ALTER DATABASE `db` CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
-- затем по каждой таблице (сгенерировать списком):
ALTER TABLE `b_iblock_element` CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

Collation выбирается по движку БД: `utf8mb4_0900_ai_ci` на MySQL 8.x/9.x и `utf8mb4_unicode_ci` на MariaDB 10.x/11.x.
Источники (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=43&LESSON_ID=7495 ; https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=2593

**Мастер perfmon.utf8.** В составе модуля Производительности есть мастер конвертации в UTF-8 (`perfmon`), которым можно сгенерировать и выполнить операции конвертации в рамках осознанной миграции. Запускать на копии с бэкапом; для сериализованных колонок всё равно требуется отдельный проход (см. выше).
Источник (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=43&LESSON_ID=7495

**Чеклист конфигов.**

```php
<?php
// /bitrix/php_interface/dbconn.php
define("BX_UTF", true);
// УДАЛИТЬ legacy-строки cp1251:
// setlocale(LC_ALL, 'ru_RU.CP1251');
// mb_internal_encoding("Windows-1251");
```

```php
<?php
// /bitrix/.settings.php — фрагмент
return [
    'utf_mode' => ['value' => true, 'readonly' => true],
    // ...
];
```

```php
<?php
// /bitrix/php_interface/after_connect.php и after_connect_d7.php
$DB->Query("SET NAMES 'utf8mb4'");
$DB->Query("SET collation_connection = 'utf8mb4_unicode_ci'");
```

В `php.ini`: `default_charset = "utf-8"`. Требование системы обновлений: `mbstring.internal_encoding` должен быть ПУСТЫМ (используется `default_charset`), и `default_charset` обязан совпадать с `mb_internal_encoding()` и с кодировкой инсталляции.
Источники (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=43&LESSON_ID=7495 ; https://helpdesk.bitrix24.ru/open/11473408/

**Граница `mbstring.func_overload` на ядре 20.100.0.** ДО ядра 20.100.0 для UTF-8 требовалось `func_overload=2`; С 20.100.0 эта директива более не требуется и не поддерживается — должна быть удалена (или `=0`). Унаследованный сайт с `func_overload=2`, пересекая границу 20.100.0, перестанет работать корректно, пока директива не снята.
Источник (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=2593

**Поисковый индекс.** Индекс поиска нужно пересобрать: строки старой кодировки не матчатся с UTF-8-запросами. По практике перед конвертацией модуль Поиск и Веб-аналитику переустанавливают «без сохранения данных» (чтобы избежать проблем символов ёЁ), затем переиндексируют: `Настройки → Поиск → Переиндексация`. Файл стемминга `/bitrix/modules/search/tools/ru/stemming.php` может быть пропущен автоматической конвертацией — проверить/сконвертировать вручную.
Источники (vendor): https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=43&LESSON_ID=7495 ; (по практике): https://dev.1c-bitrix.ru/community/blogs/howto/1466.php

## Проверка

**Режим «только файлы»** (без живого ядра):
- В `/bitrix/php_interface/dbconn.php` присутствует `define("BX_UTF", true);` и НЕТ строк `setlocale(... CP1251)` / `mb_internal_encoding("Windows-1251")`.
- В `/bitrix/.settings.php` — `'utf_mode' => ['value' => true, ...]`.
- В `after_connect.php`/`after_connect_d7.php` — `SET NAMES 'utf8mb4'` и нужный `collation_connection`.
- В `php.ini` — `default_charset = "utf-8"`; `mbstring.internal_encoding` пуст; нет `mbstring.func_overload=2` (для ядра ≥ 20.100.0).
- Файлы шаблонов сохранены без BOM (нет байтов `EF BB BF` в начале).

**Режим «живой Битрикс»**:
- `Настройки → Инструменты → Проверка системы` — нет ошибок кодировок таблиц/БД; кодировка БД и таблиц — utf8mb4.
- Сериализованные данные (карточки инфоблоков, шаблоны БП в `b_bp_workflow_template`) открываются без «кракозябр».
- Поиск находит русские слова (индекс пересобран); региональные настройки каждого языка — UTF-8.
- Кэш сброшен; ключевые сценарии (формы, авторизация, для магазина — заказ) проходят без артефактов кодировки.

## ⚠️ Риски

- **⚠️ Сериализованные колонки ломаются при байтовой конвертации.** `b_bp_workflow_template`, `b_iblock_element_prop`, HTML/Text-свойства инфоблоков конвертируйте отдельным проходом (десериализация → конвертация → пересериализация), а не общим `ALTER ... CONVERT`.
- **⚠️ Конвертация БД из админки не рекомендована вендором.** Вендор предписывает конвертировать БД напрямую через SQL (`ALTER DATABASE`/`ALTER TABLE`), а не из административной части. Многие community-инструкции советуют обратное — следуйте вендорской рекомендации.
- **⚠️ BOM в файлах** даёт лишние символы в теле составных страниц — сохранять сконвертированные файлы строго без BOM.
- **⚠️ `mbstring.func_overload=2` на ядре ≥ 20.100.0** ломает сайт — директиву удалить.
- **⚠️ Поисковый индекс старой кодировки** не матчит UTF-8-запросы — пересобрать обязательно; файл стемминга проверить вручную.
- **Большие таблицы** конвертируются долго и могут упираться в таймаут — выполнять на копии в окно обслуживания, при необходимости поэтапно (step-resume).

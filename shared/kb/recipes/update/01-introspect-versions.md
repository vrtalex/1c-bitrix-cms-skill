# 01. Интроспекция версий перед обновлением

## Цель

Собрать точную сводку «что стоит сейчас» перед запуском обновления
«1С-Битрикс: Управление сайтом» (ядро 26.x): версия ядра (main), версии всех
установленных модулей, список установленных приложений Marketplace, редакция
продукта и требования к окружению текущей версии (PHP/СУБД). Без этой сводки
нельзя спланировать поэтапный апдейт и проверить совместимость окружения.

## Когда применять

- Перед обновлением ядра/модулей или перед переходом через мажорную версию
  (24 → 25 → 26): нужен снимок исходного состояния для плана и для отката.
- Перед апгрейдом PHP/MySQL на сервере: сначала проверяют, что текущие версии
  ядра и модулей совместимы с целевой версией окружения.
- Перед синхронизацией версий боевого и тестового стенда (экспертный режим
  обновлений работает со списком версий модулей).
- В CI/деплое: зафиксировать версии до и после обновления.

## Шаги

1. **Версия ядра (модуль main).**
   Источник истины в файлах — `bitrix/modules/main/classes/general/version.php`:
   `define("SM_VERSION", "26.150.0")` (значение зависит от установки). На живой
   системе ту же версию отдаёт API
   `\Bitrix\Main\ModuleManager::getVersion('main')`.

2. **Редакция продукта.**
   Метаданные редакции лежат в `bitrix/.config.php`
   (`$bxProductConfig`), исходная конфигурация установки — в корневом
   `install.config` (имя продукта `<productName>`, список не-устанавливаемых
   модулей `<skipInstallModules>`, кодировка `<INSTALL_CHARSET>`). Это отвечает
   на вопрос «Бизнес / Малый бизнес / Стандарт / …» и какие модули в принципе
   не входят в редакцию.

3. **Версии всех установленных модулей.**
   Два сверяемых источника:
   - таблица БД `b_module` — что реально зарегистрировано как установленное
     (legacy: `ModuleTable` / `\Bitrix\Main\ModuleManager`);
   - файл `bitrix/modules/<id>/install/version.php` каждого модуля — какая
     версия лежит на диске (`$arModuleVersion = ['VERSION' => '...', ...]`).
   Расхождение «БД vs диск» указывает на незавершённое обновление.
   Программно версию модуля отдаёт `\Bitrix\Main\ModuleManager::getVersion('<id>')`
   (читает `modules/<id>/install/version.php`); для `main` это спецслучай —
   значение берётся из `version.php` / `SM_VERSION`. Полный перечень модулей на
   диске с разбивкой по происхождению (kernel / partner / local) даёт
   `\Bitrix\Main\ModuleManager::getModulesFromDisk()`.

4. **Различение ядровых и тиражных (партнёрских) модулей.**
   Модуль ядра идёт под именем без точки (`main`, `iblock`, `sale`).
   Тиражный/партнёрский модуль маркируется кодом вида `партнёр.модуль` — с
   точкой в id (`vendor.module`). Это формальный признак: id с точкой —
   сторонний/тиражный модуль, который обновляется отдельно от ядра (см. шаг 5)
   и проверяется на совместимость с целевым ядром отдельно.

5. **Установленные приложения Marketplace (сторонние решения).**
   Сторонние решения и модули устанавливаются отдельно от ядра. Их видно:
   - в админке: Marketplace (Настройки) → «Установленные решения»;
   - через REST-слой Marketplace (`\Bitrix\Rest\Marketplace`), если модуль
     `rest` установлен;
   - сторонние модули — те же `b_module` / `install/version.php`, но с
     вендорским префиксом id (не `bitrix.*`).
   Это важно: сторонние решения обновляются отдельно (Настройки → Marketplace →
   «Обновление решений») и чаще всего и являются причиной несовместимости при
   переходе через мажор.

6. **Требования окружения текущей версии (PHP/СУБД) и пороги версий.**
   Снять фактические версии окружения и сравнить с требованиями ядра 26.x:
   `php -v`, версия СУБД, ключевые директивы PHP (`default_charset`,
   `mbstring.func_overload`, `memory_limit`, `short_open_tag`, OPcache).
   Зафиксировать пороги, к которым ведёт обновление: начиная с 24.0.0 ядро
   работает только в UTF-8; минимальная PHP — 8.2.0 (порог 01.02.2026); MySQL
   8.0+ с кодировкой `utf8mb4` и collation `utf8mb4_0900_ai_ci` (на MariaDB —
   `utf8mb4_unicode_ci`). См. рецепт требований окружения (ссылка в «Связано»).

7. **Pre-flight проверка php.ini для самой системы обновлений.**
   Система обновлений может не запуститься, если не выполнены требования к
   `mbstring`: (1) расширение `mbstring` установлено; (2) `default_charset`
   задан в php.ini (для современных установок — `utf-8`) и совпадает с
   `mb_internal_encoding()`; (3) `mbstring.internal_encoding` ПУСТ — вместо него
   используется `default_charset`; заполненный `mbstring.internal_encoding`,
   конфликтующий с `default_charset`, блокирует обновления и должен быть удалён
   из php.ini/.htaccess. Эти три пункта проверяют ДО запуска мастера.

8. **Статус подписки на обновления.**
   Проверить право на обновления до старта (вкладка «Система обновлений»
   Главного модуля и блок «Ответ сервера обновлений» в мастере): право
   действует ограниченный срок после регистрации лицензии и продлевается
   купоном. На «Ограниченной лицензии» / по истечении подписки обновления
   (включая исправления безопасности) не загружаются, пока подписка не
   продлена. Снять этот статус нужно заранее, иначе план апдейта упрётся в
   отсутствие пакетов.

9. **Свести в один отчёт** «что стоит сейчас»: ядро + редакция + таблица
   модулей (БД/диск, kernel/partner/local) + список сторонних решений +
   окружение + статус подписки. Этот снимок — вход для плана обновления и точка
   отката.

## Рабочий сниппет/конфиг

Скрипт интроспекции в `/local` (например
`/local/tools/introspect_versions.php`). Запускается из CLI ядром Bitrix;
безопасен — только читает.

```php
<?php
// /local/tools/introspect_versions.php
// Запуск из DOCUMENT_ROOT: php local/tools/introspect_versions.php
// CLI-only: иначе публичный эндпоинт раскрывает инвентарь версий ядра/модулей любому посетителю.
if (PHP_SAPI !== 'cli') { http_response_code(404); exit; }

$_SERVER['DOCUMENT_ROOT'] = realpath(__DIR__ . '/../../');
define('NO_KEEP_STATISTIC', true);
define('NOT_CHECK_PERMISSIONS', true);
require $_SERVER['DOCUMENT_ROOT'] . '/bitrix/modules/main/include/prolog_before.php';

use Bitrix\Main\ModuleManager;

// 1. Версия ядра и редакция
echo 'Ядро (main): ' . ModuleManager::getVersion('main') . PHP_EOL;
echo 'SM_VERSION:  ' . (defined('SM_VERSION') ? SM_VERSION : '?') . PHP_EOL;

// 2. Установленные модули: версия в БД vs версия на диске
echo PHP_EOL . 'Модуль            | БД        | диск' . PHP_EOL;
foreach (ModuleManager::getInstalledModules() as $m) {
    $id  = $m['ID'];
    $db  = ModuleManager::getVersion($id) ?: '-';
    $disk = '-';
    foreach (["/local/modules/$id", "/bitrix/modules/$id"] as $base) {
        $vf = $_SERVER['DOCUMENT_ROOT'] . $base . '/install/version.php';
        if (file_exists($vf)) {
            $arModuleVersion = [];
            include $vf;
            $disk = $arModuleVersion['VERSION'] ?? '-';
            break;
        }
    }
    printf("%-17s | %-9s | %s%s", $id, $db, $disk, PHP_EOL);
}

// 3. Окружение (текущие фактические значения)
echo PHP_EOL . 'PHP: ' . PHP_VERSION . PHP_EOL;
echo 'default_charset:      ' . ini_get('default_charset') . PHP_EOL;
echo 'mbstring.func_overload: ' . ini_get('mbstring.func_overload') . PHP_EOL;
echo 'memory_limit:         ' . ini_get('memory_limit') . PHP_EOL;

// 4. Pre-flight для системы обновлений (mbstring)
echo PHP_EOL . 'mbstring установлен:        ' . (extension_loaded('mbstring') ? 'да' : 'НЕТ') . PHP_EOL;
echo 'mbstring.internal_encoding: "' . ini_get('mbstring.internal_encoding') . '" (должно быть пусто)' . PHP_EOL;
echo 'mb_internal_encoding():     ' . mb_internal_encoding() . PHP_EOL;
echo 'совпадает с default_charset: '
   . (strcasecmp(ini_get('default_charset'), mb_internal_encoding()) === 0 ? 'да' : 'НЕТ') . PHP_EOL;
```

Если в проекте настроен Composer + Symfony Console, доступные обновления
показывает встроенная команда (она же — индикатор, что апдейтер на связи):

```bash
# из DOCUMENT_ROOT/bitrix
php bitrix.php update:versions
```

Список приложений Marketplace из кода (только если установлен модуль `rest`):

```php
if (\Bitrix\Main\Loader::includeModule('rest')) {
    // см. методы \Bitrix\Rest\Marketplace\* для перечня установленных решений
}
```

## Проверка

**Режим «только файлы» (без запуска Bitrix):**
- `bitrix/modules/main/classes/general/version.php` содержит строку
  `SM_VERSION` — это версия ядра на диске.
- `bitrix/.config.php` и корневой `install.config` дают редакцию и список
  не-устанавливаемых модулей.
- В каждом `bitrix/modules/<id>/install/version.php` (и в
  `local/modules/<id>/...` для сторонних) виден `$arModuleVersion['VERSION']`.
- Сторонние модули отличаются по id-префиксу (не `bitrix.*`); тиражный/
  партнёрский модуль — id с точкой (`vendor.module`).
- В php.ini нет заполненного `mbstring.internal_encoding`, `default_charset`
  задан — pre-flight системы обновлений выполнен.

**Режим «живой Битрикс»:**
- Скрипт интроспекции печатает версию ядра, таблицу «БД / диск» по модулям и
  блок окружения; колонки «БД» и «диск» совпадают для корректно установленных
  модулей.
- `php bitrix.php update:versions` отвечает списком доступных обновлений (а не
  ошибкой связи с сервером обновлений) — апдейтер на связи.
- В админке Marketplace → «Установленные решения» список сторонних решений
  совпадает с тем, что показал скрипт по id-префиксам.
- Скрипт печатает `mbstring установлен: да`, пустой
  `mbstring.internal_encoding` и совпадение с `default_charset` — система
  обновлений готова к запуску.
- В блоке «Ответ сервера обновлений» статус подписки активен и дата
  корректна — иначе пакеты не загрузятся.

## ⚠️ Риски

- ⚠️ **Расхождение «БД vs диск» = незавершённое/прерванное обновление.** Перед
  новым апдейтом такое состояние нужно довести до конца, иначе следующий
  апдейт может пойти от неверной базовой версии. Сначала разобраться с
  расхождением, потом обновлять.
- ⚠️ **Снимок версий — обязательная часть отката.** Без зафиксированного «что
  стоит сейчас» нельзя восстановить точное исходное состояние, а восстановление
  из резервной копии теряет данные, накопленные после снятия копии (контент,
  заказы). Снимок делают вместе с полной резервной копией (файлы + БД).
- ⚠️ **Не запускать систему обновлений с боевым лицензионным ключом на копии.**
  Один ключ на копии и на бою фиксируется как несовпадение и блокирует ключ.
  Интроспекция (чтение версий) безопасна; этот риск касается шага собственно
  обновления — для теста использовать отдельное окружение/ключ.
- **Сторонние решения — основной источник несовместимости при переходе через
  мажор.** Их версии и совместимость с целевым ядром проверяют отдельно: они
  обновляются не вместе с ядром, а через «Обновление решений».
- **Окружение проверяют до апгрейда самого окружения.** Перед сменой
  PHP/MySQL на сервере сначала довести ядро и модули до версии, совместимой с
  целевым PHP/СУБД, и только потом переключать версию окружения.
- ⚠️ **Инвентарь версий — чувствительные данные; скрипт держать CLI-only.**
  Скрипт определяет `NOT_CHECK_PERMISSIONS=true` и печатает полный перечень
  версий ядра и всех модулей (БД vs диск). Доступный по HTTP, он раскрывает
  точные версии любому неаутентифицированному посетителю — это карта известных
  уязвимостей для прицельной атаки. Запускать только из CLI (в коде стоит
  проверка `PHP_SAPI !== 'cli'`); файл держать вне document root либо закрыть
  HTTP-доступ к каталогу `tools/` на веб-сервере; удалить скрипт сразу после
  снятия снимка.

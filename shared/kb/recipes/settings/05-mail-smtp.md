# 05. Почта и SMTP

## Цель

Настроить исходящую транзакционную почту сайта на ядре 26.x: выбрать
транспорт (SMTP через секцию `smtp` в `.settings.php`, либо `mail()`/sendmail
через `bxmail()`), понять очередь `b_event` и агент `CEvent::CheckEvents`
(отложенная отправка) против `SendImmediate` (синхронная), знать, где живут
почтовые шаблоны (`b_event_message`), и уметь диагностировать недоставку.
Сами тексты шаблонов и регистрация типов событий — в `../13-email-events.md`;
здесь — транспорт, очередь и диагностика.

## Когда применять

- Письма с сайта не доходят, и нужно понять, на каком уровне (очередь /
  шаблон / транспорт) теряется отправка.
- Нужно перевести отправку с системного `mail()` на внешний SMTP-сервер.
- Магазин/портал нагружен, отправка тормозит хиты — нужно вынести рассылку на
  крон.
- Разворачивается dev-копия и нужно гарантированно не выпустить письма наружу
  (см. ## ⚠️ Риски).

## Шаги

Поток одного письма: код вызывает `CEvent::Send($event, $lid, $arFields)` →
строка пишется в `b_event` с `SUCCESS_EXEC='N'` → агент
`CEvent::CheckEvents()` (= `\Bitrix\Main\Mail\EventManager::checkEvents`) на
хите/кроне разбирает очередь → находит активные шаблоны `b_event_message` →
компилирует тело и заголовки → `\Bitrix\Main\Mail\Mail::send()` → `bxmail()` →
`@mail()` либо SMTP-мейлер.

1. **Выбрать транспорт.** Без секции `smtp` письма уходят через `bxmail()` →
   системный `@mail()` (т.е. локальный sendmail/MTA). Чтобы слать через
   внешний SMTP-сервер — заполнить секцию `smtp` в `bitrix/.settings.php`
   (см. сниппет). Если `enabled=true` и заданы host+login, отправка идёт через
   `Smtp\Mailer`; иначе — `custom_mail()` (если определена) либо `@mail()`.

2. **Определиться: очередь или сразу.**
   - `CEvent::Send(...)` — отложенно: кладёт запись в `b_event`, письмо уйдёт
     на ближайшем хите с `CheckEvents` или по крону. Возвращает ID записи.
   - `CEvent::SendImmediate(...)` — синхронно: компилирует и отправляет прямо
     сейчас, в `b_event` не пишет. Возвращает код результата
     (`Y` — все шаблоны ушли, `F` — все зафейлились, `P` — частично,
     `0` — нет активного шаблона, `N` — событие пропущено).

3. **Решить про крон.** На нагруженном сайте включают крон-обработку, чтобы
   посетители не ждали отправку: `define('BX_CRONTAB_SUPPORT', true)` в
   `dbconn.php`/`.settings.php` + крон-задача дёргает
   `bitrix/modules/main/tools/cron_events.php`. Тогда `CheckEvents()` на
   обычном хите ничего не шлёт (возвращает `null`), и рассылку выполняет только
   крон.

4. **Настроить размер пачки и хранение очереди.** Опции модуля `main`
   (админка «Настройки → Настройки продукта → Настройки модулей → Главный
   модуль»): `mail_event_bulk` (по умолчанию 5) — сколько писем за один проход
   агента; `mail_event_period` (по умолчанию 14) — сколько дней хранить записи
   `b_event` (старые чистит агент `CEvent::CleanUpAgent()`). Адрес отправителя
   по умолчанию — опция `email_from` (плейсхолдер `#DEFAULT_EMAIL_FROM#`).

5. **Диагностировать недоставку** (порядок проверки — в ## Проверка).

## Рабочий сниппет (путь в /local)

SMTP-секция в `bitrix/.settings.php` (приоритетно — `/local/.settings_extra.php`,
который переопределяет основной файл и не уезжает в репозиторий с секретами):

```php
<?php
// /local/.settings_extra.php — поверх bitrix/.settings.php
return [
    'smtp' => [
        'value' => [
            'enabled'         => true,
            'host'            => 'smtp.example.com',
            'port'            => 465,
            'login'           => 'noreply@example.com',
            'password'        => '***',           // секрет, вне git
            'encryption_type' => 'smtps',         // тип шифрования (см. примечание ниже)
        ],
        'readonly' => false,
    ],
];
```

> `encryption_type` задаёт шифрование SMTP-соединения. По наблюдаемым практикам
> ядро принимает значения вида `'tls'` (STARTTLS) и `'ssl'`/`'smtps'` (прямой TLS),
> плюс пустое значение/`'smtp'` для соединения без шифрования. Точный перечень
> зависит от версии модуля `main` — при сомнениях сверьтесь с реализацией мейлера
> `\Bitrix\Main\Mail\Smtp\*` своей сборки, а не считайте набор фиксированным.

Отправка из своего кода — отложенно в очередь и синхронно:

```php
<?php
// /local/php_interface/include/notify.php

// Отложенно: вернёт ID записи b_event, письмо уйдёт на хите/кроне.
$id = CEvent::Send('FEEDBACK_FORM', SITE_ID, [
    'NAME'    => $name,
    'EMAIL'   => $email,
    'MESSAGE' => $text,
]);

// Синхронно (например, для CLI-скрипта без последующих хитов):
$result = CEvent::SendImmediate('FEEDBACK_FORM', SITE_ID, [
    'NAME'  => $name,
    'EMAIL' => $email,
]);
// $result === 'Y' — отправлено; '0' — нет активного шаблона; 'F' — ошибка.
```

Конкретный SMTP для одного письма (минуя глобальную секцию) — через
`\Bitrix\Main\Mail\Context::setSmtp(\Bitrix\Main\Mail\Smtp\Config)` и ключ
`CONTEXT` в `\Bitrix\Main\Mail\Mail::send()`.

## Выбор API

- **Тип события + шаблон** — `CEventType` / `CEventMessage` (фасады) либо ORM
  `\Bitrix\Main\Mail\Internal\EventTypeTable` / `EventMessageTable`. Это две
  поддерживаемые версии API одного механизма; для постановки письма в очередь
  фасад `CEvent::Send` — штатный публичный способ и не помечен `@deprecated`.
- **Отправка по событию** — `CEvent::Send` (отложенно) против
  `CEvent::SendImmediate` (синхронно). Эквиваленты в новой версии —
  `\Bitrix\Main\Mail\Event::send()` / `::sendImmediate()`.
- **Финальный транспорт** — `\Bitrix\Main\Mail\Mail::send()` → `bxmail()`.
  Прямой `mail()` в обход этого механизма не используют: теряются привязка к
  сайту, blacklist, отписка, SMTP-конфиг и трекинг.
- **Разбор очереди** — агент `CEvent::CheckEvents()`
  (= `\Bitrix\Main\Mail\EventManager::checkEvents`); вручную дёргать не нужно,
  он зарегистрирован как агент ядра.

## Проверка

**Режим «только файлы»** (без запуска Битрикса):

- В `bitrix/.settings.php` (или `/local/.settings_extra.php`) присутствует
  секция `smtp` с `enabled`, `host`, `login` — если планировался внешний SMTP.
- В `dbconn.php`/`.settings.php` виден `BX_CRONTAB_SUPPORT` — если ожидается
  крон-обработка; в crontab есть задача на `cron_events.php`.
- В коде отправки используется `CEvent::Send`/`SendImmediate`, а не прямой
  `mail()`.
- ⚠️ На dev-копии задана константа `ONLY_EMAIL` (см. ниже) — письма не уйдут
  реальным адресатам.

**Режим «живой Битрикс»** (диагностика недоставки, по порядку):

1. Очередь: `SELECT ID, EVENT_NAME, SUCCESS_EXEC, DATE_EXEC FROM b_event
   ORDER BY ID DESC LIMIT 20`. `SUCCESS_EXEC='N'` и старая дата → агент не
   отрабатывает (крон не настроен или отключён). `SUCCESS_EXEC='E'` →
   исключение при отправке (смотреть журнал событий).
2. Шаблон: для `EVENT_NAME` есть `b_event_message` с `ACTIVE='Y'`, привязанный
   к нужному сайту (`b_event_message_site`) и языку. Нет активного шаблона →
   `handleEvent` вернёт `'0'`, письмо не уйдёт без явной ошибки. Проверка:
   `CEventMessage::GetList('', '', ['EVENT_NAME' => 'FEEDBACK_FORM'])`.
3. Транспорт: тестовое письмо доходит до ящика. При SMTP — проверить host/port/
   шифрование и доступность сервера с хоста сайта (фаервол, аутентификация).
4. Blacklist: адрес получателя не в `Internal\BlacklistTable` — иначе письмо
   легально отфильтруется в `Mail::canSend()`.
5. Charset: у культуры сайта задан `CHARSET` — без него `handleEvent` молча
   выходит (`return '0'`).

## ⚠️ Риски

- ⚠️ **Письма реальным клиентам с dev-копии.** Тестовая площадка с боевыми
  данными разошлёт уведомления и заказы настоящим адресатам. На dev задавать
  константу `ONLY_EMAIL` в `dbconn.php` — весь исходящий e-mail
  перенаправляется на один адрес разработчика. Делать это сразу при
  развёртывании копии (см. `../setup/05-dev-setup.md`).
- ⚠️ **Секреты SMTP в репозитории.** Пароль из секции `smtp` не должен попасть
  в git: держать его в `/local/.settings_extra.php` (вне репозитория), а не в
  основном `.settings.php`.
- ⚠️ **`Send` ≠ моментальная отправка.** `CEvent::Send` только кладёт в
  `b_event`. Если агент `CheckEvents` не отрабатывает (нет хитов, крон не
  настроен), письма копятся и не уходят. Для гарантированной отправки в
  одноразовом скрипте — `SendImmediate`.
- ⚠️ **Тихая недоставка без активного шаблона.** Нет `b_event_message` с
  `ACTIVE='Y'` под нужный сайт/язык → письмо не уйдёт без ошибки в интерфейсе.
  Это самая частая причина «не приходят письма».
- **Очередь без чистки растёт.** Если агент `CleanUpAgent` не работает (крон/
  хиты отключены), `b_event` не очищается; проверять
  `mail_event_period` и факт выполнения агента.
- **SMTP с `tls`/`smtps` против порта.** Тип шифрования должен соответствовать
  порту сервера (обычно 465 — `smtps`, 587 — `tls`). Несоответствие даёт
  отказ соединения на транспорте, а не на уровне шаблона.

# Агенты и cron (CAgent, b_agent)

## Цель

Поставить фоновую/периодическую PHP-задачу в «1С-Битрикс: Управление сайтом»
(ядро 26.x): регистрация агента через `CAgent::AddAgent`, понимание контракта
функции-агента (возврат строки своего вызова) и перевод запуска с «на хитах»
на cron для production.

## Когда применять

- Нужна периодическая задача: очистка временных данных, импорт прайсов,
  пересчёт, отложенная постобработка заказов, плановые рассылки.
- Задача не должна выполняться синхронно в запросе пользователя (долгая, тяжёлая).
- Нужно понять, почему ранее поставленный агент «исчез» или «зациклился».

Если задача — реакция на событие (добавили элемент, оформили заказ), это не
агент, а обработчик события: см. `../../api-map.md`.

## Шаги

1. **Определить функцию-агента.** Это статический метод или функция, вызываемая
   строкой PHP. Метод ОБЯЗАН вернуть строку своего повторного вызова — иначе
   агент будет удалён после первого запуска (см. ## ⚠️ Риски). ⚠️ Строку вызова
   (`name`) собирать только из статических литералов в коде — она исполняется
   через `eval`; данные из запроса/БД в неё подставлять нельзя (персистентный
   RCE, см. ## ⚠️ Риски).

2. **Зарегистрировать агента.** В кастомном модуле — в `install/index.php`
   (снятие — в `uninstall` через `CAgent::RemoveAgent`). Для разовой настройки
   проекта можно вызвать `AddAgent` однократно (например, из консольного скрипта
   в `/local`), не на каждом хите.

   ```php
   CAgent::AddAgent(
       $name,            // строка PHP-вызова, напр. "MyModule\\Importer::run();"
       $module = "",     // ID модуля
       $period = "N",    // "Y" — интервал считается от планового времени (ровная сетка)
       $interval = 86400,// интервал в секундах
       $datecheck = "",  // дата первой проверки
       $active = "Y",
       $next_exec = "",  // время первого запуска (FULL datetime)
       $sort = 100
   );
   ```

3. **Выбрать тип интервала.**
   - `period = "N"` (точечный): следующий запуск = момент фактического выполнения
     + интервал. Между запусками гарантированно проходит не меньше `interval`.
   - `period = "Y"` (периодический): следующий запуск считается от планового
     времени — задача стремится попадать в ровную сетку (каждый час, каждые сутки
     в одно и то же время).

4. **Задать время первого запуска** через `next_exec` (формат FULL datetime),
   если задачу нужно привязать к конкретному часу (например, ночная очистка).

5. **Перевести запуск на cron для production** (см. ## Выбор API).

## Рабочий сниппет (путь в /local)

`/local/modules/mymodule.importer/install/index.php` (фрагмент `DoInstall`),
и сам класс агента в `/local/modules/mymodule.importer/lib/importer.php`:

```php
// lib/importer.php
namespace MyModule;

class Importer
{
    // Функция-агент: ОБЯЗАНА вернуть строку своего же вызова.
    public static function run(): string
    {
        try {
            self::doImport();
        } catch (\Throwable $e) {
            // Логируем сами: исключение внутри агента ядро перехватывает,
            // но молча — без своей записи причину сбоя не увидеть.
            \Bitrix\Main\Diag\Debug::writeToFile(
                $e->getMessage(),
                'importer-agent',
                '/local/logs/agent.log'
            );
        }
        // Возврат = перепланирование агента на NEXT_EXEC + interval.
        return __METHOD__ . '();';   // "MyModule\Importer::run();"
    }

    private static function doImport(): void
    {
        // global $USER внутри агента = null — не опираемся на текущего пользователя.
        // ... тяжёлая работа: чтение прайса, запись через ORM ...
    }
}
```

```php
// install/index.php — DoInstall()
\CAgent::AddAgent(
    "MyModule\\Importer::run();",  // та же строка, что вернёт run()
    $this->MODULE_ID,
    "N",                            // точечный: не чаще раза в interval
    3600                            // раз в час
);

// install/index.php — DoUninstall()
\CAgent::RemoveAgent("MyModule\\Importer::run();", $this->MODULE_ID);
```

## Выбор API

- **`CAgent` (`AddAgent/RemoveAgent/Update/Delete/GetList`)** — единственный
  поддерживаемый способ ставить агенты в ядре 26.x. Из двух поддерживаемых версий
  API отдельной ORM-обёртки для агентов нет; работаем с `CAgent` и таблицей
  `b_agent`.
- **Запуск «на хитах» (по умолчанию)** — `ExecuteAgents` дёргается на завершении
  обычных запросов. Подходит для разработки и сайтов с ровным трафиком, но
  добавляет работу в пользовательские запросы.
- **Запуск по cron (рекомендуется для production)** — вынести выполнение из
  пользовательских запросов:
  1. Включить опцию модуля `main`: `agents_use_crontab = Y`.
  2. Добавить в crontab вызов `bitrix/modules/main/tools/cron_events.php`
     (например, раз в минуту). Скрипт выставляет константу `BX_CRONTAB` и гоняет
     агенты вне хитов.
  - Лимиты выборки за один проход регулируются опциями `main`:
    `agents_limit` (хиты) и `agents_cron_limit` (cron).
- **Долгие задачи — только на cron, не на хит.** Импорт, рассылки, пересчёты
  каталога на хитах тормозят отдачу страниц всем посетителям.

## Проверка

**Режим «только файлы»** (без запущенного Битрикса):

- Строка `name` синтаксически валидна как PHP-вызов и заканчивается `;`.
- Функция-агент возвращает строку, ТОЧНО совпадающую с зарегистрированной в
  `AddAgent` (включая экранирование `\\` для неймспейса в строковом литерале).
- В `install/index.php` есть парная пара `AddAgent` (install) / `RemoveAgent`
  (uninstall).
- Внутри агента нет опоры на `global $USER` как на текущего пользователя.

**Режим «живой Битрикс»:**

- Агент виден в админке: Настройки → Инструменты → Агенты (или
  `CAgent::GetList` по фильтру `MODULE_ID`).
- После выполнения поле `NEXT_EXEC` в `b_agent` сдвинулось вперёд на `interval`
  (агент перепланировался, а не удалился).
- Для cron: запуск `php -f bitrix/modules/main/tools/cron_events.php` из консоли
  отрабатывает без ошибок; в логе агента (`/local/logs/agent.log`) нет
  необработанных исключений.
- Опция `agents_use_crontab=Y` установлена; строка `cron_events.php` есть в
  crontab.

## ⚠️ Риски

- **Пустая строка в возврате = удаление агента.** Если функция-агент вернула
  пустую строку (или забыли `return`), ядро удаляет запись из `b_agent`. Симптом:
  «агент исчез после первого запуска». Всегда возвращайте строку своего вызова.
- **`name` исполняется через `eval`.** Это исполняемый PHP-код. Опечатка или
  отсутствие `;` приводит к фатальной ошибке; агент уходит в повторные попытки и
  может зацикливаться на ретраях. Тщательно проверяйте строку вызова.
- ⚠️ **`b_agent.NAME` — это персистентный RCE-вектор.** Поле `NAME` агента
  исполняется через `eval` на каждом тике cron (`cron_events.php`) и на хитах.
  НИКОГДА не стройте строку агента из внешних или БД-данных (ввод пользователя,
  значения из других таблиц, параметры запроса) — только из статических
  литералов в коде. Тот, кто может писать в `b_agent` (через SQL-инъекцию или
  избыточно-привилегированную форму/обработчик), получает персистентное
  выполнение произвольного PHP-кода на каждом запуске агентов. То же относится к
  аргументу `name` в `CAgent::AddAgent` — передавайте туда только доверенную
  константную строку, а не собранную из ввода. Доступ на запись в `b_agent` и в
  форму регистрации агентов выдавайте по минимуму.
- **Долгие задачи на хитах замедляют сайт.** Тяжёлые агенты «на хитах» добавляют
  время к пользовательским запросам. Для production переводите на cron.
- **Исключения внутри агента ядро перехватывает молча.** Без собственного
  логирования причину сбоя не видно. Оборачивайте тело в try/catch с записью в лог.
- **`global $USER` внутри агента = null.** Нельзя полагаться на текущего
  пользователя и его права; передавайте `user_id`/контекст явно при необходимости.
- **Несовпадение строки возврата и регистрации.** Если строка из `return` не
  совпадает с зарегистрированной (например, разное экранирование неймспейса),
  перепланирование/поиск агента работает не так, как ожидается.

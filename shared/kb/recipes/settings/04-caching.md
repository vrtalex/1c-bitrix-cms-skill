# 04. Кэширование: уровни, теги, композит

## Цель

Выбрать и настроить подходящий уровень кэша «1С-Битрикс: Управление сайтом» (ядро 26.x) для конкретной задачи: автокэш компонентов, кэш данных, тегированную инвалидацию, управляемый кэш таблиц БД или композитный кэш страницы целиком — и корректно сбрасывать кэш при деплое контента.

## Когда применять

- Дорогая выборка (ORM, `CIBlockElement`, агрегаты) повторяется на хитах — нужен кэш данных с TTL.
- Контентный блок (новости, каталог) должен сбрасываться автоматически при изменении инфоблока — нужен тегированный кэш.
- Нужно ускорить отдачу лендинга/витрины целиком — нужен композит со «статичными» зонами и динамическими фреймами.
- При деплое контента старый кэш «прилипает» — нужна точечная инвалидация по тегу вместо полного сброса.
- На проде падает производительность — стоит перевести движок кэша на memcached/redis в `.settings.php`.

## Шаги

1. **Определите уровень.** Тяжёлая выборка → кэш данных. Контент инфоблока → кэш данных + теги. Готовый компонент (`bitrix:news`, `bitrix:catalog`) → автокэш компонента (`CACHE_TYPE='A'`). Страница целиком → композит.
2. **Для кэша данных** оберните расчёт в `\Bitrix\Main\Data\Cache` (`initCache` → `startDataCache` → `endDataCache`).
3. **Для авто-инвалидации** внутри области кэша откройте `getTaggedCache()->startTagCache()`, повесьте `registerTag('iblock_id_X')`, закройте `endTagCache()`.
4. **Для компонентов** задайте в параметрах `CACHE_TYPE='A'` и большой `CACHE_TIME`; стандартные компоненты вешают теги инфоблоков из коробки.
5. **Настройте движок** кэша под нагрузку в `bitrix/.settings.php` (`filesystem`/`memcache`/`redis`).
6. **При деплое** сбрасывайте точечно по тегу: `getTaggedCache()->clearByTag('iblock_id_X')`.

## Рабочий сниппет (путь в /local)

`/local/php_interface/include/cache_block.php` — кэш выборки с тегированной инвалидацией по инфоблоку:

```php
<?php
// Кэш данных + тег инфоблока: при изменении ИБ область сбросится автоматически.
function getCachedItems(int $iblockId, int $ttl = 3600): array
{
    $cache = \Bitrix\Main\Data\Cache::createInstance();
    $cacheId = 'items_' . $iblockId;
    $cacheDir = '/items/' . $iblockId;

    if ($cache->initCache($ttl, $cacheId, $cacheDir)) {
        return $cache->getVars()['items'] ?? [];
    }

    if ($cache->startDataCache($ttl, $cacheId, $cacheDir)) {
        $taggedCache = \Bitrix\Main\Application::getInstance()->getTaggedCache();
        $taggedCache->startTagCache($cacheDir);

        $items = [];
        $res = \CIBlockElement::GetList([], ['IBLOCK_ID' => $iblockId, 'ACTIVE' => 'Y']);
        while ($el = $res->GetNext()) {
            $items[] = $el;
        }

        // Тег: любое изменение элементов ИБ инвалидирует эту область.
        $taggedCache->registerTag('iblock_id_' . $iblockId);
        $taggedCache->endTagCache();

        $cache->endDataCache(['items' => $items]);
        return $items;
    }

    return [];
}

// Точечный сброс после массовой загрузки контента (деплой):
// \Bitrix\Main\Application::getInstance()->getTaggedCache()->clearByTag('iblock_id_5');
```

Настройка движка кэша в `bitrix/.settings.php` (секция `cache`) — выбор хранилища под нагрузку:

```php
// bitrix/.settings.php
'cache' => [
    'value' => [
        'type' => 'memcache',   // 'filesystem' (по умолчанию) | 'memcache' | 'redis'
        'memcache' => [
            'host' => '127.0.0.1',
            'port' => '11211',
        ],
        'sid'  => $_SERVER['DOCUMENT_ROOT'] . '#01',
    ],
    'readonly' => false,
],
```

> Движок `memcache` (расширение `php-memcache`) — legacy. На современных сборках
> предпочтительны `memcached` (расширение `php-memcached`) и `redis`: для них в
> секции `cache` задаётся свой подмассив (`'memcached' => [...]` / `'redis' => [...]`)
> с `host`/`port`. Указанный `type` должен совпадать с именем подмассива.

## Выбор API

В ядре доступны две поддерживаемые версии API. Для нового кода — D7; legacy-обёртки остаются рабочими.

| Задача | D7 (предпочтительно) | Legacy (поддерживается) |
| --- | --- | --- |
| Кэш данных | `\Bitrix\Main\Data\Cache` (`createInstance`, `initCache`/`startDataCache`/`endDataCache`) | `CPHPCache` (`InitCache`/`StartDataCache`/`EndDataCache`) |
| Управляемый кэш таблиц | `Application::getInstance()->getManagedCache()` | — |
| Тегированный кэш | `Application::getInstance()->getTaggedCache()` (`startTagCache`/`registerTag`/`endTagCache`/`clearByTag`) | — |
| HTML-кэш страницы | `\Bitrix\Main\Composite\*` | `CPageCache` |
| Полный сброс файлового кэша | `Cache::cleanDir(...)` | `BXClearCache(true)` |

Уровни и назначение:
- **Автокэш компонентов** (`CACHE_TYPE='A'`) — самый простой; ядро само решает кэшировать ли, готовые компоненты вешают теги инфоблоков сами.
- **Кэш данных** (`\Bitrix\Main\Data\Cache`) — для произвольных выборок; TTL + ручной ключ; движок берётся из `.settings.php` (`createInstance`).
- **Тегированный кэш** (`getTaggedCache`) — поверх кэша данных; инвалидация по тегу (`b_cache_tag`), не зная путей; именно он сбрасывает HTML-кэш компонентов при изменении инфоблоков/разделов.
- **Управляемый кэш** (`getManagedCache`) — привязан к таблице БД (`$tableId`); при изменении таблицы вся область инвалидируется. `set()` копит в памяти, реальная запись — в `finalize()` в конце хита.
- **Композит** (`\Bitrix\Main\Composite\*`) — HTML всей страницы в хранилище; динамика выносится в `StaticArea` (фреймы), подгружается AJAX.

## Проверка

**Режим «только файлы» (без запущенного Битрикса):**
- Сниппет лежит в `/local/php_interface/...`, не трогает ядро.
- `php -l` проходит без ошибок синтаксиса.
- В `bitrix/.settings.php` секция `cache` валидна: `type` — одно из `filesystem`/`memcache`/`redis`; для memcache/redis заданы `host`/`port`.
- В коде кэша данных есть полная триада `initCache` → `startDataCache` → `endDataCache`; теги обёрнуты в `startTagCache`/`endTagCache`.

**Режим «живой Битрикс»:**
- Проверьте реально работающий движок: `\Bitrix\Main\Data\Cache::createInstance()->getCacheEngineType()` — не должно быть `none` (при недоступности memcached/redis ядро молча переключается на «пустой» движок, и кэш фактически не сохраняется).
- Первый запрос наполняет кэш, повторный читает из него (проверьте по логам запросов БД или таймингу).
- После `clearByTag('iblock_id_X')` область пересобирается на следующем хите.
- Изменение элемента инфоблока через админку сбрасывает связанный тегированный кэш автоматически.
- Для композита: страница отдаётся из HTML-кэша, динамические зоны (корзина, авторизация) дозагружаются фреймами.

## ⚠️ Риски

- ⚠️ **Композит минует PHP.** Страница из композитного кэша отдаётся без запуска прикладного кода. Любые проверки прав, A/B-логика, персонализация вне `StaticArea` не выполнятся — это утечка чужого или просроченного по TTL контента посетителю. Всю динамику и проверки доступа выносите в динамические фреймы.
- ⚠️ **Один «голос» выключает композит для всей страницы.** Компонент с `frameMode=false` отдаёт «отрицательный голос», и композит для страницы отключается. Оборачивайте динамику в `StaticArea`, а не отключайте `frameMode`.
- ⚠️ **Движок кэша может молча стать пустым.** При недоступности memcached/redis ядро переключается на «пустой» движок: кэш «как будто работает», но ничего не хранит и каждый хит бьёт в БД. После смены движка проверяйте `getCacheEngineType()`.
- Не тегируйте большие, часто меняющиеся массивы: каждый тег пишется в `b_cache_tag` (с фильтром по `SITE_ID`), частая инвалидация раздувает таблицу и нагружает запись. Для горячих данных подбирайте TTL без тегов.
- Управляемый кэш: `set()` пишет на диск только в `finalize()` в конце хита. При раннем `die()`/`exit` данные теряются — используйте `setImmediate()` для немедленной записи.
- Тегированный кэш требует `SITE_ID`. В CLI/агенте `SITE_ID` может быть не определён — теги не сматчатся, инвалидация пройдёт мимо.
- Не используйте полный сброс (`BXClearCache(true)`) при штатном деплое контента — это снимает весь кэш сайта и даёт пик нагрузки. Сбрасывайте точечно по тегу.

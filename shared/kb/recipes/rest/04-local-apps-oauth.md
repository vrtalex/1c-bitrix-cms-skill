# Локальные приложения и OAuth на коробке БУС

## Цель

Выбрать правильный способ авторизации REST на коробке «1С-Битрикс: Управление
сайтом» (ядро 26.x) и понять границы каждого: входящий вебхук, локальное
приложение, OAuth 2.0. Отдельно — честно разобраться, что из OAuth-сценариев
работает на изолированной коробке, а что завязано на облачные сервисы вендора.

## Когда применять

- Нужно дать внешнему сервису доступ к данным сайта (свой сайт ↔ свой бэкенд).
- Выбираете между вебхуком и полноценным приложением и не уверены, какой scope
  реально доступен на голой БУС (без модуля `bitrix24`/CRM).
- Планируете локальное приложение с UI-встройкой (placement) и хотите заранее
  знать, потребуется ли выход коробки в интернет.
- Коробка изолирована (нет доступа к `oauth.bitrix.info`), и нужно понять,
  можно ли вообще использовать OAuth.

## Шаги

1. **Убедитесь, что модуль `rest` установлен.** На БУС за REST отвечает отдельный
   модуль `rest` (в дистрибутиве 26.x — версия 26.0.0). С версии 18 модуль входит
   в «Управление сайтом», поэтому работает без модуля `bitrix24` и без `crm`.
   Ставится в админке: «Настройки → Настройки продукта → Модули».

2. **Выберите способ авторизации.** На коробке доступны три механизма (точка
   расширения — событие `OnRestCheckAuth`, цепочка провайдеров OAuth → APAuth →
   SessionAuth):
   - **APAuth (вебхуки)** — самый простой путь для внутренних интеграций. Требует
     HTTPS. Подробности — в `./01-incoming-webhooks.md`.
   - **OAuth 2.0** — для локальных и Маркет-приложений; полный протокол с
     access/refresh-токенами.
   - **SessionAuth** — вызовы из браузера в рамках сессии (AJAX из админки/публички).

3. **Определите, нужен ли вам вообще OAuth.** Сравните по таблице:

   | Критерий | Входящий вебхук (APAuth) | Локальное приложение (OAuth) |
   |---|---|---|
   | Где создаётся | на самом портале | на портале (`STATUS_LOCAL`) |
   | Токен | бессрочный ключ | access/refresh, нужно продлевать |
   | UI-встройка (placement) | нет | да |
   | Своя серверная логика | вызовы методов | да (`ONAPPINSTALL`, события) |
   | Зависимость от облака | нет | да, см. шаг 5 |
   | Когда выбирать | свой сайт ↔ свой сервер | приложение с UI/жизненным циклом |

   Если задача — «дёрнуть пару REST-методов из своего бэкенда», берите вебхук:
   меньше движущихся частей и нет внешних зависимостей.

4. **Проверьте доступные scope на вашей сборке.** Набор scope определяется
   установленными модулями. На голой БУС нет `crm`, `tasks`, `im`, `telephony` —
   значит и их scope/методов нет. Доступны общесистемные (`user`, `profile`),
   работа с инфоблоками (scope `entity` — динамические сущности поверх
   инфоблоков) и собственные scope вашего модуля. ⚠️ Не хардкодьте облачный набор
   scope — модули в коробке могут отставать от облака. Запросите реальный список
   методами `scope` и `methods` у конкретного портала.

5. **Локальное приложение и OAuth — проверьте связь с облаком.** Серверное
   локальное приложение без UI реализует полный OAuth 2.0: в коде заполняются
   `C_REST_CLIENT_ID` и `C_REST_CLIENT_SECRET`, токены сохраняются при установке
   (обработчик `ONAPPINSTALL`), приложение само продлевает их через `refresh_token`.
   ⚠️ В стандартном потоке access/refresh-токены выдаёт **внешний сервер
   авторизации** `https://oauth.bitrix.info/oauth/token/` (в новых доках —
   `oauth.bitrix24.tech`). Это означает: «классический» OAuth Маркет/локальных
   приложений **требует выхода коробки в интернет** к этому серверу. На полностью
   изолированной коробке без интернета путь «из коробки» не работает. Этот пункт
   подлежит проверке на конкретной сборке 26.x — см. раздел «Открытые вопросы»
   ниже и официальную доку.

6. **Для изолированной коробки — свой провайдер авторизации.** Вендор официально
   разрешает изолированной коробке реализовать собственный провайдер авторизации
   взамен `oauth.bitrix24.tech`. Перехватываются три точки:
   - **валидатор** (`onRestCheckAuth`) — локальная проверка токена по своему префиксу;
   - **провайдер авторизации** — класс наследует штатный OAuth-провайдер,
     реализует `AuthProviderInterface`, метод `get()` выдаёт токены сам, без
     обращения вовне;
   - **провайдер событий** (`EventProvider`) — шлёт колбэки приложению напрямую
     (HTTP POST), минуя внешние сервисы.

   Обработчики регистрируются с приоритетом 90 (чтобы перехватить раньше штатного
   OAuth); код запускается на портале один раз. ⚠️ При таком подходе вопросы
   безопасности и управления авторизацией вы решаете сами — это прямое
   предупреждение вендора.

7. **Плейсменты (placement).** Если приложению нужно встроить свой iframe-UI в
   интерфейс системы — используются плейсменты (таблица `b_rest_placement`).
   Регистрируются через ключ `CRestUtil::PLACEMENTS` в описании scope или REST-методом
   `placement.bind`. ⚠️ Плейсменты требуют «контекста приложения» и вебхукам
   недоступны — это ещё одна причина выбрать локальное приложение, а не вебхук.

## Рабочий сниппет/конфиг (путь в /local)

Свой провайдер авторизации для изолированной коробки. Регистрация выполняется
один раз; разместить можно, например, в `/local/php_interface/init.php`.

```php
// /local/php_interface/init.php
use Bitrix\Main\EventManager;

$em = EventManager::getInstance();

// 1. Локальный валидатор токенов (перехватываем раньше штатного OAuth — приоритет 90)
$em->addEventHandler(
    'rest',
    'onRestCheckAuth',
    [\Local\Rest\CustomAuth::class, 'onRestCheckAuth'],
    false,
    90
);
```

```php
// /local/modules/local.rest/lib/customauth.php
namespace Local\Rest;

class CustomAuth
{
    public const TOKEN_PREFIX = 'loc_';

    /**
     * Сигнатура совпадает со штатной цепочкой провайдеров REST:
     * вернуть null  — «не мой тип токена, пусть проверяют дальше»,
     *          false — токен мой, но невалиден (плюс заполнить $error),
     *          array — данные валидного токена (USER_ID, scope и т.п.).
     */
    public static function onRestCheckAuth(array $query, $scope, &$res): ?bool
    {
        $token = $query['auth'] ?? '';
        // ⚠️ str_starts_with — это ТОЛЬКО маршрутизация «мой/не мой тип токена»,
        //    НЕ аутентификация. Проверка валидности — ниже, в resolveLocalToken().
        if (!str_starts_with($token, self::TOKEN_PREFIX)) {
            return null; // не наш префикс — отдаём следующему провайдеру
        }

        $tokenInfo = self::resolveLocalToken($token); // своя проверка, без обращения вовне
        if ($tokenInfo === null) {
            $res = ['error' => 'invalid_token', 'error_description' => 'Local token invalid'];
            return false;
        }

        $res = [
            'user_id' => $tokenInfo['USER_ID'],
            'scope'   => $tokenInfo['SCOPE'],
        ];
        return true;
    }

    /**
     * Минимальная БЕЗОПАСНАЯ проверка локального токена.
     * ⚠️ В хранилище держим ТОЛЬКО хэш токена (никогда не сырой), сравниваем по
     * постоянному времени, проверяем срок (TTL), активность и scope.
     */
    private static function resolveLocalToken(string $token): ?array
    {
        // Ищем по ХЭШУ, а не по сырому значению (sha256 — детерминированный
        // хэш для поиска; сам токен в БД не хранится).
        $tokenHash = hash('sha256', $token);

        // Своя таблица b_local_rest_token: TOKEN_HASH, USER_ID, SCOPE,
        // EXPIRES (datetime), ACTIVE ('Y'/'N').
        $row = \Bitrix\Main\Application::getConnection()
            ->query(
                "SELECT USER_ID, SCOPE, EXPIRES, ACTIVE, TOKEN_HASH "
                . "FROM b_local_rest_token "
                . "WHERE TOKEN_HASH = '" . $tokenHash . "'"
            )
            ->fetch();

        if (!$row) {
            return null;
        }

        // Сравнение хэшей по постоянному времени (защита от тайминг-атак).
        if (!hash_equals((string)$row['TOKEN_HASH'], $tokenHash)) {
            return null;
        }
        // Неактивен / отозван.
        if ($row['ACTIVE'] !== 'Y') {
            return null;
        }
        // TTL: истёкший токен отвергаем.
        if (!empty($row['EXPIRES']) && strtotime($row['EXPIRES']) < time()) {
            return null;
        }

        return [
            'USER_ID' => (int)$row['USER_ID'],
            'SCOPE'   => (string)$row['SCOPE'],
        ];
    }
}
```

> ⚠️ Сниппет показывает контракт перехвата; на конкретной сборке 26.x точную
> сигнатуру `onRestCheckAuth` и набор полей результата сверяйте по официальной
> документации модуля `rest` (custom auth provider).

## Проверка

**Режим «только файлы» (без живого Битрикса):**
- В `/local/php_interface/init.php` есть регистрация обработчика `onRestCheckAuth`
  с приоритетом 90; класс провайдера существует и автозагружается.
- Константы `C_REST_CLIENT_ID` / `C_REST_CLIENT_SECRET` (если делаете локальное
  приложение по SDK CRest) заданы и не закоммичены в публичный репозиторий.
- В коде интеграции нет хардкода облачного набора scope — список запрашивается
  у портала.

**Режим «живой Битрикс»:**
- Метод `scope` (`/rest/.../scope`) возвращает реальный список scope сборки —
  сверьте, что нужные вам scope присутствуют (на голой БУС не ждите `crm`/`tasks`).
- Метод `methods` отдаёт доступные методы для вашего scope.
- Для локального приложения с OAuth: установка проходит, `ONAPPINSTALL`
  отрабатывает, токены сохранились, обновление по `refresh_token` проходит.
- Для изолированной коробки: запрос с локальным токеном (префикс `loc_`)
  авторизуется вашим провайдером; запрос без подключения к `oauth.bitrix.info`
  не падает с ошибкой обращения вовне.

## ⚠️ Риски

- **OAuth-flow завязан на облако.** Регистрация и выдача токенов Маркет/локальных
  приложений в стандартном потоке идут через внешний сервер `oauth.bitrix.info`
  (`oauth.bitrix24.tech`). На изолированной коробке без интернета этот путь не
  работает — закладывайте либо вебхуки, либо свой провайдер авторизации.
- ⚠️ **Custom auth provider — ваша зона ответственности по безопасности.** Заменив
  штатный OAuth, вы сами отвечаете за валидацию токенов, их TTL и защиту. Минимум:
  храните в БД ТОЛЬКО хэш токена (никогда не сырой), сравнивайте по постоянному
  времени (`hash_equals`), проверяйте срок (TTL), активность и scope, отклоняйте
  неактивные/истёкшие. `str_starts_with` по префиксу — это лишь маршрутизация
  «мой/не мой тип токена», а НЕ аутентификация. Ошибка в проверке токена =
  неконтролируемый доступ к данным сайта.
- **Утечка ключа вебхука = доступ к данным.** Ключ вебхука бессрочный и не имеет
  срока действия; компрометация URL открывает доступ ко всем его scope. Вебхуки —
  только для внутренних интеграций.
- **Scope зависят от модулей.** Облачный набор методов на голой БУС недоступен;
  не проектируйте интеграцию под методы, которых на сборке физически нет.
- **Плейсменты недоступны вебхукам.** Если нужна UI-встройка — это локальное
  приложение, а не вебхук.

### Открытые вопросы (уточнять на стенде)

- Работает ли OAuth-flow для локальных приложений на изолированной БУС без
  подключения к `oauth.bitrix.info`, и что именно требует облачной регистрации —
  проверять на стенде 26.x по официальной доке.
- Точный список scope, реально доступных на голой БУС (без `crm`/`intranet`),
  вендор отдельно не публикует — определять эмпирически методами `scope`/`methods`.

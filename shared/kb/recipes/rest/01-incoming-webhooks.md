# rest/01. Входящие вебхуки и приём данных от внешних систем

## Цель

Дать внешней системе вызывать REST-методы коробки БУС с правами конкретного
пользователя — через **входящий вебхук** (авторизация APAuth). И, для случая
«внешняя система шлёт произвольный POST», принять этот POST на свою страницу/
контроллер в `/local` с проверкой токена и (где применимо) `check_bitrix_sessid()`.

Модуль `rest` ставится на чистую БУС **без модуля `bitrix24`** и без CRM —
REST работает на «Управлении сайтом» самостоятельно.

## Когда применять

- Внешний сервис (ERP, 1С, личный микросервис) должен читать/писать данные сайта
  по понятному URL без полноценного OAuth-приложения — это самый короткий путь.
- Интеграция **внутренняя** (свой сайт ↔ свой сервер): вебхук вызывает API
  с правами пользователя, создавшего вебхук, а ключ бессрочный.
- Внешняя система не умеет вызывать REST-методы Битрикса и просто шлёт свой JSON/
  form POST — тогда принимаем его собственным эндпоинтом в `/local` (вариант B).
- Нужны методы своих сущностей: для данных инфоблока через REST у инфоблока должен
  быть включён флаг «Разрешён доступ через REST».

Не подходит для публичных интеграций со сторонними потребителями: утечка URL
вебхука = доступ к данным. Для marketplace/локальных OAuth-приложений см. вариант
с OAuth (внешний сервер авторизации, отдельная тема).

## Шаги

### Вариант A. Штатный входящий вебхук (APAuth)

1. Убедиться, что установлен модуль `rest` (Настройки → Настройки продукта →
   Модули) и сайт работает по **HTTPS** — по умолчанию APAuth без HTTPS не
   авторизует (ошибка `Https required`). Для внутренних/dev-контуров требование
   можно ослабить настройкой `REST_APAUTH_ALLOW_HTTP` (см. ## ⚠️ Риски).
2. На голой БУС интерфейса «Разработчикам» может не быть — поднять эндпоинт REST
   вручную: разместить компонент `bitrix:rest.server` (после установки модуля
   автоматически создаётся ЧПУ `/rest/`) либо страницу-обёртку и правило в
   `urlrewrite.php`. Открыть страницу, выбрать «Входящий вебхук», отметить
   флажками **права доступа (scope)**, сохранить — система выдаст код и пример URL.
3. Программная альтернатива созданию: записать пароль в `b_rest_ap`
   (`Bitrix\Rest\APAuth\PasswordTable::add()`) и права в `PermissionTable`.
   Скоупы вебхука ограничены белым списком и зависят от установленных модулей.
4. Вызов вебхука: `https://<домен>/rest/<user_id>/<токен>/<метод>[.json][?params]`
   (URL-шаблон `#aplogin#/#ap#/#method#`). Пример:
   `.../rest/1/173glortu42lvpju/user.current.json`.
5. Доступный набор scope определяется модулями: на чистой БУС нет `crm`/`tasks`/
   `im` и их методов. При настройке запрашивать реальный список у портала
   (методы `scope` и `methods`), а не хардкодить облачный набор — версии модулей
   в коробке могут отставать.

### Вариант B. Приём произвольного POST на свой эндпоинт в /local

1. Создать страницу-контроллер в `/local` (см. сниппет), подключающую ядро.
2. Проверить общий **секрет токена** (заголовок или параметр) — сравнением
   с эталоном, который хранится вне VCS (см. ниже).
3. Для запросов, идущих из браузера в рамках сессии (AJAX своей же админки/
   публички), дополнительно проверять `check_bitrix_sessid()`. Для серверных
   запросов извне sessid недоступен — там работает только проверка токена.
4. Валидировать и типизировать входные данные до использования (числа — к `(int)`/
   `(float)`, строки — экранировать перед выводом/SQL), писать ответ в JSON.

## Рабочий сниппет

Файл: `/local/rest_in/index.php` — приём произвольного POST (вариант B).

```php
<?php
// Приём внешнего POST на коробке БУС. Без UI, только серверная логика.
require($_SERVER['DOCUMENT_ROOT'] . '/bitrix/modules/main/include/prolog_before.php');

use Bitrix\Main\Context;
use Bitrix\Main\Web\Json;

header('Content-Type: application/json; charset=UTF-8');

$request = Context::getCurrent()->getRequest();

// 1. Только POST.
if (!$request->isPost()) {
	http_response_code(405);
	echo Json::encode(['error' => 'method_not_allowed']);
	die();
}

// 2. Секрет токена — из конфига вне VCS (см. ниже), сравнение по постоянному времени.
//    ⚠️ ТОЛЬКО из заголовка: секрет в query-string попадает в access-логи,
//    Referer и историю прокси. Ветку $request->get('token') не добавлять.
$expected = (string)(getenv('REST_IN_TOKEN') ?: '');
$got = (string)($request->getHeader('X-Webhook-Token') ?? '');
if ($expected === '' || !hash_equals($expected, $got)) {
	http_response_code(401);
	echo Json::encode(['error' => 'unauthorized']);
	die();
}

// 3. Тело: JSON или form-data. Невалидный JSON — отказ.
$raw = $request->getInput();
$data = $raw !== '' ? json_decode($raw, true) : $request->getPostList()->toArray();
if (!is_array($data)) {
	http_response_code(400);
	echo Json::encode(['error' => 'bad_request']);
	die();
}

// 4. Валидация, типизация и ЯВНЫЙ whitelist полей ДО использования.
//    ⚠️ Никакого mass-assignment: распарсенный JSON НЕ передавать целиком в
//    Add/Update. Берём только разрешённые поля и приводим каждое к типу.
$externalId = (int)($data['id'] ?? 0);
$name = trim((string)($data['name'] ?? ''));
if ($externalId <= 0 || $name === '') {
	http_response_code(422);
	echo Json::encode(['error' => 'validation', 'fields' => ['id', 'name']]);
	die();
}

// Собираем ТОЛЬКО whitelisted-поля с приведением типов — это и есть полезная
// нагрузка для прикладной логики. Любые лишние ключи из $data игнорируются.
$safe = [
	'XML_ID' => (string)$externalId,
	'NAME'   => $name,
	'ACTIVE' => (($data['active'] ?? '') === true || ($data['active'] ?? '') === 'Y') ? 'Y' : 'N',
];

// 5. Прикладная логика (пример — запись через ORM/CIBlockElement, опуская детали).
//    В Add/Update уходит только $safe, а не сырой $data; строки экранировать при
//    выводе в HTML, числа уже приведены к int.

// 6. Формат ответа.
echo Json::encode(['status' => 'ok', 'id' => $externalId]);

require($_SERVER['DOCUMENT_ROOT'] . '/bitrix/modules/main/include/epilog_after.php');
```

Если эндпоинт вызывается из браузера в рамках сессии (AJAX своей админки/публички),
добавить перед прикладной логикой проверку токена сессии (CSRF):

```php
// Только для запросов в рамках сессии (form: bitrix_sessid_post(), link: bitrix_sessid_get()).
if (!check_bitrix_sessid()) {
	http_response_code(403);
	echo Json::encode(['error' => 'invalid_sessid']);
	die();
}
```

Секрет — вне VCS. Файл `/local/php_interface/.rest_in.php` (в `.gitignore`):

```php
<?php
// Не коммитить. Альтернатива — переменная окружения REST_IN_TOKEN на сервере.
putenv('REST_IN_TOKEN=' . 'сгенерированный_случайный_токен');
```

Подключать его из `/local/php_interface/init.php`:

```php
$secretFile = __DIR__ . '/.rest_in.php';
if (file_exists($secretFile)) {
	require $secretFile;
}
```

## Проверка

**Режим «только файлы» (без запущенного Битрикса):**

```bash
# эндпоинт и секрет на месте, секрет исключён из VCS
ls /path/to/site/local/rest_in/index.php
grep -n "rest_in\|.rest_in.php" /path/to/site/.gitignore

# нет синтаксических ошибок
php -l /path/to/site/local/rest_in/index.php

# проверка токена и валидация действительно есть в коде
grep -n "hash_equals\|check_bitrix_sessid\|(int)\|http_response_code" \
  /path/to/site/local/rest_in/index.php
```

**Режим «живой Битрикс»:**

```bash
# B: верный токен — 200, ответ JSON
curl -sS -X POST https://<домен>/local/rest_in/index.php \
  -H 'X-Webhook-Token: <токен>' -H 'Content-Type: application/json' \
  -d '{"id":42,"name":"Acme"}'

# B: без токена — 401
curl -sS -o /dev/null -w '%{http_code}\n' -X POST \
  https://<домен>/local/rest_in/index.php -d '{"id":42}'

# A: штатный вебхук отвечает (метод из выданных вебхуку scope)
curl -sS 'https://<домен>/rest/<user_id>/<токен>/user.current.json'

# A: какие scope реально доступны на этой сборке
curl -sS 'https://<домен>/rest/<user_id>/<токен>/scope.json'
```

## ⚠️ Риски

- ⚠️ **Секрет токена — вне VCS.** Токен вебхука и общий секрет эндпоинта не
  коммитить (`.gitignore` + переменная окружения или файл вне репозитория).
  Ключ вебхука бессрочный — утечка URL = постоянный доступ к данным с правами
  его владельца. Применять вебхуки только для внутренних интеграций.
- ⚠️ **Секрет — только в заголовке, не в query-string.** Токен в URL
  (`?token=...`) попадает в access-логи веб-сервера, заголовок `Referer` и
  историю прокси/CDN — то есть утекает. Принимать секрет исключительно из
  заголовка (`X-Webhook-Token`); ветку чтения из query не добавлять.
- ⚠️ **Валидация и whitelist полей обязательны (нет mass-assignment).** Данные из
  HTTP-запроса по умолчанию небезопасны. Распарсенный JSON НЕЛЬЗЯ передавать
  целиком в `Add`/`Update` — соберите отдельный массив только из разрешённых
  полей, каждое приведя к типу (`(int)`/`(float)`/`(string)`). Строки
  экранировать перед выводом в HTML (`htmlspecialcharsbx()`) и перед SQL;
  невалидный JSON/недостающие поля — отклонять (4xx). Не доверять клиентской
  валидации.
- ⚠️ **`check_bitrix_sessid()` — только для запросов в рамках сессии.** Для
  серверных вызовов извне (ERP, 1С) cookie сессии нет, и проверка sessid там
  неприменима — защищать такие вызовы секретом токена. Применять sessid-проверку
  к AJAX из собственной админки/публички, иначе возможен CSRF.
- ⚠️ **Вебхуки требуют HTTPS (по умолчанию).** На HTTP штатный входящий вебхук
  по умолчанию молча не авторизуется. Ограничение можно ослабить настройкой
  `REST_APAUTH_ALLOW_HTTP` (опция модуля `rest`) — только для внутренних/dev-
  контуров. Для рабочих проектов оставлять HTTPS.
- **Права вебхука = права создавшего пользователя.** Вебхук вызывает API
  с правами своего владельца и фиксированным набором scope; выдавать минимально
  необходимые scope (принцип наименьших привилегий).
- **Набор scope зависит от установленных модулей** и может отставать от облака —
  запрашивать реальный список (`scope`/`methods`) на конкретной сборке, а не
  предполагать облачный набор.
- **Инфоблок через REST.** Чтобы данные инфоблока отдавались по REST, у инфоблока
  должен быть включён флаг «Разрешён доступ через REST».

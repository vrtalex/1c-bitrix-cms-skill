## 1. /local в git: что коммитить, что НЕ коммитить

Весь кастомный код проекта держат в `/local` — папка имеет приоритет над `/bitrix` и
не перезаписывается при обновлениях ядра. `/bitrix` (ядро) в git не кладут целиком.

📌 Обрабатываемые подпапки `/local`: `activities`, `components`, `gadgets`, `modules`,
`php_interface` (`init.php`, `user_lang`), `templates`.
Приоритет всегда у `/local`: при одинаковых именах подключаются шаблоны/компоненты/
модули из `/local`. Кастомизация ядрового компонента — копией в `/local/components/bitrix/`.
([курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=43&LESSON_ID=2705),
[блог](https://dev.1c-bitrix.ru/community/blogs/vad/local-folder.php))

📌 С версии главного модуля 24.100.0 (актуально для 26.x) файлы настроек ядра
`.settings.php` и `.settings_extra.php` можно размещать в `/local`; папка `/local`
переживает обновления ядра.
([курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=43&LESSON_ID=2705))

👍 Новые проекты: шаблоны сайтов в `/local/templates` (а не `/bitrix/templates`),
пользовательские компоненты — в `/local/components/<namespace>/`. Это упрощает git и
переживает обновления ядра.
([курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=43&LESSON_ID=2826))

**Коммитят (в git):** только свой код в `/local` — компоненты, модули, шаблоны,
`init.php`, миграции. Репозиторий держат лёгким (бинарь и медиа не коммитят).

⚠️ **НЕ коммитить** (безопасность и среда):
- `/bitrix/` — ядро (нарушение лицензии при копировании ядра, ломает обновления);
- `/upload/` — пользовательский контент (большой объём);
- кэши: `/bitrix/cache`, `managed_cache`, `stack_cache`, `html_pages`, `tmp`;
- ⚠️ конфиги с паролями БД: `.settings.php` и `/bitrix/php_interface/dbconn.php` —
  боевой `.settings.php` лежит на сервере и в git не идёт;
- `.htaccess` и `robots.txt` — зависят от среды (разные на dev/stage/prod);
- `*.log`, `*.bak`, дампы и архивы; `sitemap*.xml`.
([типовой .gitignore](https://i-kochnev.ru/notes/git/tipovoy-gitignore-dlya-bitriks/) — партнёр)

🔧 Локальные переопределения через `.settings_extra.php`: добавляют в `.gitignore`,
каждый разработчик создаёт свой файл и переопределяет ключи (своя БД, отладка) —
Битрикс мержит `extra` поверх основного `.settings.php`.
([habr](https://habr.com/ru/articles/266573/) — партнёр)

🔧 На dev/stage блокируют исходящую почту через `define("ONLY_EMAIL","admin@site.ru")`
в `dbconn.php` и запрещают индексацию в `robots.txt` — поэтому `robots.txt` разный по
средам и не коммитится.
([habr](https://habr.com/ru/articles/571744/) — партнёр)

⚠️ Не оставлять демо/тестовый ключ на проде: на dev/stage — ключ разработки, на prod —
боевой. Иначе сайт перестанет работать по истечении демо-периода.
([habr](https://habr.com/ru/articles/571744/) — партнёр)

## 2. Перенос между средами (restore.php и бэкапы tar.gz)

Перенос сайта между хостингами/средами штатно делают через браузерный скрипт
`restore.php` и резервную копию в формате `*.tar.gz`.

🔧 **Резервная копия** (Настройки → Инструменты → Резервное копирование): файлы + дамп БД
в `*.tar.gz`. Из БД можно исключить таблицы статистики, журнала событий и поискового
индекса. Большой `/upload` архивируют отдельно: `tar -czvf upload.tar.gz ./upload`.
⚠️ Обязательно исключить из архива саму папку с бэкапами — иначе рекурсия.
([бэкап](https://docs.1c-bitrix.ru/pages/advanced/backup.html),
[дамп](https://dev.1c-bitrix.ru/user_help/settings/utilities/dump/dump.php))

👍 Бэкап «по шагам»: параметры «Длительность шага» и «Интервал» (сек) снижают нагрузку
на сервер при создании архива.
([дамп](https://dev.1c-bitrix.ru/user_help/settings/utilities/dump/dump.php))

🔧 **Перенос через restore.php** (браузерный):
1. Положить скрипт `restore.php`
   (`https://www.1c-bitrix.ru/download/files/scripts/restore.php`) в корень нового сайта;
   владелец файла — пользователь `bitrix`.
2. Загрузить ВСЕ части архива в корень.
3. Открыть `https://домен/restore.php` (строго HTTPS — параметры БД и логин админа иначе
   уходят открытым текстом), выбрать архив, ввести параметры новой БД,
   отметить «Создать базу данных, если не существует». Параметры пишутся в `.settings.php`.
   На время миграции ограничить доступ к скрипту по IP (allow-list своего адреса
   в конфиге веб-сервера/`.htaccess`) и/или basic-auth, а после переноса — удалить.
([курс](https://dev.1c-bitrix.ru/learning/course/?COURSE_ID=35&LESSON_ID=6979))

🔧 Мастер восстановления поддерживает полное восстановление (файлы + БД) и режим
«только файлы» (БД не трогается) — полезно при частичной потере файлов. Из админки:
Настройки → Инструменты → Резервное копирование → Список резервных копий → Восстановить.
([курс](https://dev.1c-bitrix.ru/learning/course/?COURSE_ID=35&LESSON_ID=6979&LESSON_PATH=3906.4833.6979))

🔧 После переноса: запустить Проверку системы (Настройки → Инструменты → Проверка
системы) — проверяет требования сервера и права на запись; красные параметры обязательны
к исправлению. Очистить `bitrix/cache/` и `bitrix/managed_cache/`, проверить ЧПУ, формы,
`robots.txt` и домен.
([Проверка системы](https://dev.1c-bitrix.ru/user_help/settings/utilities/site_checker.php),
[перенос rsync](https://cloud4box.com/blog/perenos-sajta-bitriks-na-drugoj-hosting-podrobnoe-rukovodstvo-bez-poteri-dannyh/) — партнёр)

⚠️ **Безопасность:** `restore.php`, оставленный на сервере после переноса, — дыра
безопасности. На окно миграции открывать его только по HTTPS и ограничить доступ
по IP/basic-auth; обязательно нажать «Удалить локальную резервную копию и служебные
скрипты» финальной кнопкой мастера, затем снять временное IP-/basic-auth-ограничение.
([курс](https://dev.1c-bitrix.ru/learning/course/?COURSE_ID=35&LESSON_ID=6979))

⚠️ **Многотомный архив:** загрузить ВСЕ части. Макс. размер одной части — 200 МБ
(оптимально 100 МБ, избегает ошибок распаковки). При ошибке HTTP 413 (Request Entity
Too Large) грузить партиями по ~9–10 файлов, при необходимости уменьшать партию.
([курс](https://dev.1c-bitrix.ru/learning/course/?COURSE_ID=35&LESSON_ID=6979),
[бэкап](https://docs.1c-bitrix.ru/pages/advanced/backup.html))

⚠️ Типовые ошибки переноса:
- права/владелец: папки `0755`, файлы `0644` (для `/upload` и cache иногда `0777`); можно
  задать `BX_FILE_PERMISSIONS`/`BX_DIR_PERMISSIONS` в `dbconn.php`;
- не скопирован `.access.php` → «Запрещено»;
- неверный «Путь к корневой папке» → видна только форма входа;
- `ERROR 1062` → несовпадение кодировки, лечить `character_set_server` / `SET NAMES`.
([курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=3293&LESSON_PATH=3903.4862.4894.3293))

👍 **Облако 1С-Битрикс** (автобэкап): бесплатно для всех редакций, кроме «Первый сайт»;
нужна активная коммерческая лицензия; хранится максимум 3 последние копии.
⚠️ Копии всегда шифруются вашим паролем, отключить шифрование нельзя; 1С-Битрикс не имеет
доступа к содержимому и не может восстановить пароль — забыли пароль = восстановление
невозможно.
([автобэкап](https://dev.1c-bitrix.ru/user_help/settings/bitrixcloud/bitrixcloud_backup.php),
[курс](https://dev.1c-bitrix.ru/learning/course/?COURSE_ID=35&LESSON_ID=7080))

## 3. Обновления (базово): SiteUpdate, UTF-8, окружение

🔧 **SiteUpdate** — штатная система обновлений ядра, модулей и компонентов без участия
разработчика; закрывает известные и новые уязвимости, не затрагивает публичную часть и
сохраняет данные. Marketplace → Обновление платформы.
📌 Адрес сервера обновлений в настройках Главного модуля обязан быть `www.1c-bitrix.ru`.
([SiteUpdate](https://www.1c-bitrix.ru/products/cms/security/siteupdate.php),
[курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&CHAPTER_ID=04745))

📌 **Порядок обновления:** сначала ядро (`main`) до промежуточной и далее до актуальной →
системные модули → сторонние решения/шаблоны. «Установить рекомендуемые обновления»
ставит всё сразу.
👍 Опция «Загружать только стабильные обновления» включена по умолчанию — для боевых
проектов не выключать (бета — только на тестовом окружении).
([курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=2693),
[курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&LESSON_ID=2689))

👍 **Безопасное обновление:** полный бэкап (файлы + БД) перед апдейтом, обновить и
протестировать на копии/staging, затем перенести на боевой. Не прерывать установку.
После — полная очистка кеша, включая композитный (Настройки → Автокеширование → очистить
все опции).
([SiteUpdate](https://www.1c-bitrix.ru/products/cms/security/siteupdate.php),
[Intervolga](https://www.intervolga.ru/faq/services-1c/kak-obnovit-yadro-i-moduli-v-1s-bitriks-bezopasno/) — партнёр)

📌 **UTF-8 с версии main 24.0.0:** продукты полностью на UTF-8, однобайтовые установки
(cp1251) больше не поддерживаются. Фактически апдейт сайта на cp1251 блокируется уже с
23.200.0. Все сайты одного ядра — в единой кодировке (разные кодировки в многосайтовости
невозможны).
🔧 Мастер конвертации в UTF-8 доступен с 23.200.0: `/bitrix/admin/wizard_list.php?lang=ru`;
на время работы публичная часть закрывается. ⚠️ Перед конвертацией обязателен полный
бэкап файлов + БД; потренироваться на копии.
([курс](https://dev.1c-bitrix.ru/learning/course/?COURSE_ID=43&LESSON_ID=2919))

📌 **Окружение (минимум для обновлений):** с 01.02.2026 без PHP ≥8.2 нельзя ставить
обновления (рекоменд. 8.3+, для нового ядра 8.4+). MySQL ≥8.0 (рекоменд. 8.4, допустим
Percona); с 01.09.2026 ограничивается поддержка версий ниже 8.0.0. `memory_limit=256M`,
`default_charset=UTF-8`, `OPcache`. ⚠️ При включённом `mbstring.func_overload`
система обновлений не работает — отключить (`mbstring.func_overload=0`).
([требования](https://dev.1c-bitrix.ru/user_help/reqintro.php),
[helpdesk](https://helpdesk.bitrix24.ru/open/5825131/))

📌 Право на обновления действует 1 год после регистрации лицензии. После истечения CMS
«Управление сайтом» продолжает работать в «замороженном» виде (нет багфиксов и
обновлений безопасности, но сайт функционирует). ⚠️ Накопление дыр безопасности — главный
риск отложенных апдейтов.
([курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=32&CHAPTER_ID=04745),
[martsoft](https://martsoft.ru/license/prodlenie-bitriks/) — партнёр)

⚠️ **ГЛАВНАЯ ГРАБЛЯ обновлений:** бан лицензионного ключа в SiteUpdate, если один ключ
используется и на копии, и на боевом — ошибка `[ERROR_WRONG_CODE]` «Проверка сайта не
пройдена». Снятие бана — через техподдержку. Профилактика: отдельный тестовый ключ/
окружение, не запускать систему обновлений с боевым ключом на копии.
([webdev](https://dev.1c-bitrix.ru/community/webdev/user/14686/blog/1375/))

⚠️ Между мажорами удалены базовые классы `CAll*` и JS-расширения `core_fx`/`core_ls`
(перенесены в `core`) — старый код, опирающийся на них, требует адаптации. Перед
апгрейдом PHP сначала обновить ядро/модули до совместимой версии, потом переключать PHP.
([bxmax](https://bxmax.ru/blog/bolshoe-yanvarskoe-obnovlenie-bitriks-2026-chto-vazhno-razrabotchikam) — партнёр)

## 4. Мультисайт (базово)

📌 Многосайтовость = неограниченное число сайтов на одной копии/лицензии: одно ядро
(`/bitrix`), одна БД, одна админка. Ограничение по числу сайтов — только в редакциях
«Первый сайт» и «Старт».
([мультисайт](https://www.1c-bitrix.ru/products/cms/multisite/),
[курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=35&LESSON_ID=1968))

🔧 **Две модели:**
- **Разные домены:** каждый сайт в своём `DOCUMENT_ROOT`, «Папка сайта» = `/`, ядро в
  одном месте + симлинки на `/bitrix`, `/upload`, `/local`.
- **Один домен по папкам:** `/s1/`, `/s2/`, один корень, различение по «Папке сайта»;
  «Доменное имя» пусто (или одинаковое), настройка `httpd.conf` не требуется; структуры
  файлов сайтов не должны пересекаться.
([курс](https://dev.1c-bitrix.ru/learning/course/?COURSE_ID=103&LESSON_ID=287),
[курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=103&LESSON_ID=286))

📌 Во второй сайт обязательно скопировать `/.htaccess` (ЧПУ), `/404.php`, `index.php` и
создать `.access.php` с правом чтения для гостей: `<?php $PERM["/"]["*"]="R"; ?>` — без него
публичная часть потребует авторизации.
([курс](https://dev.1c-bitrix.ru/learning/course/?COURSE_ID=103&LESSON_ID=287),
[блог](https://dev.1c-bitrix.ru/community/blogs/howto/336.php))

🔧 Поля формы сайта (Настройки → Сайты → Список сайтов): ID (2 символа, он же LID),
Название, Доменное имя, Папка сайта, Сортировка, Название сайта (для писем), URL сервера,
E-mail по умолчанию (`DEFAULT_EMAIL_FROM`), Путь к корневой папке веб-сервера
(`DOCUMENT_ROOT`), Язык (`LANGUAGE_ID`). В публичке текущий сайт = константа `SITE_ID`.
([курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=103&LESSON_ID=289))

👍 Инфоблоки и компоненты — единое хранилище для всех сайтов; разделение через выборки по
`SITE_ID`, разные шаблоны компонентов и разные меню (`.left.menu.php`, `.top.menu.php`).
Привязка инфоблока — поле «Сайты» в форме инфоблока + свои URL-шаблоны с макросами
(`#SITE_DIR#`, `#SERVER_NAME#`, `#SECTION_CODE#`, `#ELEMENT_CODE#`).
([блог](https://dev.1c-bitrix.ru/community/blogs/howto/336.php),
[справка](https://dev.1c-bitrix.ru/user_help/content/iblock/iblock_edit.php))

⚠️ **Алгоритм определения сайта** двухуровневый: 1) по доменному имени (сравнение до точки —
поэтому ВСЕ поддомены автоматически относятся к основному домену); 2) по «Папке сайта».
Если второй сайт — поддомен первого (`shop.site.ru`), его индекс сортировки должен быть
МЕНЬШЕ (выше в списке), чем у основного, иначе Битрикс отдаст контент основного сайта.
([курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=103&LESSON_ID=1675),
[блог](https://dev.1c-bitrix.ru/community/blogs/howto/336.php))

⚠️ Ядро (`/bitrix`) копировать запрещено — нарушает лицензию и ломает обновления; на доп.
сайты ставятся только симлинки (`ln -s`) на `/bitrix`, `/upload`, `/local`. На
shared-хостинге `open_basedir` может блокировать создание симлинков через PHP.
([блог](https://dev.1c-bitrix.ru/community/blogs/howto/336.php))

⚠️ При мультисайте на разных доменах добавить ВСЕ домены в список разрешённых хостов
модуля «Проактивная защита», иначе 403 Forbidden. Для HTML-кеширования на разных доменах
кеш/шаблоны разносят через `BX_PERSONAL_ROOT` (`SetEnv BX_PERSONAL_ROOT /bitrix_personal`)
+ симлинки на `/bitrix/php_interface` и `/bitrix/templates`.
([блог](https://dev.1c-bitrix.ru/community/blogs/howto/336.php))

👍 **Мультиязычность** — два подхода: отдельные сайты с разными `LANGUAGE_ID` (раздельные
статистика/валюты/заказы, рекомендуется) или папки внутри одного сайта (`/en/`, `/de/` —
проще бэкап, но нужен код распознавания). Отдельная лицензия на языковую версию НЕ нужна.
⚠️ Нет встроенной поддержки нескольких языков в ОДНОМ элементе инфоблока — для контента
нужны отдельные инфоблоки/элементы под язык.
([курс](https://dev.1c-bitrix.ru/learning/course/index.php?COURSE_ID=35&LESSON_ID=1968),
[блог](https://dev.1c-bitrix.ru/community/blogs/product_features/7974.php))

🔧 SEO мультисайта: посайтовый `sitemap.xml` и `robots.txt` (Маркетинг → Поисковая
оптимизация — выбор сайта при настройке). Зеркала доменов система не различает
автоматически — закрывать редиректом на основной домен в `.htaccess`.
([SEO+sitemap](https://dev.1c-bitrix.ru/community/blogs/product_features/seo-and-sitemapxml-in-version-14-1cbitrix-site-management.php),
[блог](https://dev.1c-bitrix.ru/community/blogs/howto/336.php))
